## 1. Что такое SDLC

```
┌────────────────────────────────────────────────────────────┐
│                         SDLC                               │
│         Software Development Life Cycle                    │
│         Жизненный цикл разработки ПО                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  SDLC — это структурированный процесс, описывающий         │
│  этапы создания программного обеспечения от идеи           │
│  до вывода из эксплуатации.                                │
│                                                            │
│  Цели SDLC:                                                │
│  • Обеспечить качество продукта                            │
│  • Минимизировать риски                                    │
│  • Уложиться в сроки и бюджет                              │
│  • Создать понятный и повторяемый процесс                  │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    ЭТАПЫ SDLC                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. ПЛАНИРОВАНИЕ (Planning)                                 │
│     • Определение целей проекта                             │
│     • Оценка ресурсов, сроков, бюджета                      │
│     • Анализ рисков                                        │
│     • Создание плана проекта                               │
│                                                             │
│  2. АНАЛИЗ ТРЕБОВАНИЙ (Requirements Analysis)              │
│     • Сбор требований от заказчика/пользователей           │
│     • Документирование функциональных требований           │
│     • Документирование нефункциональных требований         │
│     • Согласование требований                              │
│                                                             │
│  3. ПРОЕКТИРОВАНИЕ (Design)                                │
│     • Архитектура системы                                  │
│     • Проектирование БД                                    │
│     • Проектирование интерфейсов (UI/UX)                   │
│     • Выбор технологий                                     │
│     • Создание технической документации                    │
│                                                             │
│  4. РАЗРАБОТКА (Development/Implementation)                │
│     • Написание кода                                       │
│     • Unit-тестирование                                    │
│     • Code review                                          │
│     • Интеграция компонентов                               │
│                                                             │
│  5. ТЕСТИРОВАНИЕ (Testing)                                 │
│     • Функциональное тестирование                          │
│     • Интеграционное тестирование                          │
│     • Нагрузочное тестирование                             │
│     • Приёмочное тестирование (UAT)                        │
│     • Исправление багов                                    │
│                                                             │
│  6. РАЗВЁРТЫВАНИЕ (Deployment)                             │
│     • Подготовка инфраструктуры                            │
│     • Установка на production                              │
│     • Миграция данных                                      │
│     • Обучение пользователей                               │
│                                                             │
│  7. ПОДДЕРЖКА (Maintenance)                                │
│     • Мониторинг                                           │
│     • Исправление багов                                    │
│     • Обновления и улучшения                               │
│     • Техническая поддержка                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 2. Модели SDLC

### Waterfall (Каскадная модель)

text

```
┌────────────────────────────────────────────────────────────┐
│                  WATERFALL (ВОДОПАД)                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Последовательное выполнение этапов.                      │
│  Каждый этап завершается полностью перед началом          │
│  следующего. Возврат к предыдущим этапам затруднён.       │
│                                                            │
│       ┌──────────────┐                                     │
│       │ Требования   │                                     │
│       └──────┬───────┘                                     │
│              ▼                                             │
│       ┌──────────────┐                                     │
│       │ Проектирование│                                    │
│       └──────┬───────┘                                     │
│              ▼                                             │
│       ┌──────────────┐                                     │
│       │  Разработка  │                                     │
│       └──────┬───────┘                                     │
│              ▼                                             │
│       ┌──────────────┐                                     │
│       │ Тестирование │                                     │
│       └──────┬───────┘                                     │
│              ▼                                             │
│       ┌──────────────┐                                     │
│       │ Развёртывание│                                     │
│       └──────┬───────┘                                     │
│              ▼                                             │
│       ┌──────────────┐                                     │
│       │  Поддержка   │                                     │
│       └──────────────┘                                     │
│                                                            │
│  ✅ Плюсы:                                                 │
│  • Простота и понятность                                  │
│  • Чёткие этапы и документация                            │
│  • Легко управлять                                        │
│  • Подходит для проектов с фиксированными требованиями    │
│                                                            │
│  ❌ Минусы:                                                │
│  • Негибкость к изменениям                                │
│  • Поздняя обратная связь (продукт виден только в конце)  │
│  • Высокий риск провала при ошибке на ранних этапах       │
│  • Долгий Time-to-Market                                  │
│                                                            │
│  Когда использовать:                                      │
│  • Требования полностью известны и не изменятся           │
│  • Проект небольшой и простой                             │
│  • Критичные системы (медицина, авиация)                  │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### Agile (Гибкая методология)

text

```
┌────────────────────────────────────────────────────────────┐
│                        AGILE                               │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Итеративная разработка короткими циклами (спринтами).    │
│  Постоянная обратная связь, адаптация к изменениям.       │
│                                                            │
│    ┌──────────────────────────────────────────────┐       │
│    │              ИТЕРАЦИЯ (Спринт)               │       │
│    │  ┌─────────┬─────────┬─────────┬─────────┐  │       │
│    │  │Планиров.│Разработка│Тестиров.│ Релиз  │  │       │
│    │  └─────────┴─────────┴─────────┴─────────┘  │       │
│    └──────────────────────────────────────────────┘       │
│                          │                                │
│                          ▼                                │
│    ┌──────────────────────────────────────────────┐       │
│    │              ИТЕРАЦИЯ (Спринт)               │       │
│    │  ┌─────────┬─────────┬─────────┬─────────┐  │       │
│    │  │Планиров.│Разработка│Тестиров.│ Релиз  │  │       │
│    │  └─────────┴─────────┴─────────┴─────────┘  │       │
│    └──────────────────────────────────────────────┘       │
│                          │                                │
│                          ▼                                │
│                        ...                                │
│                                                            │
│  Ценности Agile Manifesto (2001):                        │
│  ─────────────────────────────────                        │
│  • Люди и взаимодействие ВАЖНЕЕ процессов и инструментов │
│  • Работающий продукт ВАЖНЕЕ документации                │
│  • Сотрудничество с заказчиком ВАЖНЕЕ контрактов         │
│  • Готовность к изменениям ВАЖНЕЕ следования плану       │
│                                                            │
│  Принципы:                                                │
│  • Короткие итерации (1-4 недели)                        │
│  • Частая поставка работающего ПО                        │
│  • Постоянная обратная связь                             │
│  • Самоорганизующиеся команды                            │
│  • Непрерывное улучшение                                 │
│                                                            │
│  ✅ Плюсы:                                                 │
│  • Гибкость к изменениям                                 │
│  • Ранняя и частая обратная связь                        │
│  • Быстрый Time-to-Market                                │
│  • Снижение рисков                                       │
│                                                            │
│  ❌ Минусы:                                                │
│  • Сложнее планировать бюджет и сроки                    │
│  • Требует вовлечённости заказчика                       │
│  • Меньше документации                                   │
│  • Сложнее для больших распределённых команд             │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### Сравнение Waterfall и Agile

text

```
┌──────────────────┬────────────────────┬────────────────────┐
│    Критерий      │     Waterfall      │       Agile        │
├──────────────────┼────────────────────┼────────────────────┤
│ Подход           │ Последовательный   │ Итеративный        │
│ Требования       │ Фиксированные      │ Меняющиеся         │
│ Изменения        │ Затруднены         │ Приветствуются     │
│ Планирование     │ В начале проекта   │ Перед каждой итер. │
│ Документация     │ Подробная          │ Минимальная        │
│ Тестирование     │ В конце            │ Непрерывное        │
│ Обратная связь   │ В конце проекта    │ После каждой итер. │
│ Релиз            │ Один в конце       │ После каждой итер. │
│ Риски            │ Высокие            │ Низкие             │
│ Вовлечённость    │ Низкая             │ Высокая            │
│ заказчика        │                    │                    │
│ Команда          │ Специализированная │ Кросс-функциональн.│
└──────────────────┴────────────────────┴────────────────────┘
```

### Другие модели SDLC

text

```
┌────────────────────────────────────────────────────────────┐
│                    ДРУГИЕ МОДЕЛИ                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  V-MODEL (V-образная модель)                              │
│  ────────────────────────────                              │
│  Расширение Waterfall. Каждому этапу разработки           │
│  соответствует этап тестирования.                         │
│                                                            │
│  Требования ─────────────────────── Приёмочное тест.      │
│       Дизайн ───────────────────── Системное тест.        │
│           Архитектура ───────── Интеграц. тест.           │
│               Код ───── Unit-тесты                        │
│                                                            │
│  ────────────────────────────────────────────────────────  │
│                                                            │
│  SPIRAL (Спиральная модель)                               │
│  ──────────────────────────                                │
│  Комбинация Waterfall и итеративного подхода.             │
│  Акцент на анализе рисков.                                │
│                                                            │
│  Каждый виток спирали:                                    │
│  1. Определение целей                                     │
│  2. Анализ рисков                                         │
│  3. Разработка и тестирование                             │
│  4. Планирование следующего витка                         │
│                                                            │
│  ────────────────────────────────────────────────────────  │
│                                                            │
│  ITERATIVE (Итеративная модель)                           │
│  ──────────────────────────────                            │
│  Разработка через повторяющиеся циклы.                    │
│  Каждая итерация — работающая версия продукта.            │
│                                                            │
│  ────────────────────────────────────────────────────────  │
│                                                            │
│  INCREMENTAL (Инкрементальная модель)                     │
│  ────────────────────────────────────                      │
│  Продукт разбивается на части (инкременты).               │
│  Каждый инкремент — законченный функционал.               │
│                                                            │
│  ────────────────────────────────────────────────────────  │
│                                                            │
│  RAD (Rapid Application Development)                      │
│  ───────────────────────────────────                       │
│  Быстрая разработка через прототипирование.               │
│  Минимум планирования, акцент на быстрый результат.       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 3. DevOps

text

```
┌────────────────────────────────────────────────────────────┐
│                        DEVOPS                              │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  DevOps = Development + Operations                        │
│                                                            │
│  Это КУЛЬТУРА и ПРАКТИКИ, направленные на объединение     │
│  разработки (Dev) и эксплуатации (Ops) для ускорения      │
│  и улучшения процесса поставки ПО.                        │
│                                                            │
│  DevOps — это НЕ:                                         │
│  • Отдельная должность (хотя так называют инженеров)     │
│  • Инструмент или технология                              │
│  • Отдел в компании                                       │
│                                                            │
│  DevOps — это:                                            │
│  • Культура сотрудничества                                │
│  • Набор практик и принципов                              │
│  • Философия непрерывного улучшения                       │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│                  ПРИНЦИПЫ DEVOPS                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. CULTURE (Культура)                                    │
│     • Разрушение стены между Dev и Ops                    │
│     • Общая ответственность за продукт                    │
│     • Культура без обвинений (blameless culture)          │
│     • Обучение на ошибках                                 │
│                                                            │
│  2. AUTOMATION (Автоматизация)                            │
│     • Автоматизация рутинных задач                        │
│     • Инфраструктура как код (IaC)                        │
│     • Автоматическое тестирование                         │
│     • Автоматическое развёртывание                        │
│                                                            │
│  3. LEAN (Бережливость)                                   │
│     • Устранение потерь                                   │
│     • Малые партии работ                                  │
│     • Ограничение незавершённой работы (WIP)              │
│                                                            │
│  4. MEASUREMENT (Измерение)                               │
│     • Метрики производительности                          │
│     • Мониторинг всего                                    │
│     • Принятие решений на основе данных                   │
│                                                            │
│  5. SHARING (Обмен знаниями)                              │
│     • Прозрачность процессов                              │
│     • Документирование                                    │
│     • Обмен опытом между командами                        │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│                  ПРАКТИКИ DEVOPS                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  CI/CD (Continuous Integration / Continuous Delivery)     │
│  ─────────────────────────────────────────────────────     │
│  • CI: автоматическая сборка и тестирование при коммите  │
│  • CD: автоматическое развёртывание в окружения          │
│                                                            │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   │
│  │  Code   │──►│  Build  │──►│  Test   │──►│ Deploy  │   │
│  └─────────┘   └─────────┘   └─────────┘   └─────────┘   │
│                                                            │
│  Infrastructure as Code (IaC)                             │
│  ─────────────────────────────                             │
│  • Terraform, Ansible, Pulumi                             │
│  • Версионирование инфраструктуры                         │
│  • Воспроизводимость окружений                            │
│                                                            │
│  Monitoring & Logging                                     │
│  ────────────────────                                      │
│  • Prometheus, Grafana, ELK                               │
│  • Alerting                                               │
│  • Observability                                          │
│                                                            │
│  Containerization & Orchestration                         │
│  ────────────────────────────────                          │
│  • Docker, Kubernetes                                     │
│  • Microservices                                          │
│                                                            │
│  GitOps                                                   │
│  ──────                                                    │
│  • Git как единственный источник правды                  │
│  • Декларативная инфраструктура                          │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│              DEVOPS LIFECYCLE (∞)                          │
├────────────────────────────────────────────────────────────┤
│                                                            │
│            ┌─────────────────────────────────┐            │
│            │                                 │            │
│     ┌──────▼──────┐                ┌─────────┴────┐       │
│     │    PLAN     │                │   MONITOR    │       │
│     └──────┬──────┘                └─────────▲────┘       │
│            │                                 │            │
│     ┌──────▼──────┐                ┌─────────┴────┐       │
│     │    CODE     │     DEV        │   OPERATE    │  OPS  │
│     └──────┬──────┘                └─────────▲────┘       │
│            │                                 │            │
│     ┌──────▼──────┐                ┌─────────┴────┐       │
│     │    BUILD    │                │   DEPLOY     │       │
│     └──────┬──────┘                └─────────▲────┘       │
│            │                                 │            │
│     ┌──────▼──────┐                ┌─────────┴────┐       │
│     │    TEST     │────────────────►   RELEASE    │       │
│     └─────────────┘                └──────────────┘       │
│                                                            │
│  Plan → Code → Build → Test → Release → Deploy →          │
│  → Operate → Monitor → Plan → ...                         │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 4. Роли в команде разработки

text

```
┌────────────────────────────────────────────────────────────┐
│                  РАЗРАБОТЧИК (Developer)                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Основные обязанности:                                    │
│  • Написание кода согласно требованиям                   │
│  • Написание unit-тестов                                  │
│  • Code review кода коллег                                │
│  • Исправление багов                                      │
│  • Участие в проектировании                              │
│  • Документирование кода                                  │
│  • Оценка задач                                          │
│                                                            │
│  Взаимодействие:                                          │
│  • С тестировщиками — по багам и тест-кейсам             │
│  • С DevOps — по развёртыванию и инфраструктуре          │
│  • С аналитиками — по требованиям                        │
│  • С другими разработчиками — code review, парное прогр. │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│               ТЕСТИРОВЩИК (QA Engineer)                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Основные обязанности:                                    │
│  • Анализ требований на тестируемость                    │
│  • Написание тест-кейсов и тест-планов                   │
│  • Проведение функционального тестирования               │
│  • Проведение регрессионного тестирования                │
│  • Заведение и отслеживание багов                        │
│  • Приёмочное тестирование                               │
│  • Тестирование юзабилити                                │
│                                                            │
│  Виды тестирования:                                       │
│  • Функциональное / Нефункциональное                     │
│  • Ручное / Автоматизированное                           │
│  • Smoke / Regression / Integration                       │
│  • Performance / Security / Usability                    │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│            ПРОЕКТНЫЙ МЕНЕДЖЕР (Project Manager)            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Основные обязанности:                                    │
│  • Планирование проекта (сроки, ресурсы, бюджет)         │
│  • Управление scope (объёмом работ)                      │
│  • Управление рисками                                    │
│  • Коммуникация с заказчиком и стейкхолдерами           │
│  • Контроль выполнения задач                             │
│  • Отчётность по проекту                                 │
│  • Решение проблем и устранение блокеров                │
│  • Управление изменениями                                │
│                                                            │
│  Ключевые навыки:                                         │
│  • Планирование и организация                            │
```

## 7. Scrum vs Kanban

text

```
┌────────────────────────────────────────────────────────────┐
│                        KANBAN                              │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Kanban — метод управления потоком работы.                │
│  Визуализация задач на доске, ограничение WIP.            │
│                                                            │
│  ПРИНЦИПЫ:                                                │
│  • Визуализация потока работы                            │
│  • Ограничение WIP (Work In Progress)                    │
│  • Управление потоком                                    │
│  • Явные правила процесса                                │
│  • Петли обратной связи                                  │
│  • Постоянное улучшение                                  │
│                                                            │
│  KANBAN-ДОСКА:                                            │
│  ┌──────────┬──────────┬──────────┬──────────┐           │
│  │ To Do    │ In Prog  │ Review   │   Done   │           │
│  │          │ (WIP: 3) │ (WIP: 2) │          │           │
│  ├──────────┼──────────┼──────────┼──────────┤           │
│  │ [Task 5] │ [Task 2] │ [Task 1] │ [Task A] │           │
│  │ [Task 6] │ [Task 3] │          │ [Task B] │           │
│  │ [Task 7] │ [Task 4] │          │ [Task C] │           │
│  │ [Task 8] │          │          │          │           │
│  └──────────┴──────────┴──────────┴──────────┘           │
│                                                            │
│  WIP Limit — ограничение незавершённой работы.            │
│  Нельзя брать новую задачу, пока есть незавершённые.      │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌───────────────────┬────────────────────┬───────────────────┐
│     Критерий      │       SCRUM        │      KANBAN       │
├───────────────────┼────────────────────┼───────────────────┤
│ Итерации          │ Фиксированные      │ Непрерывный поток │
│                   │ спринты            │                   │
│ Роли              │ PO, SM, Team       │ Не обязательны    │
│ Планирование      │ В начале спринта   │ По мере надобности│
│ Изменения         │ В следующем спринте│ В любой момент    │
│ WIP-лимит         │ Косвенно (ёмкость) │ Явный             │
│ Церемонии         │ Обязательные       │ Нет обязательных  │
│ Метрики           │ Velocity           │ Lead Time,        │
│                   │                    │ Cycle Time        │
│ Доска             │ Сбрасывается       │ Постоянная        │
│ Лучше для         │ Продуктовая разраб.│ Поддержка, ops    │
│ Обучение          │ Сложнее            │ Проще             │
└───────────────────┴────────────────────┴───────────────────┘

Когда использовать SCRUM:
• Новый продукт с неопределёнными требованиями
• Нужна регулярная обратная связь
• Команда готова к изменениям процессов
• Важна предсказуемость поставок

Когда использовать KANBAN:
• Поддержка и сопровождение продукта
• Непредсказуемый поток задач (баги, запросы)
• Команда уже имеет процесс и хочет улучшить
• Нужна гибкость в приоритетах
```

## 9. Lean Development

```
┌────────────────────────────────────────────────────────────┐
│                   LEAN DEVELOPMENT                         │
├────────────────────────────────────────────────────────────┤
│                                                            │

│  производства Toyota (Toyota Production System).          │
│                                                            │
│  Основная идея: УСТРАНЕНИЕ ПОТЕРЬ и максимизация         │

│                                                            │

│     ────────────────────────────────                       │
│     Потери в разработке:                                  │
│     • Незавершённая работа (частично сделанные фичи)     │
│     • Лишние функции (которые не используются)           │
│     • Переключение контекста (многозадачность)           │
│     • Ожидание (согласований, ответов, сборки)           │
│     • Передача задач (handoffs между командами)          │

│     • Лишние процессы (ненужная документация)            │
│                                                            │
│  2. УСИЛЕНИЕ ОБУЧЕНИЯ (Amplify Learning)                  │
│     ────────────────────────────────                       │

│     • Эксперименты и прототипы                           │
│     • Ретроспективы                                      │
│     • Парное программирование                            │

│  3. ПРИНЯТИЕ РЕШЕНИЙ КАК МОЖНО ПОЗЖЕ                      │
│     (Decide as Late as Possible)                          │
│     ────────────────────────────                           │
│     • Не принимать необратимых решений рано              │
│     • Собирать больше информации                         │
│     • Сохранять гибкость                                 │
│                                                            │
│  4. ПОСТАВКА КАК МОЖНО РАНЬШЕ                             │
│     (Deliver as Fast as Possible)                         │
│     ──────────────────────────────                         │
│     • Короткий Time-to-Market                            │
│     • Быстрая обратная связь                             │
│     • Малые партии работ                                 │
│                                                            │
│  5. РАСШИРЕНИЕ ПОЛНОМОЧИЙ КОМАНДЫ                         │
│     (Empower the Team)                                    │
│     ─────────────────────────────                          │
│     • Самоорганизация                                    │
│     • Принятие решений на уровне команды                 │
│     • Доверие профессионалам                             │
│                                                            │
│  6. ВСТРАИВАНИЕ КАЧЕСТВА (Build Integrity In)             │
│     ─────────────────────────────────────                  │
│     • Качество на каждом этапе, а не в конце             │
│     • Автоматическое тестирование                        │
│     • Рефакторинг                                        │
│     • Code review                                        │
│                                                            │
│  7. ВИДЕНИЕ ЦЕЛОГО (See the Whole)                        │
│     ──────────────────────────────                         │
│     • Оптимизация всего потока, а не отдельных частей    │
│     • Системное мышление                                 │
│     • Кросс-функциональные команды                       │
│                                                            │
└────────────────────────────────────────────────────────────┘

│                                                            │

│                                                            │
│  ┌─────────────────────────────────────────────────────┐  │
│  │                      LEAN                           │  │
│  │  ┌───────────────────────────────────────────────┐ │  │
│  │  │                    AGILE                      │ │  │
│  │  │  ┌─────────┐  ┌─────────┐  ┌─────────┐      │ │  │
│  │  │  │  SCRUM  │  │ KANBAN  │  │   XP    │ ...  │ │  │
│  │  │  └─────────┘  └─────────┘  └─────────┘      │ │  │
│  │  └───────────────────────────────────────────────┘ │  │
│  └─────────────────────────────────────────────────────┘  │
```

## 12. SRE (Site Reliability Engineering)

```
┌────────────────────────────────────────────────────────────┐
│                          SRE                               │
│              Site Reliability Engineering                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  SRE — инженерная дисциплина, созданная в Google для       │
│  обеспечения НАДЁЖНОСТИ масштабных систем.                 │
│                                                            │
│  "SRE — это то, что получается, когда программиста         │
│  просят заняться эксплуатацией" — Ben Treynor, Google      │
│                                                            │
│  Основная идея: применить ИНЖЕНЕРНЫЕ ПОДХОДЫ к             │
│  проблемам эксплуатации.                                   │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│                   КЛЮЧЕВЫЕ КОНЦЕПЦИИ SRE                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  SLI (Service Level Indicator)                             │
│  ─────────────────────────────                             │
│  Метрика, измеряющая поведение сервиса.                    │
│  Примеры:                                                  │
│  • Latency (время ответа)                                  │
│  • Availability (доступность)                              │
│  • Error Rate (процент ошибок)                             │
│  • Throughput (пропускная способность)                     │
│                                                            │
│  SLO (Service Level Objective)                             │
│  ─────────────────────────────                             │
│  Целевое значение SLI.                                     │
│  Примеры:                                                  │
│  • 99.9% запросов быстрее 200ms                            │
│  • 99.95% доступность в месяц                              │
│  • Менее 0.1% ошибок                                       │
│                                                            │
│  SLA (Service Level Agreement)                             │
│  ─────────────────────────────                             │
│  Договор с клиентом, включающий SLO и последствия          │
│  невыполнения (компенсации, штрафы).                       │
│                                                            │
│  ERROR BUDGET (Бюджет ошибок)                              │
│  ────────────────────────────                              │
│  Допустимый уровень ненадёжности.                          │
│  Если SLO = 99.9%, то error budget = 0.1%                  │
│                                                            │
│  Пока есть бюджет — можно выпускать фичи.                  │
│  Бюджет исчерпан — стоп релизам, фокус на надёжность.      │
│                                                            │
│  TOIL (Рутина)                                             │
│  ─────────────                                             │
│  Ручная, повторяющаяся работа, которую нужно               │
│  автоматизировать.                                         │
│  Цель SRE: не более 50% времени на toil.                   │
│                                                            │
│  POSTMORTEM (Разбор инцидентов)                            │
│  ──────────────────────────────                            │
│  Анализ инцидентов без обвинений (blameless).              │
│  Цель: предотвратить повторение.                           │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│                    SRE vs DEVOPS                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  DevOps и SRE — близкие, но разные концепции.              │
│                                                            │
│  ОБЩЕЕ:                                                    │
│  • Устранение стены между Dev и Ops                        │
│  • Автоматизация                                           │
│  • Мониторинг и observability                              │
│  • Культура сотрудничества                                 │
│  • CI/CD                                                   │
│                                                            │
│  SRE можно рассматривать как КОНКРЕТНУЮ РЕАЛИЗАЦИЮ         │
│  принципов DevOps с фокусом на надёжность.                 │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌───────────────────┬────────────────────┬───────────────────┐
│     Аспект        │       DevOps       │        SRE        │
├───────────────────┼────────────────────┼───────────────────┤
│ Происхождение     │ Сообщество (2008)  │ Google (2003)     │
│ Что это          │ Культура, философия│ Инженерная практ.  │
│ Фокус            │ Скорость поставки  │ Надёжность системы │
│ Метрики          │ Deployment freq.,  │ SLI, SLO, Error    │
│                  │ Lead time          │ Budget             │
│ Подход к рискам  │ Fail fast, iterate │ Error budget       │
│ Автоматизация    │ CI/CD пайплайны    │ Устранение toil    │
│ Команда          │ Dev + Ops вместе   │ SRE как отдельная  │
│                  │                    │ инженерная роль    │
│ On-call          │ Возможно           │ Обязательно        │
│ Код              │ Опционально        │ 50%+ времени       │
│ Баланс           │ Скорость/качество  │ Надёжность/фичи    │
└───────────────────┴────────────────────┴───────────────────┘

┌────────────────────────────────────────────────────────────┐
│               ОБЯЗАННОСТИ SRE-ИНЖЕНЕРА                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  • Определение и мониторинг SLI/SLO                        │
│  • Управление error budget                                 │
│  • Автоматизация рутинных операций (устранение toil)       │
│  • Разработка инструментов надёжности                      │
│  • Capacity planning (планирование мощностей)              │
│  • Incident response (реагирование на инциденты)           │
│  • On-call дежурства                                       │
│  • Postmortem и предотвращение инцидентов                  │
│  • Performance engineering                                 │
│  • Chaos engineering                                       │
│  • Консультирование Dev-команд по надёжности               │
│                                                            │
└────────────────────────────────────────────────────────────┘
```
```
┌────────────────────────────────────────────────────────────┐
│            КОГДА НУЖЕН DevOps, КОГДА SRE                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  DevOps подходит:                                          │
│  • Стартапам и небольшим компаниям                         │
│  • Когда нужно ускорить поставку                           │
│  • Для начала трансформации                                │
│  • Когда команды небольшие                                 │
│                                                            │
│  SRE подходит:                                             │

│  • При масштабных распределённых системах                  │
│  • Когда есть ресурсы на отдельную SRE-команду             │                          
│  На практике DevOps и SRE часто ДОПОЛНЯЮТ друг друга:      │
│  • DevOps — культура и процессы CI/CD                      │
│  • SRE — инженерные практики надёжности                    │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 15. Метрики и KPI

```
┌────────────────────────────────────────────────────────────┐
│                    МЕТРИКИ DEVOPS                          │
│                   (DORA Metrics)                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. DEPLOYMENT FREQUENCY                                  │
│     Как часто код попадает в production                  │

│     Low: реже раза в месяц                               │
│                                                            │
│  2. LEAD TIME FOR CHANGES                                 │
│     Время от коммита до production                       │
│     Elite: менее 1 часа                                  │
│     Low: более 6 месяцев                                 │
│                                                            │
│  3. CHANGE FAILURE RATE                                   │
│     Процент деплоев, вызвавших инцидент                  │
│     Elite: 0-15%                                         │
│     Low: 46-60%                                          │
│                                                            │
│  4. TIME TO RESTORE SERVICE                               │
│     Время восстановления после инцидента                 │
│     Elite: менее 1 часа                                  │
│     Low: более 6 месяцев                                 │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│                     МЕТРИКИ SRE                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  AVAILABILITY (Доступность)                               │
│  ─────────────────────────                                 │
│  Процент времени, когда сервис доступен.                 │
│                                                            │
│  99%    = 3.65 дня простоя в год                         │
│  99.9%  = 8.76 часов простоя в год                       │
│  99.99% = 52.6 минут простоя в год                       │
│  99.999%= 5.26 минут простоя в год ("five nines")        │
│                                                            │
│  LATENCY (Задержка)                                       │
│  ──────────────────                                        │
│  P50, P90, P95, P99 — перцентили времени ответа          │
│  P99 = 200ms означает: 99% запросов быстрее 200ms        │
│                                                            │
│  ERROR RATE (Частота ошибок)                              │
│  ───────────────────────────                               │
│  Процент запросов, завершившихся ошибкой                 │
│                                                            │
│  THROUGHPUT (Пропускная способность)                      │
│  ───────────────────────────────────                       │
│  Количество запросов в секунду (RPS)                     │
│                                                            │
│  ERROR BUDGET CONSUMPTION                                 │
│  ────────────────────────                                  │
│  Сколько из допустимого бюджета ошибок потрачено         │
│                                                            │
│  MTTR (Mean Time To Recovery)                             │
│  ────────────────────────────                              │
│  Среднее время восстановления после инцидента            │
│                                                            │
│  MTTD (Mean Time To Detect)                               │
│  ──────────────────────────                                │
│  Среднее время обнаружения инцидента                     │
│                                                            │
│  MTBF (Mean Time Between Failures)                        │
│  ─────────────────────────────────                         │
│  Среднее время между отказами                            │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│                   МЕТРИКИ SCRUM                            │
```
