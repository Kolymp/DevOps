## 1. Что такое GitOps?

### Определение

**GitOps** — это методология управления инфраструктурой и приложениями, где **Git репозиторий является единственным источником правды (Single Source of Truth)** для декларативного описания желаемого состояния системы.

```
│  • Автоматическая синхронизация                                   │
│  • Обнаружение и исправление drift                                │
```

### Pull vs Push модели

text

```
╔═══════════════════════════════════════════════════════════════════╗
║                    PUSH MODEL (традиционная)                       ║
╠═══════════════════════════════════════════════════════════════════╣
║                                                                    ║
║  CI/CD Pipeline                                                    ║
║       │                                                            ║
║       │  kubectl apply / helm upgrade                              ║
║       │  (push changes)                                            ║
║       ▼                                                            ║
║  Kubernetes Cluster                                                ║
║                                                                    ║
║  Недостатки:                                                       ║
║  • CI/CD система нужен доступ к кластеру (credentials)            ║
║  • Нет автоматической синхронизации при drift                     ║
║  • Сложнее управлять несколькими кластерами                       ║
║                                                                    ║
╚═══════════════════════════════════════════════════════════════════╝

╔═══════════════════════════════════════════════════════════════════╗
║                    PULL MODEL (GitOps)                             ║
╠═══════════════════════════════════════════════════════════════════╣
║                                                                    ║
║  Git Repository                                                    ║
║       ▲                                                            ║
║       │  poll / webhook                                            ║
║       │  (pull changes)                                            ║
║       │                                                            ║
║  GitOps Operator (в кластере)                                     ║
║       │                                                            ║
║       │  применяет изменения                                       ║
║       ▼                                                            ║
║  Kubernetes Cluster                                                ║
║                                                                    ║
║  Преимущества:                                                     ║
║  • Кластер тянет изменения сам (pull)                             ║
║  • Не нужно давать внешним системам доступ к кластеру             ║
║  • Автоматическая синхронизация и drift detection                 ║
║  • Легко масштабируется на много кластеров                        ║
║                                                                    ║
╚═══════════════════════════════════════════════════════════════════╝
```

### Workflow разработчика с GitOps

text

```
┌─────────────────────────────────────────────────────────────────────┐
│                  ТИПИЧНЫЙ GITOPS WORKFLOW                            │
└─────────────────────────────────────────────────────────────────────┘

1️⃣  РАЗРАБОТКА
   Developer пишет код приложения
        │
        ▼
   git commit + push в app repository
        │
        ▼

2️⃣  CI: СБОРКА
   CI pipeline (GitHub Actions, GitLab CI)
        │
        ├─ Запускает тесты
        ├─ Собирает Docker образ
        ├─ Публикует образ в registry (с тегом v1.2.3)
        │
        ▼

3️⃣  CD: ОБНОВЛЕНИЕ МАНИФЕСТОВ
   CI pipeline обновляет config repository
        │
        ├─ Клонирует config repo
        ├─ Обновляет image tag в манифестах
        │  (v1.2.2 → v1.2.3)
        ├─ git commit + push
        │
        ▼

4️⃣  GITOPS: СИНХРОНИЗАЦИЯ
   ArgoCD/Flux обнаруживает изменения в config repo
        │
        ├─ Сравнивает Git (желаемое) vs Cluster (текущее)
        ├─ Создаёт Sync план
        ├─ Применяет изменения к кластеру
        │
        ▼

5️⃣  ДЕПЛОЙ
   Kubernetes выполняет rolling update
        │
        ├─ Создаёт новые Pods с образом v1.2.3
        ├─ Проверяет readiness probes
        ├─ Переключает трафик
        ├─ Удаляет старые Pods
        │
        ▼

6️⃣  ВЕРИФИКАЦИЯ
   Мониторинг и логи
        │
        ├─ Prometheus собирает метрики
        ├─ Алерты при проблемах
        │
        ▼

7️⃣  ОТКАТ (при проблемах)
   Developer делает git revert
        │
        ├─ git revert <commit>
        ├─ git push
        │
        ▼
   ArgoCD автоматически применяет откат
```

## 2. Преимущества и недостатки GitOps

### Преимущества

text

```
╔═══════════════════════════════════════════════════════════════════╗
║                        ПРЕИМУЩЕСТВА GITOPS                         ║
╠═══════════════════════════════════════════════════════════════════╣
║                                                                    ║
║  ✅ ПОЛНАЯ ИСТОРИЯ ИЗМЕНЕНИЙ                                      ║
║     • Каждое изменение в Git — это commit                         ║
║     • Можно увидеть кто, когда и зачем изменил                   ║
║     • git log, git blame для аудита                               ║
║     • Compliance и регуляторные требования                        ║
║                                                                    ║
║  ✅ CODE REVIEW ПЕРЕД ДЕПЛОЕМ                                     ║

║     • Peer review конфигурации                                    ║
║     • Automated checks (линтеры, policy validation)               ║
║     • Снижение человеческих ошибок                                ║
║                                                                    ║
║  ✅ ПРОСТОЙ ОТКАТ                                                 ║
║     • git revert + push = автоматический откат                    ║
║     • Не нужно помнить как было "раньше"                          ║
║     • Откат за секунды вместо минут                               ║
║                                                                    ║
║  ✅ ДЕКЛАРАТИВНОСТЬ                                               ║
║     • Описываем "что должно быть", а не "как сделать"             ║

║                                                                    ║

║     • Изменения применяются автоматически                         ║
║     • Не нужно помнить применить манифесты                        ║
║     • Consistency между окружениями                               ║
║                                                                    ║
║  ✅ DRIFT DETECTION И ИСПРАВЛЕНИЕ                                 ║
║     • Автоматическое обнаружение ручных изменений                 ║
║     • Возврат к желаемому состоянию                               ║
║     • Кластер всегда соответствует Git                            ║
║                                                                    ║
║     Пример drift:                                                 ║
║     Developer: kubectl scale deployment myapp --replicas=10       ║

║  ✅ БЕЗОПАСНОСТЬ                                                  ║
║     • Кластер не выставлен наружу (pull model)                    ║
║     • CI/CD системе не нужны credentials к кластеру               ║
║     • RBAC для контроля кто может деплоить                        ║
║     • Signed commits для верификации                              ║
║                                                                    ║
║  ✅ DISASTER RECOVERY                                             ║
║     • Полное состояние кластера в Git                             ║
║     • Восстановление кластера = git clone + ArgoCD sync           ║
║     • Тестирование DR сценариев на staging                        ║
║                                                                    ║
║  ✅ MULTI-CLUSTER / MULTI-TENANT                                  ║
║     • Один GitOps оператор для множества кластеров                ║
║     • Consistency конфигурации между кластерами                   ║
║     • Централизованное управление                                 ║
║                                                                    ║
║  ✅ РАЗДЕЛЕНИЕ ОТВЕТСТВЕННОСТИ                                    ║

║     • Чёткие границы и ownership                                  ║
║                                                                    ║
║  ✅ OBSERVABILITY                                                 ║
║     • Видимость что задеплоено и когда                            ║
║     • Статус синхронизации в UI                                   ║
║     • История деплоев                                             ║
```

### Недостатки

text

```
╔═══════════════════════════════════════════════════════════════════╗
║                        НЕДОСТАТКИ GITOPS                           ║
╠═══════════════════════════════════════════════════════════════════╣
║                                                                    ║
║  ❌ НАЧАЛЬНАЯ СЛОЖНОСТЬ                                           ║
║     • Нужно установить и настроить GitOps оператор                ║
║     • Изучение новых концепций и инструментов                     ║
║     • Настройка CI/CD для обновления манифестов                   ║
║     • Кривая обучения для команды                                 ║
║                                                                    ║
║     Время на внедрение: 1-4 недели для команды                    ║
║                                                                    ║
║  ❌ ДОПОЛНИТЕЛЬНАЯ ИНФРАСТРУКТУРА                                 ║
║     • ArgoCD/Flux работают в кластере (потребляют ресурсы)        ║
║     • Нужен отдельный Git репозиторий для манифестов              ║
║     • Дополнительные компоненты для мониторинга                   ║
║                                                                    ║
║     Ресурсы ArgoCD: ~500MB RAM, ~100m CPU                         ║
║                                                                    ║
║  ❌ ЗАДЕРЖКА ДЕПЛОЯ                                               ║
║     • Polling: задержка до следующей проверки (обычно 3 мин)      ║
║     • Webhook: требует настройки и доступности                    ║
║     • Два шага: CI обновляет Git → ArgoCD синхронизирует          ║
║                                                                    ║
║     Типичная задержка: 30 сек - 3 минуты                          ║
║     vs kubectl apply: мгновенно                                   ║
║                                                                    ║
║  ❌ СЛОЖНОСТИ С СЕКРЕТАМИ                                         ║
║     • Нельзя хранить секреты в plain text в Git                   ║
║     • Нужно использовать Sealed Secrets, SOPS, External Secrets   ║
║     • Дополнительная настройка и инструменты                      ║
║                                                                    ║
║  ❌ ОТЛАДКА СЛОЖНЕЕ                                               ║
║     • Изменения не мгновенные (из-за polling)                     ║
║     • Дополнительный слой абстракции                              ║
║     • Нужно смотреть логи ArgoCD для диагностики                  ║
║                                                                    ║
║  ❌ УПРАВЛЕНИЕ СОСТОЯНИЕМ                                         ║
║     • Не всё можно описать декларативно                           ║
║     • Миграции БД, одноразовые Jobs                               ║
║     • StatefulSets с persistent data                              ║
║                                                                    ║
║  ❌ GIT КАК BOTTLENECK                                            ║
║     • Git должен быть доступен всегда                             ║
║     • Если Git недоступен — нет деплоев                           ║
║     • Нужна высокая доступность Git (GitLab HA, GitHub Enterprise)│
║                                                                    ║
║  ❌ РАЗМЕР РЕПОЗИТОРИЯ                                            ║
║     • Большие бинарные файлы (Helm charts с зависимостями)        ║
║     • История может разрастись                                    ║
║     • Медленное клонирование                                      ║
║                                                                    ║
║  ❌ КОНФЛИКТЫ ПРИ ПАРАЛЛЕЛЬНЫХ ИЗМЕНЕНИЯХ                         ║
║     • Несколько людей меняют одни файлы                           ║
║     • Merge conflicts в YAML                                      ║
║     • Нужна координация между командами                           ║
║                                                                    ║
╚═══════════════════════════════════════════════════════════════════╝
```

### Настройка Ingress для ArgoCD

YAML

```
# argocd-ingress.yaml
#
# Ingress для доступа к ArgoCD через доменное имя

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: argocd-server-ingress
  namespace: argocd
  annotations:
    # Для nginx-ingress
    nginx.ingress.kubernetes.io/ssl-redirect: "true"
    nginx.ingress.kubernetes.io/backend-protocol: "HTTPS"
    
    # Для cert-manager (автоматический TLS)
    cert-manager.io/cluster-issuer: letsencrypt-prod
    
    # Увеличить таймауты для long-running sync
    nginx.ingress.kubernetes.io/proxy-connect-timeout: "300"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "300"
    nginx.ingress.kubernetes.io/proxy-read-timeout: "300"

spec:
  ingressClassName: nginx  # Или ваш Ingress Controller
  
  rules:
    - host: argocd.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: argocd-server
                port:
                  name: https
  
  tls:
    - hosts:
        - argocd.example.com
      secretName: argocd-tls  # Cert-manager создаст этот Secret
```

```
  # - url: git@github.com:myorg/my-private-repo.git

  #     name: repo-ssh-key
  #     key: sshPrivateKey
  
  # Приватный репозиторий через HTTPS с токеном
  # - url: https://github.com/myorg/my-private-repo
  #   type: git
  #   passwordSecret:
  #     name: repo-credentials
  #     key: password
  #   usernameSecret:
  #     name: repo-credentials
  #     key: username
  
  # Helm репозиторий
  # - url: https://charts.bitnami.com/bitnami
  #   type: helm
  #   name: bitnami
  
  
  # ===== GIT TIMEOUT =====
  # Таймаут для Git операций
  timeout.reconciliation: 180s
  
  
  # ===== RESOURCE TRACKING =====
  # Метод отслеживания ресурсов ArgoCD
  # annotation — через аннотации (по умолчанию)
  # label — через метки
  # annotation+label — оба способа
  application.resourceTrackingMethod: annotation+label
  
  
  # ===== DIFF CUSTOMIZATION =====
  # Игнорировать определенные поля при сравнении
  resource.customizations: |
    # Игнорировать поле replicas в HPA (управляется HPA, не Git)
    autoscaling/HorizontalPodAutoscaler:
      ignoreDifferences: |
        jsonPointers:
        - /spec/replicas
    
    # Игнорировать status для всех ресурсов
    all:
      ignoreDifferences: |
        managedFieldsManagers:
        - kube-controller-manager
  
  
  # ===== HEALTH ASSESSMENT =====
  # Кастомные health checks
  resource.customizations.health.argoproj.io_Application: |
    hs = {}
    hs.status = "Progressing"
    hs.message = ""
    if obj.status ~= nil then
      if obj.status.health ~= nil then
        hs.status = obj.status.health.status
        if obj.status.health.message ~= nil then
          hs.message = obj.status.health.message
        end
      end
    end
    return hs
  
  
  # ===== URL =====
  # URL ArgoCD (для callback, webhooks)
  url: https://argocd.example.com
  
  
  # ===== DEXCONFIG (SSO) =====
  # Настройка SSO через Dex
  # dex.config: |

  #       name: GitHub

  #         clientSecret: $dex.github.clientSecret
  #         orgs:
  #         - name: myorganization
```

### Создание первого Application в ArgoCD

ArgoCD Application — это CRD (Custom Resource Definition), который описывает какое приложение из какого Git репозитория деплоить в какой namespace.

YAML

```
# argocd-apps/myapp.yaml
#
# ArgoCD Application манифест

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  # Имя Application в ArgoCD
  name: myapp
  
  # Application обычно создаётся в namespace argocd
  namespace: argocd
  
  # Finalizer гарантирует что ресурсы будут удалены при удалении Application
  finalizers:
    - resources-finalizer.argocd.argoproj.io

spec:
  # ===== PROJECT =====
  # ArgoCD Project для группировки приложений и RBAC
  # "default" — проект по умолчанию
  project: default
  
  # ===== SOURCE =====
  # Откуда брать манифесты
  source:
    # URL Git репозитория
    repoURL: https://github.com/myorg/k8s-configs
    
    # Ветка / тег / commit SHA
    targetRevision: HEAD  # HEAD = последний commit в default ветке
    
    # Путь к директории с манифестами
    path: apps/myapp
    
    # Опционально: для Helm чартов
    # helm:
    #   valueFiles:
    #     - values.yaml
    #   parameters:
    #     - name: image.tag
    #       value: "1.2.3"
    
    # Опционально: для Kustomize
    # kustomize:
    #   namePrefix: prod-
    #   commonLabels:
    #     env: production
  
  # ===== DESTINATION =====
  # Куда деплоить
  destination:
    # URL Kubernetes API server
    # https://kubernetes.default.svc — тот же кластер где ArgoCD
    server: https://kubernetes.default.svc
    
    # Namespace для деплоя приложения
    namespace: default
  
  # ===== SYNC POLICY =====
  # Политика синхронизации
  syncPolicy:
    # Автоматическая синхронизация (опционально)
    automated:
      # Автоматически применять изменения из Git
      selfHeal: true
      
      # Автоматически удалять ресурсы, которых нет в Git
      prune: true
      
      # Разрешить пустые (empty) commits
      allowEmpty: false
    
    # Опции синхронизации
    syncOptions:
      # Создать namespace если не существует
      - CreateNamespace=true
      
      # Проверить ресурсы перед sync
      - Validate=true
      
      # Использовать kubectl apply вместо create/patch
      - ApplyOutOfSyncOnly=false
      
      # Уважать resource hooks (pre-sync, post-sync)
      - RespectIgnoreDifferences=true
    
    # Retry policy при ошибках sync
    retry:
      # Максимум попыток
      limit: 5
      
      # Backoff между попытками
      backoff:
        duration: 5s        # Начальная задержка
        factor: 2           # Множитель (5s, 10s, 20s, ...)
        maxDuration: 3m     # Максимальная задержка
  
  # ===== IGNORE DIFFERENCES =====
  # Игнорировать определённые различия между Git и кластером
  ignoreDifferences:
    # Игнорировать replicas для Deployment (если управляется HPA)
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/replicas
    
    # Игнорировать всё поле status
    - group: "*"
      kind: "*"
      jsonPointers:
        - /status
  
  # ===== INFO =====
  # Дополнительная информация для UI
  info:
    - name: Description
      value: "My first ArgoCD application"
    - name: Owner
      value: "platform-team@example.com"
```

```
  # Куда можно деплоить
  destinations:
    # Разрешить деплой в backend namespace
    - namespace: backend-dev
      server: https://kubernetes.default.svc
    
    - namespace: backend-staging
      server: https://kubernetes.default.svc
    
    - namespace: backend-production
      server: https://kubernetes.default.svc

    # - namespace: 'backend-*'
    #   server: https://kubernetes.default.svc

  clusterResourceWhitelist:
    - group: ''
      kind: Namespace  # Разрешить создавать Namespaces

  # clusterResourceBlacklist:
  #   - group: ''

    - group: ''

    - group: 'apps'

    - name: developer

        - backend-developers  # SSO группа

        - backend-admins

    - kind: deny

        - '*-production'
      namespaces:
        - backend-production
      clusters:
        - https://kubernetes.default.svc

    - kind: allow

      duration: 8h
      manualSync: true  # Разрешить ручной sync вне окна
  
  # ===== ORPHANED RESOURCES =====
  # Отслеживать "осиротевшие" ресурсы (созданные вручную в namespace)
  orphanedResources:
    warn: true  # Показывать предупреждение
    # ignore:   # Игнорировать определённые ресурсы
    #   - group: ''
    #     kind: ConfigMap
    #     name: kube-root-ca.crt
```

```
  # ===== POLICY CSV =====
  # Формат: p, subject, resource, action, object, effect
  
  policy.csv: |
    # Роль: read-only для всех приложений
    p, role:readonly, applications, get, */*, allow
    p, role:readonly, repositories, get, *, allow
    p, role:readonly, clusters, get, *, allow
    
    # Роль: developer может sync, но не delete
    p, role:developer, applications, get, */*, allow
    p, role:developer, applications, sync, */*, allow
    p, role:developer, applications, override, */*, allow
    p, role:developer, logs, get, */*, allow
    
    # Роль: admin — полный доступ
    p, role:admin, *, *, */*, allow
    
    # Привязка ролей к пользователям/группам
    g, alice@example.com, role:admin
    g, bob@example.com, role:developer
    
    # Привязка к SSO группам
    g, platform-team, role:admin
    g, developers, role:developer
    g, qa-team, role:readonly
  
  # ===== DEFAULT POLICY =====
  # Политика по умолчанию для неаутентифицированных пользователей
  # '' — нет доступа (по умолчанию)
  # 'role:readonly' — доступ на чтение
  policy.default: ''
  
  # ===== SCOPES =====
  # Какие scopes использовать из JWT токена для групп
  scopes: '[groups, email]'
```

### GitHub Actions + ArgoCD

YAML

```
# .github/workflows/deploy.yml
#
# GitHub Actions workflow для CI/CD с ArgoCD

name: Build and Deploy

on:
  push:
    branches:
      - main
      - develop
  pull_request:
    branches:
      - main

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ===== JOB 1: BUILD AND PUSH IMAGE =====
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
      
      - name: Log in to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      
      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=ref,event=branch
            type=ref,event=pr
            type=sha,prefix={{branch}}-
            type=semver,pattern={{version}}
      
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
  
  # ===== JOB 2: UPDATE MANIFESTS =====
  update-manifests:
    needs: build
    runs-on: ubuntu-latest
    
    # Только для main ветки (не для PR)
    if: github.ref == 'refs/heads/main'
    
    steps:
      - name: Checkout config repository
        uses: actions/checkout@v4
        with:
          # Отдельный репозиторий с манифестами
          repository: myorg/k8s-configs
          token: ${{ secrets.CONFIG_REPO_TOKEN }}
          ref: main
      
      - name: Update image tag in Kustomize
        run: |
          # Извлекаем короткий SHA
          IMAGE_TAG="${GITHUB_SHA::8}"
          
          # Обновляем kustomization.yaml
          cd apps/myapp/overlays/production
          
          # Используем kustomize edit set image
          kustomize edit set image myapp=ghcr.io/${{ github.repository }}:main-${IMAGE_TAG}
      
      - name: Commit and push changes
        run: |
          git config user.name "GitHub Actions"
          git config user.email "actions@github.com"
          
          git add apps/myapp/overlays/production/kustomization.yaml
          git commit -m "Update myapp image to ${GITHUB_SHA::8}"
          git push
      
      # ArgoCD автоматически обнаружит изменения и синхронизирует
  
  # ===== JOB 3: WAIT FOR ARGOCD SYNC (опционально) =====
  wait-for-sync:
    needs: update-manifests
    runs-on: ubuntu-latest
    
    steps:
      - name: Install ArgoCD CLI
        run: |
          curl -sSL -o /usr/local/bin/argocd \
            https://github.com/argoproj/argo-cd/releases/latest/download/argocd-linux-amd64
          chmod +x /usr/local/bin/argocd
      
      - name: Login to ArgoCD
        run: |
          argocd login ${{ secrets.ARGOCD_SERVER }} \
            --username ${{ secrets.ARGOCD_USERNAME }} \
            --password ${{ secrets.ARGOCD_PASSWORD }} \
            --insecure
      
      - name: Wait for sync
        run: |
          # Ждём синхронизации (максимум 10 минут)
          argocd app wait myapp-production \
            --sync \
            --health \
            --timeout 600
      
      - name: Check application health
        run: |
          # Проверяем что приложение здорово
          argocd app get myapp-production
```

### Sync Waves и Hooks

Sync Waves позволяют контролировать порядок деплоя ресурсов.

```
# ===== SYNC WAVES =====

  name: myapp

# Wave 0: ConfigMap и Secrets (по умолчанию)

# Wave 1: Database

  name: postgres
  namespace: myapp
  annotations:
    argocd.argoproj.io/sync-wave: "1"
spec:
  # ... StatefulSet spec
---
# Wave 2: Application (после database)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: myapp
  annotations:
    argocd.argoproj.io/sync-wave: "2"
spec:
  # ... Deployment spec
---
# Wave 3: Ingress (после application)
apiVersion: networking.k8s.io/v1

  name: myapp

  # ... Ingress spec
```

```
# ===== HOOKS =====

  name: db-migration
  namespace: myapp
```

### Resource Health Checks

Настройка кастомных health checks для CRDs и особых ресурсов.

YAML

```
# argocd-cm.yaml — добавить кастомные health checks

apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-cm
  namespace: argocd
data:
  # ===== CUSTOM HEALTH CHECKS =====
  
  # Health check для Kafka Topic (Strimzi)
  resource.customizations.health.kafka.strimzi.io_KafkaTopic: |
    hs = {}
    if obj.status ~= nil then
      if obj.status.conditions ~= nil then
        for i, condition in ipairs(obj.status.conditions) do
          if condition.type == "Ready" then
            if condition.status == "True" then
              hs.status = "Healthy"
              hs.message = "Topic is ready"
            else
              hs.status = "Degraded"
              hs.message = condition.message
            end
            return hs
          end
        end
      end
    end
    hs.status = "Progressing"
    hs.message = "Waiting for topic to be ready"
    return hs
  
  # Health check для Certificate (cert-manager)
  resource.customizations.health.cert-manager.io_Certificate: |
    hs = {}
    if obj.status ~= nil then
      if obj.status.conditions ~= nil then
        for i, condition in ipairs(obj.status.conditions) do
          if condition.type == "Ready" then
            if condition.status == "True" then
              hs.status = "Healthy"
              hs.message = "Certificate is ready"
            else
              hs.status = "Degraded"
              hs.message = condition.message
            end
            return hs
          end
        end
      end
    end
    hs.status = "Progressing"
    hs.message = "Waiting for certificate"
    return hs
  
  # Health check для Rollout (Argo Rollouts)
  resource.customizations.health.argoproj.io_Rollout: |
    hs = {}
    if obj.status ~= nil then
      if obj.status.phase == "Healthy" then
        hs.status = "Healthy"
      elseif obj.status.phase == "Paused" then
        hs.status = "Suspended"
        hs.message = "Rollout is paused"
      elseif obj.status.phase == "Degraded" then
        hs.status = "Degraded"
      else
        hs.status = "Progressing"
      end
    end
    return hs
  
  # ===== CUSTOM ACTIONS =====
  # Кастомные действия для ресурсов
  
  resource.customizations.actions.argoproj.io_Rollout: |
    discovery.lua: |
      actions = {}
      actions["resume"] = {["disabled"] = false}
      actions["abort"] = {["disabled"] = false}
      actions["retry"] = {["disabled"] = false}
      return actions
    definitions:
    - name: resume
      action.lua: |
        obj.status.pauseConditions = nil
        return obj
    - name: abort
      action.lua: |
        obj.status.abort = true
        return obj
    - name: retry
      action.lua: |
        obj.status.abort = nil
        if obj.status.pauseConditions ~= nil then
          obj.status.pauseConditions = nil
        end
        return obj
```

## 8. Troubleshooting

### Частые проблемы и решения

Bash

```
# ===== ПРОБЛЕМА 1: APPLICATION STUCK IN "SYNCING" =====

# Проверить статус sync
argocd app get myapp

# Посмотреть ресурсы в процессе sync
argocd app resources myapp

# Проверить логи repo-server
kubectl logs -n argocd deployment/argocd-repo-server -f

# Проверить логи application-controller
kubectl logs -n argocd deployment/argocd-application-controller -f

# Принудительно refresh
argocd app get myapp --refresh

# Hard refresh (очистить кэш)
argocd app get myapp --hard-refresh


# ===== ПРОБЛЕМА 2: OUT OF SYNC НО РЕСУРСЫ ИДЕНТИЧНЫ =====

# Причина: drift в полях, управляемых Kubernetes (status, metadata.resourceVersion, etc.)

# Посмотреть diff
argocd app diff myapp

# Решение 1: Игнорировать определённые поля
# В Application spec:
# ignoreDifferences:
#   - group: apps
#     kind: Deployment
#     jsonPointers:
#       - /spec/replicas

# Решение 2: В argocd-cm.yaml
kubectl edit configmap argocd-cm -n argocd
# Добавить:
# data:
#   resource.customizations: |
#     apps/Deployment:
#       ignoreDifferences: |
#         jsonPointers:
#         - /spec/replicas


# ===== ПРОБЛЕМА 3: CANNOT CONNECT TO REPOSITORY =====

# Проверить репозитории
argocd repo list

# Тестировать подключение
argocd repo get https://github.com/myorg/k8s-configs

# Проверить credentials
kubectl get secret -n argocd -l argocd.argoproj.io/secret-type=repository

# Обновить credentials
argocd repo add https://github.com/myorg/k8s-configs \
  --username git \
  --password <token>

# Для SSH
argocd repo add git@github.com:myorg/k8s-configs \
  --ssh-private-key-path ~/.ssh/id_rsa


# ===== ПРОБЛЕМА 4: HELM CHART NOT FOUND =====

# Проверить что Helm репозиторий добавлен
argocd repo list

# Добавить Helm репозиторий
argocd repo add https://charts.bitnami.com/bitnami --type helm --name bitnami

# Обновить индекс репозитория
# ArgoCD делает это автоматически, но можно форсировать через refresh


# ===== ПРОБЛЕМА 5: SYNC FAILED - RBAC ERROR =====

# Проверить права ArgoCD ServiceAccount
kubectl auth can-i create deployments \
  --as system:serviceaccount:argocd:argocd-application-controller \
  -n target-namespace

# Проверить ClusterRole
kubectl describe clusterrole argocd-application-controller

# Добавить права если нужно
kubectl edit clusterrole argocd-application-controller


# ===== ПРОБЛЕМА 6: RESOURCE IS NOT BEING TRACKED =====

# Причина: ресурс создан без label'ов ArgoCD

# Проверить labels на ресурсе
kubectl get deployment myapp -o jsonpath='{.metadata.labels}'

# Должны быть:
# app.kubernetes.io/instance: <application-name>

# Решение: удалить ресурс и пересинхронизировать
kubectl delete deployment myapp -n target-namespace
argocd app sync myapp


# ===== ПРОБЛЕМА 7: APPLICATION HEALTH UNKNOWN =====

# Проверить какие ресурсы Unknown
argocd app resources myapp

# Проверить health check для CRD
kubectl get configmap argocd-cm -n argocd -o yaml | grep -A 50 "resource.customizations.health"

# Добавить custom health check для CRD
# (см. раздел Resource Health Checks)


# ===== ПРОБЛЕМА 8: WEBHOOK NOT TRIGGERING SYNC =====

# Проверить webhook конфигурацию в Git (GitHub/GitLab)
# Payload URL: https://argocd.example.com/api/webhook

# Проверить логи argocd-server
kubectl logs -n argocd deployment/argocd-server | grep webhook

# Проверить что webhook secret настроен

# ===== ПРОБЛЕМА 9: ARGOCD UI NOT LOADING =====

# Проверить логи argocd-server
```

### Naming Conventions

YAML

```
# ===== NAMING CONVENTIONS =====

# Application names
# Формат: {app}-{environment}
# Примеры:
# - backend-api-production
# - frontend-staging
# - monitoring-production

# Namespace names
# Формат: {team}-{app} или {app}-{environment}
# Примеры:
# - team-backend
# - myapp-production

# Labels (стандартные Kubernetes labels)
# https://kubernetes.io/docs/concepts/overview/working-with-objects/common-labels/

metadata:
  labels:
    # Имя приложения
    app.kubernetes.io/name: backend-api
    
    # Instance (часто = release name)
    app.kubernetes.io/instance: backend-api-production
    
    # Версия приложения
    app.kubernetes.io/version: "1.2.3"
    
    # Компонент
    app.kubernetes.io/component: api
    
    # Часть большей системы
    app.kubernetes.io/part-of: my-platform
    
    # Инструмент управления
    app.kubernetes.io/managed-by: argocd
    
    # Кастомные labels
    team: backend
    environment: production
    tier: backend
```

### Security Best Practices

YAML

```
# ===== SECURITY CHECKLIST =====

# 1. RBAC: Минимальные права
# - Используйте Projects для ограничения доступа
# - Настройте роли с минимальными правами
# - Не используйте default project для production

# 2. Секреты: Никогда в plain text
# - Используйте Sealed Secrets, External Secrets, или SOPS
# - Регулярно ротируйте секреты
# - Аудит доступа к секретам

# 3. Кластеры: Ограничение доступа
# - Используйте отдельные credentials для каждого кластера
# - Регулярно ротируйте cluster credentials
# - Мониторинг доступа

# 4. Git: Защита репозитория
# - Branch protection rules
# - Обязательный code review
# - Signed commits
# - Регулярный аудит доступа

# 5. ArgoCD: Безопасная конфигурация
---
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-cm
  namespace: argocd
data:
  # Отключить создание admin аккаунта после настройки SSO
  admin.enabled: "false"
  
  # Таймаут сессии
  server.sessionDuration: "24h"
  
  # Ограничить anonymous доступ
  users.anonymous.enabled: "false"
  
  # Включить аудит
  audit.enabled: "true"
---
# RBAC: строгие политики
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-rbac-cm
  namespace: argocd
data:
  # Политика по умолчанию — нет доступа
  policy.default: ""
  
  # Явные права для каждой роли
  policy.csv: |
    # Только чтение для всех
    p, role:readonly, applications, get, */*, allow
    
    # Разработчики могут sync в свои проекты
    p, role:developer, applications, sync, team-*/*, allow
    
    # Админы имеют полный доступ
    p, role:admin, *, *, */*, allow
    
    # Привязка групп
    g, developers, role:developer
    g, platform-team, role:admin
```

## 10. Полная шпаргалка команд ArgoCD

Bash

```
# ===== УСТАНОВКА И НАСТРОЙКА =====

# Установка ArgoCD
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# Получить начальный пароль
kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}" | base64 -d

# Port-forward для доступа к UI
kubectl port-forward svc/argocd-server -n argocd 8080:443

# Логин через CLI
argocd login localhost:8080 --username admin --password <password> --insecure

# Смена пароля
argocd account update-password


# ===== РЕПОЗИТОРИИ =====

# Добавить Git репозиторий
argocd repo add https://github.com/myorg/k8s-configs

# Добавить приватный репозиторий (HTTPS)
argocd repo add https://github.com/myorg/k8s-configs --username git --password <token>

# Добавить приватный репозиторий (SSH)
argocd repo add git@github.com:myorg/k8s-configs --ssh-private-key-path ~/.ssh/id_rsa

# Добавить Helm репозиторий
argocd repo add https://charts.bitnami.com/bitnami --type helm --name bitnami

# Список репозиториев
argocd repo list

# Удалить репозиторий
argocd repo rm https://github.com/myorg/k8s-configs


# ===== КЛАСТЕРЫ =====

# Добавить кластер
argocd cluster add <context-name>

# Список кластеров
argocd cluster list

# Удалить кластер
argocd cluster rm https://prod-k8s.example.com


# ===== APPLICATIONS =====

# Создать Application
argocd app create myapp \
  --repo https://github.com/myorg/k8s-configs \
  --path apps/myapp \
  --dest-server https://kubernetes.default.svc \
  --dest-namespace default

# Список Applications
argocd app list

# Детали Application
argocd app get myapp

# Синхронизация
argocd app sync myapp

# Sync с prune
argocd app sync myapp --prune

# Dry-run sync
argocd app sync myapp --dry-run

# Refresh (проверить Git на изменения)
argocd app get myapp --refresh

# Hard refresh (очистить кэш)
argocd app get myapp --hard-refresh

# Diff (показать различия)
argocd app diff myapp

# История синхронизаций
argocd app history myapp

# Rollback
argocd app rollback myapp <revision>

# Удалить Application (без ресурсов)
argocd app delete myapp --cascade=false

# Удалить Application с ресурсами
argocd app delete myapp --cascade=true

# Логи приложения
argocd app logs myapp

# Ресурсы приложения
argocd app resources myapp

# Манифесты
argocd app manifests myapp


# ===== PROJECTS =====

# Создать Project
argocd proj create myproject

# Список Projects
argocd proj list

# Детали Project
argocd proj get myproject

# Добавить разрешённый репозиторий
argocd proj add-source myproject https://github.com/myorg/k8s-configs

# Добавить разрешённый destination
argocd proj add-destination myproject https://kubernetes.default.svc myapp

# Удалить Project
argocd proj delete myproject


# ===== ACCOUNTS =====

# Список аккаунтов
argocd account list

# Получить информацию о текущем пользователе
argocd account get-user-info

# Сгенерировать токен
argocd account generate-token --account <account-name>

# Проверить права
argocd account can-i sync applications '*'


# ===== ADMIN =====

# Экспорт всех Applications
argocd admin export > backup.yaml

# Импорт Applications
argocd admin import < backup.yaml

# Получить cluster settings
argocd admin cluster kubeconfig https://kubernetes.default.svc
```
