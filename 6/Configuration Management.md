## 1. Что такое Configuration Management?

text

```
┌────────────────────────────────────────────────────────────┐
│         CONFIGURATION MANAGEMENT (CM)                      │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Configuration Management — автоматизация управления       │
│  конфигурацией серверов и инфраструктуры                  │
│                                                            │
│  ПРОБЛЕМА БЕЗ CM:                                         │
│  ────────────────                                          │
│  • 100 серверов → 100 раз руками устанавливать ПО         │
│  • Конфигурация различается между серверами               │
│  • "Snowflake servers" — каждый сервер уникален           │
│  • Невозможно воспроизвести окружение                     │
│  • Документация устаревает                                │
│  • Человеческие ошибки                                    │
│                                                            │
│  РЕШЕНИЕ С CM:                                            │
│  ─────────────                                             │
│  • Инфраструктура как код (Infrastructure as Code)        │
│  • Декларативное описание желаемого состояния             │
│  • Автоматическое применение конфигурации                 │
│  • Версионирование (Git)                                  │
│  • Идемпотентность (можно запускать повторно)             │
│  • Воспроизводимость                                      │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐ │
│  │              БЕЗ CM vs С CM                          │ │
│  ├──────────────────────────────────────────────────────┤ │
│  │                                                      │ │
│  │  БЕЗ CM:                                             │ │
│  │  ───────                                             │ │
│  │  1. SSH на сервер                                    │ │
│  │  2. sudo apt install nginx                           │ │
│  │  3. vim /etc/nginx/nginx.conf (ручное редактирование)│ │
│  │  4. systemctl restart nginx                          │ │
│  │  5. Повторить 100 раз для всех серверов             │ │
│  │  6. Надеяться, что ничего не забыл                   │ │
│  │                                                      │ │
│  │  С CM (например, Ansible):                           │ │
│  │  ─────────────────────────                           │ │
│  │  1. Описать желаемое состояние в playbook:          │ │
│  │     - nginx установлен                               │ │
│  │     - конфиг из шаблона                              │ │
│  │     - сервис запущен                                 │ │
│  │  2. ansible-playbook nginx.yml                       │ │
│  │  3. Применится на всех серверах автоматически        │ │
│  │                                                      │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  ОСНОВНЫЕ КОНЦЕПЦИИ:                                      │
│  ───────────────────                                       │
│  • Идемпотентность — можно запускать многократно          │
│  • Декларативность — описываем ЧТО, а не КАК              │
│  • Конвергенция — приведение к желаемому состоянию        │
│  • Drift detection — обнаружение расхождений              │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**Пример Ansible playbook:**

YAML

```
# ═══════════════════════════════════════════════════════════
#          ANSIBLE PLAYBOOK — nginx.yml
# ═══════════════════════════════════════════════════════════

---
# Плейбук для установки и настройки Nginx
- name: Configure Nginx web servers
  hosts: webservers  # Группа серверов из inventory
  become: yes        # Выполнять с sudo
  
  vars:
    nginx_port: 80
    server_name: example.com
  
  tasks:
    # ───────────────────────────────────────────────────────
    # 1. Установить Nginx
    # ───────────────────────────────────────────────────────
    - name: Install Nginx
      apt:
        name: nginx
        state: present
        update_cache: yes
      # Идемпотентно: если уже установлен — ничего не делает
    
    # ───────────────────────────────────────────────────────
    # 2. Скопировать конфиг из шаблона
    # ───────────────────────────────────────────────────────
    - name: Configure Nginx
      template:
        src: templates/nginx.conf.j2
        dest: /etc/nginx/nginx.conf
        owner: root
        group: root
        mode: '0644'
      notify: Restart Nginx  # Триггер handler'а
    
    # ───────────────────────────────────────────────────────
    # 3. Скопировать site конфиг
    # ───────────────────────────────────────────────────────
    - name: Configure site
      template:
        src: templates/site.conf.j2
        dest: /etc/nginx/sites-available/{{ server_name }}
      notify: Restart Nginx
    
    # ───────────────────────────────────────────────────────
    # 4. Создать симлинк (enabled)
    # ───────────────────────────────────────────────────────
    - name: Enable site
      file:
        src: /etc/nginx/sites-available/{{ server_name }}
        dest: /etc/nginx/sites-enabled/{{ server_name }}
        state: link
      notify: Restart Nginx
    
    # ───────────────────────────────────────────────────────
    # 5. Убедиться, что Nginx запущен
    # ───────────────────────────────────────────────────────
    - name: Start and enable Nginx
      service:
        name: nginx
        state: started
        enabled: yes
    
    # ───────────────────────────────────────────────────────
    # 6. Открыть порт в firewall
    # ───────────────────────────────────────────────────────
    - name: Allow HTTP in firewall
      ufw:
        rule: allow
        port: '{{ nginx_port }}'
        proto: tcp
  
  # ─────────────────────────────────────────────────────────
  # Handlers — выполняются только если были изменения
  # ─────────────────────────────────────────────────────────
  handlers:
    - name: Restart Nginx
      service:
        name: nginx
        state: restarted
```

**Запуск:**

Bash

```
# Выполнить playbook
ansible-playbook -i inventory nginx.yml

# С дополнительными переменными
ansible-playbook -i inventory nginx.yml \
  -e "nginx_port=8080" \
  -e "server_name=myapp.com"

# Проверить без применения (dry-run)
ansible-playbook -i inventory nginx.yml --check
```

**Пример Terraform:**

hcl

```
# ═══════════════════════════════════════════════════════════
#          TERRAFORM — main.tf
# ═══════════════════════════════════════════════════════════

# ───────────────────────────────────────────────────────────
# Provider (AWS в данном случае)
# ───────────────────────────────────────────────────────────
provider "aws" {
  region = "us-east-1"
}

# ───────────────────────────────────────────────────────────
# VPC (виртуальная сеть)
# ───────────────────────────────────────────────────────────
resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
  
  tags = {
    Name = "main-vpc"
  }
}

# ───────────────────────────────────────────────────────────
# Subnet
# ───────────────────────────────────────────────────────────
resource "aws_subnet" "public" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.1.0/24"
  availability_zone = "us-east-1a"
  
  tags = {
    Name = "public-subnet"
  }
}

# ───────────────────────────────────────────────────────────
# Security Group (firewall)
# ───────────────────────────────────────────────────────────
resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Allow HTTP and SSH"
  vpc_id      = aws_vpc.main.id
  
  # Разрешить входящий HTTP
  ingress {
    from_port   = 80
    to_port     = 80
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  # Разрешить входящий SSH
  ingress {
    from_port   = 22
    to_port     = 22
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
  
  # Разрешить весь исходящий трафик
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

# ───────────────────────────────────────────────────────────
# EC2 Instance (виртуальная машина)
# ───────────────────────────────────────────────────────────
resource "aws_instance" "web" {
  count         = 3  # Создать 3 сервера
  ami           = "ami-0c55b159cbfafe1f0"  # Ubuntu
  instance_type = "t2.micro"
  subnet_id     = aws_subnet.public.id
  
  vpc_security_group_ids = [aws_security_group.web.id]
  
  tags = {
    Name = "web-server-${count.index + 1}"
  }
  
  # User data — скрипт при создании
  user_data = <<-EOF
              #!/bin/bash
              apt-get update
              apt-get install -y nginx
              systemctl start nginx
              EOF
}

# ───────────────────────────────────────────────────────────
# Load Balancer
# ───────────────────────────────────────────────────────────
resource "aws_lb" "main" {
  name               = "web-lb"
  internal           = false
  load_balancer_type = "application"
  security_groups    = [aws_security_group.web.id]
  subnets            = [aws_subnet.public.id]
}

# ───────────────────────────────────────────────────────────
# Output (вывести результаты)
# ───────────────────────────────────────────────────────────
output "instance_ips" {
  value = aws_instance.web[*].public_ip
}

output "lb_dns" {
  value = aws_lb.main.dns_name
}
```

**Пример Chef recipe:**

Ruby

```
# ═══════════════════════════════════════════════════════════
#          CHEF RECIPE — nginx/recipes/default.rb
# ═══════════════════════════════════════════════════════════

# ───────────────────────────────────────────────────────────
# Установить Nginx
# ───────────────────────────────────────────────────────────
package 'nginx' do
  action :install
end

# ───────────────────────────────────────────────────────────
# Шаблон конфигурации
# ───────────────────────────────────────────────────────────
template '/etc/nginx/nginx.conf' do
  source 'nginx.conf.erb'
  owner 'root'
  group 'root'
  mode '0644'
  notifies :restart, 'service[nginx]'  # Перезапустить если изменился
end

# ───────────────────────────────────────────────────────────
# Запустить сервис
# ───────────────────────────────────────────────────────────
service 'nginx' do
  action [:enable, :start]
  supports :status => true, :restart => true, :reload => true
end

# ───────────────────────────────────────────────────────────
# Цикл для создания нескольких конфигов
# ───────────────────────────────────────────────────────────
%w(site1 site2 site3).each do |site|
  template "/etc/nginx/sites-available/#{site}" do
    source 'site.conf.erb'
    variables(
      server_name: "#{site}.example.com",
      root_path: "/var/www/#{site}"
    )
    notifies :reload, 'service[nginx]'
  end
end
```

### Push Model (Ansible, Terraform)

text

```
┌────────────────────────────────────────────────────────────┐
│                    PUSH MODEL                              │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  КАК РАБОТАЕТ:                                            │
│  1. Control Server инициирует подключение                 │
│  2. Подключается к серверам (SSH, API)                    │
│  3. Отправляет (PUSH) конфигурацию                        │
│  4. Выполняет на серверах                                 │
│  5. Получает результат                                    │
│  6. Отключается                                           │
│                                                            │
│  ПРЕИМУЩЕСТВА:                                            │
│  ✅ Простота — не нужен агент на серверах                 │
│  ✅ Контроль — вы решаете КОГДА применять конфигурацию    │
│  ✅ Безопасность — не нужно открывать порты на серверах   │
│  ✅ Легко начать (только SSH)                             │
│  ✅ Нет daemon'ов, потребляющих ресурсы на серверах       │
│                                                            │
│  НЕДОСТАТКИ:                                              │
│  ❌ Масштабируемость — медленнее на тысячах серверов      │
│  ❌ Нужна сетевая связь с control server → servers        │
│  ❌ Последовательное выполнение (по умолчанию)            │
│  ❌ Control server — single point of failure              │
│  ❌ Нет автоматического исправления drift                 │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  • Малые и средние окружения (до 500 серверов)            │
│  • Нужна простота                                         │
│  • Изменения применяются редко (по требованию)            │
│  • Нельзя установить агента (security policy)             │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**Пример Push Model (Ansible):**

YAML

```
# ═══════════════════════════════════════════════════════════
#          PUSH MODEL — ANSIBLE
# ═══════════════════════════════════════════════════════════

---
# ───────────────────────────────────────────────────────────
# Control Server выполняет:
# ansible-playbook -i inventory deploy.yml
# ───────────────────────────────────────────────────────────

- name: Deploy application
  hosts: webservers  # Список серверов
  become: yes
  
  tasks:
    # Control Server ПОДКЛЮЧАЕТСЯ к каждому серверу по SSH
    # и ВЫПОЛНЯЕТ эти задачи
    
    - name: Install packages
      apt:
        name:
          - nginx
          - nodejs
          - npm
        state: present
    
    - name: Copy application
      copy:
        src: /local/app/
        dest: /var/www/app/
    
    - name: Start services
      service:
        name: "{{ item }}"
        state: started
      loop:
        - nginx
        - app
```

text

```

│     └─ ansible-playbook deploy.yml                        │

### Pull Model (Chef, Puppet)

text

```
┌────────────────────────────────────────────────────────────┐
│                    PULL MODEL                              │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  КАК РАБОТАЕТ:                                            │
│  1. На каждом сервере установлен АГЕНТ                    │
│  2. Агент периодически (например, каждые 30 мин)          │
│     подключается к Control Server                         │
│  3. Агент ЗАБИРАЕТ (PULL) свою конфигурацию               │
│  4. Агент применяет конфигурацию локально                 │
│  5. Агент отправляет отчёт на Control Server              │
│  6. Цикл повторяется                                      │
│                                                            │
│  ПРЕИМУЩЕСТВА:                                            │
│  ✅ Масштабируемость — отлично для тысяч серверов         │
│  ✅ Автоматическое исправление drift                      │
│  ✅ Децентрализация — серверы сами управляют собой        │
│  ✅ Control Server менее нагружен                         │
│  ✅ Серверы работают даже если Control Server недоступен  │
│                                                            │
│  НЕДОСТАТКИ:                                              │
│  ❌ Сложность — нужно установить агента везде             │
│  ❌ Ресурсы — агент потребляет CPU/RAM на серверах        │
│  ❌ Безопасность — нужно открыть порт для агента          │
│  ❌ Задержка — изменения применяются с задержкой          │
│  ❌ Сложнее debugging (агент работает автономно)          │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  • Большие окружения (1000+ серверов)                     │
│  • Нужна автоматическая коррекция конфигурации            │
│  • Compliance — серверы должны всегда соответствовать     │
│  • Control Server не может достучаться до всех серверов   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

**Пример Pull Model (Chef):**

Ruby

```
# ═══════════════════════════════════════════════════════════
#          PULL MODEL — CHEF
# ═══════════════════════════════════════════════════════════

# ───────────────────────────────────────────────────────────
# На Chef Server загружен cookbook:
# knife cookbook upload nginx
# ───────────────────────────────────────────────────────────

# Cookbook: nginx/recipes/default.rb
package 'nginx' do
  action :install
end

service 'nginx' do
  action [:enable, :start]
end

# ───────────────────────────────────────────────────────────
# На каждом сервере работает Chef Client (агент)
# Он запускается каждые 30 минут (по умолчанию)
# ───────────────────────────────────────────────────────────
```

text

│                                                            │

```

### Сравнение Push vs Pull

text

```
┌──────────────────────┬─────────────────────┬─────────────────────┐
│      Критерий        │     PUSH MODEL      │    PULL MODEL       │
│                      │  (Ansible)          │  (Chef/Puppet)      │
├──────────────────────┼─────────────────────┼─────────────────────┤
│ Агент на серверах    │ НЕ нужен ✅         │ Нужен ❌            │
│                      │                     │                     │
│ Инициатор            │ Control Server      │ Агент на сервере    │
│                      │                     │                     │
│ Частота применения   │ По требованию       │ Периодически        │
│                      │ (когда запустили)   │ (каждые 30 мин)     │
│                      │                     │                     │
│ Скорость применения  │ Немедленно          │ С задержкой         │
│                      │                     │ (до 30 мин)         │
│                      │                     │                     │
│ Масштабируемость     │ Средняя             │ Высокая             │
│                      │ (до 500 серверов)   │ (1000+ серверов)    │
│                      │                     │                     │
│ Сложность настройки  │ Низкая ⭐           │ Высокая ⭐⭐⭐       │
│                      │                     │                     │
│ Ресурсы на серверах  │ Нет                 │ Агент (CPU/RAM)     │
│                      │                     │                     │
│ Drift correction     │ Нет (ручной запуск) │ Автоматически       │
│                      │                     │                     │
│ Сетевое подключение  │ Control → Servers   │ Servers → Control   │
│                      │ (исходящее)         │ (исходящее)         │
│                      │                     │                     │
│ Работа offline       │ Нет                 │ Да (кэш конфига)    │
│                      │                     │                     │
│ Безопасность         │ Проще (SSH)         │ Сложнее (агент)     │
│                      │                     │                     │
│ Типичные инструменты │ Ansible, Terraform  │ Chef, Puppet, Salt  │
└──────────────────────┴─────────────────────┴─────────────────────┘
```
