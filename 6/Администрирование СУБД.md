## 1. Какие бывают типы бэкапов в СУБД?

text

```
┌────────────────────────────────────────────────────────────┐
│         ТИПЫ БЭКАПОВ (BACKUP) В СУБД                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ЗАЧЕМ НУЖНЫ БЭКАПЫ:                                      │
│  • Восстановление после сбоя (hardware failure)           │
│  • Восстановление после ошибок (случайный DELETE)         │
│  • Disaster Recovery (пожар, наводнение)                  │
│  • Миграция / клонирование БД                             │
│  • Compliance / Audit (хранение истории)                  │
│  • Тестирование на prod данных                            │
│                                                            │
│  ОСНОВНЫЕ ТИПЫ:                                           │
│  1. FULL BACKUP (Полный)                                  │
│  2. INCREMENTAL BACKUP (Инкрементальный)                  │
│  3. DIFFERENTIAL BACKUP (Дифференциальный)                │
│  4. LOGICAL BACKUP (Логический)                           │
│  5. PHYSICAL BACKUP (Физический)                          │
│  6. CONTINUOUS / WAL BACKUP (Непрерывный)                 │
│  7. SNAPSHOT BACKUP (Снимок)                              │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 1.1 FULL BACKUP (Полный бэкап)

text

```
┌────────────────────────────────────────────────────────────┐
│         FULL BACKUP                                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Полная копия ВСЕЙ базы данных на момент времени          │
│                                                            │
│  СТРУКТУРА:                                               │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ База данных (100 GB)                                 │ │
│  │ ┌──────────┬──────────┬──────────┬──────────────┐    │ │
│  │ │ Таблица1 │ Таблица2 │ Таблица3 │ Индексы      │    │ │
│  │ └──────────┴──────────┴──────────┴──────────────┘    │ │
│  │                      ↓                                │ │
│  │              FULL BACKUP                              │ │
│  │                      ↓                                │ │
│  │ ┌────────────────────────────────────────────────┐   │ │
│  │ │ backup_full_2024-01-15.dump   (100 GB)        │   │ │
│  │ │ • Все таблицы                                  │   │ │
│  │ │ • Все данные                                   │   │ │
│  │ │ • Все индексы                                  │   │ │
│  │ │ • Схема БД                                     │   │ │
│  │ └────────────────────────────────────────────────┘   │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  КОМАНДЫ:                                                 │
│                                                            │
│  PostgreSQL:                                              │
│  pg_dump mydb > backup_full.sql                           │
│  pg_dump -Fc mydb > backup_full.dump  # Формат custom     │
│  pg_basebackup -D /backup/full  # Физический бэкап        │
│                                                            │
│  MySQL:                                                   │
│  mysqldump mydb > backup_full.sql                         │
│  mysqldump --all-databases > backup_all.sql               │
│                                                            │
│  ВОССТАНОВЛЕНИЕ:                                          │
│  PostgreSQL:                                              │
│  psql mydb < backup_full.sql                              │
│  pg_restore -d mydb backup_full.dump                      │
│                                                            │
│  MySQL:                                                   │
│  mysql mydb < backup_full.sql                             │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Простота (один файл)                                   │
│  • Быстрое восстановление (один шаг)                      │
│  • Независимость (не нужны другие бэкапы)                 │
│  • Легко тестировать                                      │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Долго создаётся (часы для больших БД)                  │
│  • Много места на диске                                   │
│  • Нагрузка на БД (I/O, CPU)                              │
│  • Дублирование данных (каждый день 100GB)                │
│                                                            │
│  РАСПИСАНИЕ:                                              │
│  • Раз в неделю (воскресенье ночью)                       │
│  • Раз в месяц (архивный)                                 │
│  • + Инкрементальные бэкапы ежедневно                     │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 1.2 INCREMENTAL BACKUP (Инкрементальный)

text

```
┌────────────────────────────────────────────────────────────┐
│         INCREMENTAL BACKUP                                 │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Копирует ТОЛЬКО данные, изменённые с ПОСЛЕДНЕГО бэкапа   │
│  (любого типа: full или incremental)                      │
│                                                            │
│  СХЕМА:                                                   │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ Воскресенье: FULL BACKUP        (100 GB)            │ │
│  │              ████████████████████████████████         │ │
│  │                                                      │ │
│  │ Понедельник: INCREMENTAL         (5 GB)             │ │
│  │              ██ (изменения с воскресенья)            │ │
│  │                                                      │ │
│  │ Вторник:     INCREMENTAL         (3 GB)             │ │
│  │              █ (изменения с понедельника)            │ │
│  │                                                      │ │
│  │ Среда:       INCREMENTAL         (4 GB)             │ │
│  │              █ (изменения со вторника)               │ │
│  │                                                      │ │
│  │ ...                                                  │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  ВОССТАНОВЛЕНИЕ (среда):                                  │
│  1. Восстановить FULL (воскресенье)                       │
│  2. Применить INCREMENTAL (понедельник)                   │
│  3. Применить INCREMENTAL (вторник)                       │
│  4. Применить INCREMENTAL (среда)                         │
│                                                            │
│  → 4 шага!                                                │
│                                                            │
│  МЕХАНИЗМ (PostgreSQL):                                   │
│  • WAL (Write-Ahead Log) архивирование                    │
│  • pg_basebackup (full) + WAL segments                    │
│                                                            │
│  postgresql.conf:                                         │
│  wal_level = replica                                      │
│  archive_mode = on                                        │
│  archive_command = 'cp %p /backup/wal/%f'                 │
│                                                            │
│  Создание:                                                │
│  # Воскресенье: Full                                      │
│  pg_basebackup -D /backup/full                            │
│                                                            │
│  # Пн-Сб: Автоматическая архивация WAL                    │
│  # (PostgreSQL копирует WAL файлы в /backup/wal/)         │
│                                                            │
│  Восстановление:                                          │
│  # 1. Восстановить base backup                            │
│  cp -r /backup/full /var/lib/postgresql/data              │
│                                                            │
│  # 2. recovery.conf (PostgreSQL 11-)                      │
│  restore_command = 'cp /backup/wal/%f %p'                 │
│  recovery_target_time = '2024-01-17 14:30:00'             │
│                                                            │
│  # 3. Запустить PostgreSQL                                │
│  # → Применит все WAL файлы до указанного времени         │
│                                                            │
│  MySQL (не нативная поддержка):                           │
│  • Используйте сторонние инструменты:                     │
│    - Percona XtraBackup                                   │
│    - MySQL Enterprise Backup                              │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Быстро создаётся (минуты вместо часов)                 │
│  • Мало места (только изменения)                          │
│  • Низкая нагрузка на БД                                  │
│  • Частые бэкапы (каждый час возможно)                    │
│  • Point-in-Time Recovery (восстановление на момент)      │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Сложное восстановление (много шагов)                   │
│  • Зависимость от предыдущих бэкапов (цепочка)            │
│  • Если один бэкап повреждён → цепочка рвётся             │
│  • Долгое восстановление (нужно применить все)            │
│                                                            │
│  СТРАТЕГИЯ:                                               │
│  • Воскресенье: FULL                                      │
│  • Пн-Сб: INCREMENTAL каждые 6 часов                      │
│  • Хранить: 4 недели                                      │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 1.3 DIFFERENTIAL BACKUP (Дифференциальный)

text

```
┌────────────────────────────────────────────────────────────┐
│         DIFFERENTIAL BACKUP                                │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Копирует данные, изменённые с ПОСЛЕДНЕГО ПОЛНОГО бэкапа  │
│  (не incremental, а именно FULL!)                         │
│                                                            │
│  СХЕМА:                                                   │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ Воскресенье: FULL BACKUP        (100 GB)            │ │
│  │              ████████████████████████████████         │ │
│  │                                                      │ │
│  │ Понедельник: DIFFERENTIAL        (5 GB)             │ │
│  │              ██ (изменения с воскресенья)            │ │
│  │                                                      │ │
│  │ Вторник:     DIFFERENTIAL        (8 GB)             │ │
│  │              ███ (изменения с воскресенья!)          │ │
│  │              ↑ Больше, т.к. накапливаются            │ │
│  │                                                      │ │
│  │ Среда:       DIFFERENTIAL        (12 GB)            │ │
│  │              ████ (изменения с воскресенья!)         │ │
│  │                                                      │ │
│  │ ...                                                  │ │
│  │                                                      │ │
│  │ Суббота:     DIFFERENTIAL        (30 GB)            │ │
│  │              █████████ (все изменения за неделю!)    │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  ВОССТАНОВЛЕНИЕ (среда):                                  │
│  1. Восстановить FULL (воскресенье)                       │
│  2. Применить DIFFERENTIAL (среда)                        │
│                                                            │
│  → 2 шага! (проще чем incremental)                        │
│                                                            │
│  СРАВНЕНИЕ С INCREMENTAL:                                 │
│  ┌────────────┬──────────────┬──────────────────────────┐ │
│  │ День       │ Incremental  │ Differential             │ │
│  ├────────────┼──────────────┼──────────────────────────┤ │
│  │ Воскр.     │ 100 GB (full)│ 100 GB (full)            │ │
│  │ Пн.        │ 5 GB         │ 5 GB                     │ │
│  │ Вт.        │ 3 GB         │ 8 GB ← накапливается     │ │
│  │ Ср.        │ 4 GB         │ 12 GB ← ещё больше       │ │
│  │ ...        │ ...          │ ...                      │ │
│  │ Сб.        │ 6 GB         │ 30 GB                    │ │
│  ├────────────┼──────────────┼──────────────────────────┤ │
│  │ Итого      │ 125 GB       │ 155 GB                   │ │
│  │ за неделю  │              │ ↑ Больше места           │ │
│  ├────────────┼──────────────┼──────────────────────────┤ │
│  │ Восст-е    │ 7 шагов      │ 2 шага                   │ │
│  │ (суббота)  │ (долго)      │ (быстро) ✅              │ │
│  └────────────┴──────────────┴──────────────────────────┘ │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Быстрое восстановление (2 шага: full + последний diff) │
│  • Проще чем incremental                                  │
│  • Меньше зависимостей (только от full)                   │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Больше места чем incremental                           │
│  • Каждый день diff растёт (дублирование)                 │
│  • К концу недели diff почти = full                       │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  • Баланс между incremental и full                        │
│  • Средние БД (100-500 GB)                                │
│  • Нужно быстрое восстановление                           │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 1.4 LOGICAL vs PHYSICAL BACKUP

text

```
┌────────────────────────────────────────────────────────────┐
│         LOGICAL BACKUP (Логический)                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Экспорт данных в виде SQL-команд (CREATE, INSERT)        │
│  или в формате данных (CSV, JSON)                         │
│                                                            │
│  ФОРМАТ (текстовый):                                      │
│  CREATE TABLE users (                                     │
│      id SERIAL PRIMARY KEY,                               │
│      name VARCHAR(100),                                   │
│      email VARCHAR(255)                                   │
│  );                                                       │
│                                                            │
│  INSERT INTO users (id, name, email) VALUES               │
│  (1, 'Ivan', 'ivan@mail.ru'),                             │
│  (2, 'Maria', 'maria@mail.ru'),                           │
│  (3, 'Petr', 'petr@mail.ru');                             │
│                                                            │
│  CREATE INDEX idx_email ON users(email);                  │
│  ...                                                      │
│                                                            │
│  ИНСТРУМЕНТЫ:                                             │
│  PostgreSQL:                                              │
│  pg_dump mydb > backup.sql         # SQL формат           │
│  pg_dump -Fc mydb > backup.dump    # Custom формат        │
│  pg_dump -Ft mydb > backup.tar     # Tar формат           │
│                                                            │
│  MySQL:                                                   │
│  mysqldump mydb > backup.sql                              │
│  mysqldump --single-transaction mydb > backup.sql         │
│                                                            │
│  ВОССТАНОВЛЕНИЕ:                                          │
│  psql mydb < backup.sql                                   │
│  mysql mydb < backup.sql                                  │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Читаемый формат (можно редактировать)                  │
│  • Переносимость (разные версии СУБД)                     │
│  • Выборочное восстановление (одна таблица)               │
│  • Миграция между СУБД (PostgreSQL → MySQL)               │
│  • Компактный (сжатие текста)                             │
│  • Независимость от платформы (OS, архитектура)           │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Медленно (нужно выполнить все INSERT)                  │
│  • Нагрузка на БД (полный table scan)                     │
│  • Не включает WAL (нет point-in-time recovery)           │
│  • Большой размер файла для больших БД                    │
│  • Несовместимость версий (иногда)                        │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  • Миграция между серверами                               │
│  • Клонирование БД для разработки                         │
│  • Архивирование (долгосрочное хранение)                  │
│  • Малые-средние БД (<100 GB)                             │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│         PHYSICAL BACKUP (Физический)                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Копирование ФАЙЛОВ БД на уровне файловой системы         │
│  (data files, WAL, конфиги)                               │
│                                                            │
│  СТРУКТУРА (PostgreSQL):                                  │
│  /var/lib/postgresql/data/                                │
│  ├── base/            ← Файлы таблиц                      │
│  │   ├── 16384/                                           │
│  │   │   ├── 16385   ← Файл таблицы                      │
│  │   │   ├── 16386                                        │
│  ├── global/          ← Системные таблицы                 │
│  ├── pg_wal/          ← WAL файлы                         │
│  ├── pg_tblspc/       ← Tablespaces                       │
│  └── postgresql.conf                                      │
│                                                            │
│  ИНСТРУМЕНТЫ:                                             │
│  PostgreSQL:                                              │
│  pg_basebackup -D /backup/base -Ft -z  # tar.gz формат    │
│                                                            │
│  Или вручную (БД должна быть остановлена!):               │
│  systemctl stop postgresql                                │
│  cp -r /var/lib/postgresql/data /backup/                  │
│  systemctl start postgresql                               │
│                                                            │
│  MySQL:                                                   │
│  # InnoDB:                                                │
│  mysqlbackup --backup-dir=/backup backup-and-apply-log    │
│                                                            │
│  Percona XtraBackup:                                      │
│  xtrabackup --backup --target-dir=/backup/                │
│                                                            │
│  ВОССТАНОВЛЕНИЕ:                                          │
│  systemctl stop postgresql                                │
│  rm -rf /var/lib/postgresql/data/*                        │
│  cp -r /backup/data/* /var/lib/postgresql/data/           │
│  systemctl start postgresql                               │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Очень быстро (копирование файлов)                      │
│  • Быстрое восстановление                                 │
│  • Point-in-Time Recovery (с WAL)                         │
│  • Низкая нагрузка на БД (особенно с pg_basebackup)       │
│  • Точная копия (включая настройки)                       │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Привязка к версии СУБД (major version)                 │
│  • Привязка к платформе (OS, CPU архитектура)             │
│  • Нельзя восстановить одну таблицу                       │
│  • Больше размер (все файлы, индексы)                     │
│  • Нужен совместимый сервер                               │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  • Production бэкапы (быстрое восстановление критично)    │
│  • Большие БД (>500 GB)                                   │
│  • Disaster Recovery                                      │
│  • Репликация (base backup для реплики)                   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 1.5 CONTINUOUS BACKUP (WAL Archiving)

text

```
┌────────────────────────────────────────────────────────────┐
│         CONTINUOUS BACKUP / WAL ARCHIVING                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Непрерывное архивирование WAL (Write-Ahead Log) файлов   │
│  → Возможность восстановления на ЛЮБУЮ секунду!           │
│                                                            │
│  ПРИНЦИП (PostgreSQL):                                    │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ 1. PostgreSQL пишет изменения в WAL                  │ │
│  │ 2. Когда WAL сегмент заполнен (16 MB)                │ │
│  │    → archive_command копирует его в архив            │ │
│  │ 3. Повторяется непрерывно                            │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  НАСТРОЙКА (postgresql.conf):                             │
│  wal_level = replica                                      │
│  archive_mode = on                                        │
│  archive_command = 'cp %p /backup/wal/%f'                 │
│  # или через сжатие:                                      │
│  archive_command = 'gzip < %p > /backup/wal/%f.gz'        │
│  # или в S3:                                              │
│  archive_command = 'aws s3 cp %p s3://bucket/wal/%f'      │
│                                                            │
│  ПОЛНЫЙ BACKUP + WAL:                                     │
│  # Воскресенье: base backup                               │
│  pg_basebackup -D /backup/base -Ft -z                     │
│                                                            │
│  # Пн-Сб: автоматическая архивация WAL                    │
│  # (PostgreSQL делает сам!)                               │
│                                                            │
│  ВОССТАНОВЛЕНИЕ НА КОНКРЕТНЫЙ МОМЕНТ:                     │
│  # recovery.conf (PostgreSQL 12+: postgresql.auto.conf)   │
│  restore_command = 'cp /backup/wal/%f %p'                 │
│  recovery_target_time = '2024-01-17 14:35:42'             │
│  # Восстановить БД до 14:35:42 (секунда!)                 │
│                                                            │
│  # Запустить PostgreSQL                                   │
│  systemctl start postgresql                               │
│  # → Применит base backup + все WAL до указанного времени │
│                                                            │
│  POINT-IN-TIME RECOVERY (PITR):                           │
│  "Откатить случайный DROP TABLE через 5 минут после"      │
│                                                            │
│  14:30:00 - Всё работает                                  │
│  14:35:42 - Случайно: DROP TABLE orders;                  │
│  14:40:00 - Обнаружили ошибку!                            │
│                                                            │
│  Решение:                                                 │
│  recovery_target_time = '2024-01-17 14:35:41'             │
│  → Восстановить на 1 секунду ДО дропа!                    │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Point-in-Time Recovery (восстановление на секунду!)    │
│  • Минимальная потеря данных (RPO < 1 мин)                │
│  • Непрерывная защита                                     │
│  • Комбинируется с base backup                            │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Сложная настройка                                      │
│  • Много места (WAL накапливается)                        │
│  • Нужна очистка старых WAL                               │
│                                                            │
│  ИНСТРУМЕНТЫ:                                             │
│  • pg_probackup (PostgreSQL)                              │
│  • Barman (Backup and Recovery Manager)                   │
│  • WAL-E / WAL-G (архивация в S3)                         │
│  • pgBackRest                                             │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

text

┌────────────────────────────────────────────────────────────┐
│         SNAPSHOT BACKUP                                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Мгновенный снимок диска на уровне файловой системы       │

│                                                            │
│  ТЕХНОЛОГИИ:                                              │
│                                                            │

│  # Примонтировать                                         │
│  mount /dev/vg/db_snap /mnt/snapshot                      │
│                                                            │
│  # Скопировать                                            │

│                                                            │
│  # Удалить snapshot                                       │
│  lvremove /dev/vg/db_snap                                 │
│                                                            │
│  2. ZFS:                                                  │
│  ──────                                                    │
│  zfs snapshot tank/db@backup_2024_01_17                   │
│  zfs send tank/db@backup_2024_01_17 | gzip > backup.gz    │
│                                                            │
│  3. AWS EBS Snapshot:                                     │
│  ────────────────────                                      │
│  # Через консоль или CLI                                  │
│  aws ec2 create-snapshot --volume-id vol-abc123           │
│                                                            │
│  4. Google Cloud Persistent Disk Snapshot                 │
│  5. Azure Managed Disk Snapshot                           │
│                                                            │
│  ПРОЦЕСС:                                                 │
│  1. Подготовить БД (flush, freeze)                        │
│     PostgreSQL: SELECT pg_start_backup('label');          │
│                                                            │
│  2. Создать snapshot (секунды!)                           │
│     lvcreate --snapshot ...                               │
│                                                            │
│  3. Разморозить БД                                        │
│     PostgreSQL: SELECT pg_stop_backup();                  │
│                                                            │
│  4. Скопировать snapshot в безопасное место               │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Мгновенное создание (секунды, не часы!)                │
│  • Консистентный снимок                                   │
│  • Минимальная блокировка БД                              │
│  • Быстрое восстановление                                 │
│  • Copy-on-Write (экономия места)                         │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Требует специальной инфраструктуры (LVM, ZFS, cloud)   │
│  • Snapshot занимает место при изменениях                 │
│  • Привязка к платформе                                   │
│  • Нужна дополнительная настройка                         │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  • Облачные БД (AWS RDS, Google Cloud SQL)                │
│  • Большие БД (терабайты)                                 │
│  • Нужно минимальное окно бэкапа                          │
│  • Есть инфраструктура (LVM, ZFS)                         │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 2. Что такое репликация?

text

```
┌────────────────────────────────────────────────────────────┐
│         РЕПЛИКАЦИЯ (REPLICATION)                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Процесс автоматического копирования данных с одного      │
│  сервера БД (Master/Primary) на другие серверы (Replicas) │
│                                                            │
│  ЗАЧЕМ НУЖНА:                                             │
│  1. HIGH AVAILABILITY (Высокая доступность)               │
│     • Master падает → Replica становится Master           │
│     • Минимизация downtime                                │
│                                                            │
│  2. МАСШТАБИРОВАНИЕ ЧТЕНИЯ (Read Scaling)                 │
│     • Master: запись (INSERT, UPDATE, DELETE)             │
│     • Replicas: чтение (SELECT)                           │
│     • 1 Master + 10 Replicas = 10x читающая нагрузка      │
│                                                            │
│  3. ГЕОГРАФИЧЕСКОЕ РАСПРЕДЕЛЕНИЕ                          │
│     • Master в США, Replica в Европе                      │
│     • Низкая задержка для локальных пользователей         │
│                                                            │
│  4. BACKUP                                                │
│     • Бэкап с Replica (не нагружает Master)               │
│                                                            │
│  5. АНАЛИТИКА                                             │
│     • Тяжёлые запросы на Replica (не тормозят OLTP)       │
│                                                            │
│  АРХИТЕКТУРА:                                             │
│  ┌──────────────────────────────────────────────────────┐ │
│  │                                                      │ │
│  │       ┌─────────────┐                                │ │
│  │       │   Master    │ ← Запись (Write)              │ │
│  │       │  (Primary)  │                                │ │
│  │       └──────┬──────┘                                │ │
│  │              │                                        │ │
│  │         Репликация                                   │ │
│  │        (копирование                                  │ │
│  │         изменений)                                   │ │
│  │              │                                        │ │
│  │     ┌────────┼────────┐                              │ │
│  │     ▼        ▼        ▼                              │ │
│  │ ┌────────┐┌────────┐┌────────┐                       │ │
│  │ │Replica1││Replica2││Replica3│ ← Чтение (Read)       │ │
│  │ │ (Slave)││ (Slave)││ (Slave)│                       │ │
│  │ └────────┘└────────┘└────────┘                       │ │
│  │                                                      │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 3. Какие бывают типы репликации?

### 3.1 По направлению данных

text

```
┌────────────────────────────────────────────────────────────┐
│  1. MASTER-SLAVE (PRIMARY-SECONDARY) REPLICATION           │
│     ─────────────────────────────────────────              │
│     Однонаправленная репликация                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  СХЕМА:                                                   │
│       ┌─────────┐                                         │
│       │ Master  │ ← Write (INSERT, UPDATE, DELETE)        │
│       └────┬────┘                                         │
│            │                                              │
│     Репликация →                                          │
│            │                                              │
│     ┌──────┴──────┐                                       │
│     ▼             ▼                                       │
│  ┌──────┐      ┌──────┐                                  │
│  │Slave1│      │Slave2│ ← Read Only (SELECT)             │
│  └──────┘      └──────┘                                  │
│                                                            │
│  ХАРАКТЕРИСТИКИ:                                          │
│  • Master: READ + WRITE                                   │
│  • Slaves: READ ONLY (попытка записи → ошибка)            │
│  • Изменения идут ТОЛЬКО Master → Slaves                  │
│  • Slaves НЕ реплицируют друг другу                       │
│                                                            │
│  ПРИМЕР (MySQL):                                          │
│  ──────────────────                                        │
│  # На Master:                                             │
│  mysql> SHOW MASTER STATUS;                               │
│  +------------------+----------+                           │
│  | File             | Position |                           │
│  +------------------+----------+                           │
│  | mysql-bin.000003 | 1234     |                           │
│  +------------------+----------+                           │
│                                                            │
│  # На Slave:                                              │
│  CHANGE MASTER TO                                         │
│    MASTER_HOST='master.example.com',                      │
│    MASTER_USER='repl_user',                               │
│    MASTER_PASSWORD='password',                            │
│    MASTER_LOG_FILE='mysql-bin.000003',                    │
│    MASTER_LOG_POS=1234;                                   │
│                                                            │
│  START SLAVE;                                             │
│                                                            │
│  # Проверка                                               │
│  SHOW SLAVE STATUS\G                                      │
│  Slave_IO_Running: Yes                                    │
│  Slave_SQL_Running: Yes                                   │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Простота настройки                                     │
│  • Масштабирование чтения (N slaves)                      │
│  • Бэкап со slave (не нагружает master)                   │
│  • Аналитика на slave                                     │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Одна точка отказа (master)                             │
│  • Ручной failover (нужно промоутить slave в master)      │
│  • Задержка репликации (replication lag)                  │
│  • Нет масштабирования записи                             │
│                                                            │
│  ПРИМЕНЕНИЕ:                                              │
│  • Стандартная конфигурация (90% случаев)                 │
│  • Web-приложения (много чтения, мало записи)             │
│  • Read-heavy workloads                                   │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  2. MULTI-MASTER (MASTER-MASTER) REPLICATION               │
│     ───────────────────────────────────────                │
│     Двунаправленная репликация                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  СХЕМА:                                                   │
│       ┌─────────┐ ←──── репликация ────→ ┌─────────┐     │
│       │ Master1 │                         │ Master2 │     │
│       └─────────┘ ←──── репликация ────→ └─────────┘     │
│           ↑                                     ↑          │
│         Write                                 Write        │
│         Read                                  Read         │
│                                                            │
│  ХАРАКТЕРИСТИКИ:                                          │
│  • ОБА сервера: READ + WRITE                              │
│  • Изменения реплицируются в ОБЕ стороны                  │
│  • Активный-Активный (Active-Active)                      │
│                                                            │
│  ПРОБЛЕМА: КОНФЛИКТЫ!                                     │
│  ─────────────────────                                     │
│  T1: Master1: UPDATE users SET balance=100 WHERE id=1;    │
│  T2: Master2: UPDATE users SET balance=200 WHERE id=1;    │
│                                                            │
│  → Конфликт! Какое значение правильное?                   │
│                                                            │
│  РАЗРЕШЕНИЕ КОНФЛИКТОВ:                                   │
│  1. Last Write Wins (последняя запись побеждает)          │
│     • По timestamp                                        │
│     • Может потерять данные!                              │
│                                                            │
│  2. Application-level resolution                          │
│     • Приложение решает конфликт                          │
│     • Сложно                                              │
│                                                            │
│  3. Избегание конфликтов                                  │
│     • Разделение данных (Master1 для USA, Master2 для EU) │
│     • Auto-increment ID с разным шагом:                   │
│       Master1: 1, 3, 5, 7, ...                            │
│       Master2: 2, 4, 6, 8, ...                            │
│                                                            │
│  ПРИМЕР (MySQL):                                          │
│  ──────────────────                                        │
│  # Master1                                                │
│  server-id = 1                                            │
│  auto_increment_increment = 2                             │
│  auto_increment_offset = 1                                │
│  log-bin = mysql-bin                                      │
│                                                            │
│  # Master2                                                │
│  server-id = 2                                            │
│  auto_increment_increment = 2                             │
│  auto_increment_offset = 2                                │
│  log-bin = mysql-bin                                      │
│                                                            │
│  # На каждом настроить репликацию с другого               │
│  CHANGE MASTER TO ...                                     │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Нет единой точки отказа                                │
│  • Высокая доступность (автофейловер)                     │
│  • Масштабирование записи (оба принимают write)           │
│  • Географическое распределение                           │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Сложность (конфликты!)                                 │
│  • Возможна потеря данных                                 │
│  • Не для всех приложений                                 │
│  • Нужна тщательная архитектура                           │
│                                                            │
│  ПРИМЕНЕНИЕ:                                              │
│  • Географически распределённые приложения                │
│  • Когда нужна высокая доступность записи                 │
│  • CMS системы                                            │
│  ⚠️ НЕ для: финансы, e-commerce (риск конфликтов!)        │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  3. CASCADE (ЦЕПНАЯ) REPLICATION                           │
│     ──────────────────────────────                         │
│     Slave реплицирует от другого Slave                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  СХЕМА:                                                   │
│       ┌─────────┐                                         │
│       │ Master  │                                         │
│       └────┬────┘                                         │
│            │                                              │
│            ▼                                              │
│       ┌─────────┐                                         │
│       │ Slave 1 │                                         │
│       └────┬────┘                                         │
│            │                                              │
│     ┌──────┴──────┐                                       │
│     ▼             ▼                                       │
│  ┌──────┐      ┌──────┐                                  │
│  │Slave2│      │Slave3│                                  │
│  └──────┘      └──────┘                                  │
│                                                            │
│  ЗАЧЕМ:                                                   │
│  • Снизить нагрузку на Master (много slaves)              │
│  • Master → 1 Slave → 10 Slaves (дерево)                  │
│                                                            │
│  НАСТРОЙКА (MySQL):                                       │
│  ──────────────────────                                    │
│  # На Slave 1 (реплицирует от Master)                     │
│  log-slave-updates = 1  ← Важно! Пишет в binlog           │
│                                                            │
│  # На Slave 2 и 3 (реплицируют от Slave 1)                │
│  CHANGE MASTER TO MASTER_HOST='slave1.example.com', ...   │
│                                                            │
│  ⚠️ ПРОБЛЕМЫ:                                              │
│  • Увеличенная задержка (Master → S1 → S2 = 2x lag)       │
│  • Сложность диагностики                                  │
│  • Одна ошибка в цепи → рвётся вся цепь                   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```
### 3.2 По способу передачи данных

text

```
┌────────────────────────────────────────────────────────────┐
│  СИНХРОННАЯ vs АСИНХРОННАЯ РЕПЛИКАЦИЯ                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. АСИНХРОННАЯ РЕПЛИКАЦИЯ (Asynchronous)                 │
│     ────────────────────────────────────────               │
│     ПО УМОЛЧАНИЮ в большинстве СУБД                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ПРИНЦИП РАБОТЫ:                                          │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ Шаг 1: Клиент отправляет запрос на Master            │ │
│  │         INSERT INTO users VALUES (...);              │ │
│  │                                                      │ │
│  │ Шаг 2: Master записывает в свою БД                   │ │
│  │         → Возвращает OK клиенту СРАЗУ! ⚡            │ │
│  │         (НЕ ждёт реплик!)                            │ │
│  │                                                      │ │
│  │ Шаг 3: Master асинхронно отправляет на Replicas      │ │
│  │         (в фоне, через WAL/binlog)                   │ │
│  │                                                      │ │
│  │ Шаг 4: Replicas применяют изменения                  │ │
│  │         (с задержкой 0.1с - несколько секунд)        │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  ВРЕМЕННАЯ ДИАГРАММА:                                     │
│  ┌─────────┬──────────────┬──────────────┐               │
│  │  Время  │   Master     │   Replica    │               │
│  ├─────────┼──────────────┼──────────────┤               │
│  │  t0     │ Write start  │              │               │
│  │  t1     │ Write done   │              │               │
│  │  t2     │ OK → Client  │              │ ← Клиент OK!  │
│  │  t3     │ Send to rep. │              │               │
│  │  t4     │              │ Receive      │               │
│  │  t5     │              │ Apply ✅     │               │
│  └─────────┴──────────────┴──────────────┘               │
│           ↑                              ↑                │
│       Клиент получил                 Реплика              │
│       ответ РАНЬШЕ!                  обновилась           │
│                                                            │
│  REPLICATION LAG (Задержка репликации):                   │
│  ─────────────────────────────────────                     │
│  Время между записью на Master и появлением на Replica    │
│                                                            │
│  Нормальная задержка: 0.1-1 секунда                       │
│  При нагрузке: до 10 секунд                               │
│  При проблемах: минуты/часы ⚠️                            │
│                                                            │
│  ПРОБЛЕМА: STALE READS (Чтение устаревших данных)         │
│  ────────────────────────────────────────────              │
│  t1: Client A → Master: UPDATE balance = 100 WHERE id=1;  │
│  t2: Master: OK (balance=100)                             │
│  t3: Client B → Replica: SELECT balance WHERE id=1;       │
│      → Replica ещё НЕ получила обновление!                │
│      → Возвращает старое значение: balance=50 ❌          │
│                                                            │
│  МОНИТОРИНГ LAG:                                          │
│  ──────────────────                                        │
│  PostgreSQL:                                              │
│  SELECT                                                   │
│    client_addr,                                           │
│    state,                                                 │
│    pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn)     │
│      AS lag_bytes                                         │
│  FROM pg_stat_replication;                                │
│                                                            │
│  MySQL:                                                   │
│  SHOW SLAVE STATUS\G                                      │
│  Seconds_Behind_Master: 2  ← Задержка в секундах         │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Быстрая запись (не ждём реплик)                        │
│  • Высокая производительность                             │
│  • Replicas могут быть далеко (другой континент)          │
│  • Master не зависит от состояния реплик                  │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Возможна потеря данных (если Master упал до репликации)│
│  • Stale reads (чтение старых данных)                     │
│  • Replication lag                                        │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  ✅ Большинство web-приложений                             │
│  ✅ Когда задержка в секунду приемлема                     │
│  ✅ Высокая производительность важнее консистентности      │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  2. СИНХРОННАЯ РЕПЛИКАЦИЯ (Synchronous)                   │
│     ──────────────────────────────────────                 │
│     Master ЖДЁТ подтверждения от Replica(s)               │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ПРИНЦИП РАБОТЫ:                                          │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ Шаг 1: Клиент отправляет запрос                      │ │
│  │         INSERT INTO users VALUES (...);              │ │
│  │                                                      │ │
│  │ Шаг 2: Master записывает в свою БД                   │ │
│  │                                                      │ │
│  │ Шаг 3: Master отправляет на Replicas                 │ │
│  │         И ЖДЁТ ПОДТВЕРЖДЕНИЯ! ⏳                     │ │
│  │                                                      │ │
│  │ Шаг 4: Replica применяет изменения                   │ │
│  │         И отправляет ACK → Master                    │ │
│  │                                                      │ │
│  │ Шаг 5: Master возвращает OK клиенту                  │ │
│  │         ТОЛЬКО после получения ACK                   │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  ВРЕМЕННАЯ ДИАГРАММА:                                     │
│  ┌─────────┬──────────────┬──────────────┐               │
│  │  Время  │   Master     │   Replica    │               │
│  ├─────────┼──────────────┼──────────────┤               │
│  │  t0     │ Write start  │              │               │
│  │  t1     │ Write done   │              │               │
│  │  t2     │ Send to rep. │              │               │
│  │  t3     │ WAIT ⏳      │ Receive      │               │
│  │  t4     │ WAIT ⏳      │ Apply        │               │
│  │  t5     │ WAIT ⏳      │ Send ACK →   │               │
│  │  t6     │ OK → Client  │              │ ← Только сейчас│
│  └─────────┴──────────────┴──────────────┘               │
│                          ↑                                 │
│                    Клиент ждёт ДОЛЬШЕ!                    │
│                    Но данные гарантированно               │
│                    на обеих серверах                      │
│                                                            │
│  НАСТРОЙКА (PostgreSQL):                                  │
│  ──────────────────────────                                │
│  # На Master (postgresql.conf)                            │
│  synchronous_commit = on  # По умолчанию                  │
│  synchronous_standby_names = 'replica1,replica2'          │
│                                                            │
│  # Варианты:                                              │
│  synchronous_standby_names = 'ANY 1 (replica1, replica2)' │
│  # → Ждать подтверждения от ЛЮБОЙ одной реплики          │
│                                                            │
│  synchronous_standby_names = 'FIRST 2 (r1, r2, r3)'       │
│  # → Ждать подтверждения от ПЕРВЫХ двух доступных        │
│                                                            │
│  synchronous_standby_names = '*'                          │
│  # → Ждать подтверждения от ВСЕХ реплик                  │
│                                                            │
│  MySQL (Semi-Synchronous):                                │
│  ───────────────────────────                               │
│  # На Master                                              │
│  INSTALL PLUGIN rpl_semi_sync_master                      │
│    SONAME 'semisync_master.so';                           │
│  SET GLOBAL rpl_semi_sync_master_enabled = 1;             │
│  SET GLOBAL rpl_semi_sync_master_timeout = 1000; # 1 сек  │
│                                                            │
│  # На Slave                                               │
│  INSTALL PLUGIN rpl_semi_sync_slave                       │
│    SONAME 'semisync_slave.so';                            │
│  SET GLOBAL rpl_semi_sync_slave_enabled = 1;              │
│                                                            │
│  ⚠️ MySQL Semi-Sync:                                       │
│  • Если timeout → откат к асинхронной                     │
│  • "Полусинхронная" (не гарантирует 100%)                 │
│                                                            │
│  ГАРАНТИЯ: ZERO DATA LOSS                                 │
│  ────────────────────────────                              │
│  Если Master упал ПОСЛЕ подтверждения клиенту,            │
│  данные ГАРАНТИРОВАННО на Replica!                        │
│                                                            │
│  Сценарий:                                                │
│  t1: Master записал данные                                │
│  t2: Replica подтвердила                                  │
│  t3: Master → OK клиенту                                  │
│  t4: Master УПАЛ! 💥                                       │
│  → Данные НА РЕПЛИКЕ! Можно промоутить её в Master ✅     │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Гарантия сохранности данных (zero data loss)           │
│  • Нет stale reads (реплика всегда актуальна)             │
│  • Предсказуемая консистентность                          │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Медленнее (ждём подтверждения)                         │
│  • Производительность зависит от сети                     │
│  • Если реплика недоступна → блокировка записи            │
│  • Не подходит для географически распределённых реплик    │
│    (задержка сети 100ms+ убьёт производительность)        │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  ✅ Финансовые системы (zero data loss критично)           │
│  ✅ Критичные транзакции                                   │
│  ✅ Когда консистентность важнее производительности        │
│  ⚠️  Реплики должны быть БЛИЗКО (низкая задержка сети)    │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  3. ПОЛУСИНХРОННАЯ (Semi-Synchronous)                     │
│     ────────────────────────────────────                   │
│     Компромисс между async и sync                         │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ПРИНЦИП (MySQL):                                         │
│  • Обычно работает СИНХРОННО                              │
│  • Если timeout (реплика не ответила) → переход на ASYNC  │
│  • Когда реплика вернулась → обратно на SYNC              │
│                                                            │
│  КОНФИГУРАЦИЯ:                                            │
│  rpl_semi_sync_master_timeout = 1000  # 1 секунда         │
│                                                            │
│  Если реплика не ответила за 1сек → async mode            │
│                                                            │
│  ⚖️ БАЛАНС:                                                │
│  • Обычно: гарантия сохранности                           │
│  • При проблемах: не блокируем запись                     │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 3.3 По методу репликации

text

```
┌────────────────────────────────────────────────────────────┐
│  ФИЗИЧЕСКАЯ vs ЛОГИЧЕСКАЯ РЕПЛИКАЦИЯ                      │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. ФИЗИЧЕСКАЯ РЕПЛИКАЦИЯ (Physical/Binary)               │
│     ─────────────────────────────────────────              │
│     Копирование на уровне ФАЙЛОВ и БЛОКОВ                 │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ПРИНЦИП:                                                 │
│  • Master записывает изменения в WAL/redo log             │
│  • WAL содержит БАЙТЫ изменённых блоков данных            │
│  • Replica копирует WAL и применяет НА УРОВНЕ БЛОКОВ      │
│                                                            │
│  ФОРМАТ (PostgreSQL WAL):                                 │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ WAL Record:                                          │ │
│  │ • LSN (Log Sequence Number): 0/1A2B3C4D              │ │
│  │ • Transaction ID: 12345                              │ │
│  │ • Block: File=1234, Offset=5678                      │ │
│  │ • Old data: 0x4A3F... (старые байты)                 │ │
│  │ • New data: 0x5B4E... (новые байты)                  │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  Replica:                                                 │
│  1. Читает WAL                                            │
│  2. Находит блок File=1234, Offset=5678                   │
│  3. Заменяет байты: 0x4A3F... → 0x5B4E...                 │
│                                                            │
│  → Точная ПОБАЙТОВАЯ копия Master!                        │
│                                                            │
│  НАСТРОЙКА (PostgreSQL Streaming Replication):            │
│  ────────────────────────────────────────────              │
│  # На Master (postgresql.conf)                            │
│  wal_level = replica                                      │
│  max_wal_senders = 10                                     │
│  wal_keep_size = 1GB  # Хранить WAL для реплик            │
│                                                            │
│  # pg_hba.conf (разрешить репликацию)                     │
│  host replication repl_user 192.168.1.0/24 md5            │
│                                                            │
│  # Создать пользователя репликации                        │
│  CREATE ROLE repl_user WITH REPLICATION LOGIN PASSWORD '..';│
│                                                            │
│  # На Replica: создать из base backup                     │
│  pg_basebackup -h master.example.com -U repl_user \       │
│    -D /var/lib/postgresql/data -P --wal-method=stream     │
│                                                            │
│  # standby.signal (файл для включения режима реплики)     │
│  touch /var/lib/postgresql/data/standby.signal            │
│                                                            │
│  # postgresql.conf на Replica                             │
│  primary_conninfo = 'host=master.example.com port=5432 \  │
│    user=repl_user password=...'                           │
│                                                            │
│  MySQL (Binary Log Replication):                          │
│  ──────────────────────────────────                        │
│  # Master (my.cnf)                                        │
│  server-id = 1                                            │
│  log-bin = mysql-bin                                      │
│  binlog_format = ROW  # или MIXED                         │
│                                                            │
│  # Slave (my.cnf)                                         │
│  server-id = 2                                            │
│                                                            │
│  # На Slave                                               │
│  CHANGE MASTER TO                                         │
│    MASTER_HOST='master.example.com',                      │
│    MASTER_USER='repl_user',                               │
│    MASTER_PASSWORD='...',                                 │
│    MASTER_LOG_FILE='mysql-bin.000001',                    │
│    MASTER_LOG_POS=154;                                    │
│  START SLAVE;                                             │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Быстрая (байты, не SQL)                                │
│  • Низкая нагрузка на Master                              │
│  • Реплицирует ВСЁ (таблицы, индексы, последовательности) │
│  • Проще настроить                                        │
│  • Стандартная репликация (по умолчанию)                  │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Replica = ТОЧНАЯ копия (включая версию СУБД!)          │
│  • Нельзя реплицировать одну таблицу                      │
│  • Нельзя реплицировать между разными СУБД                │
│    (PostgreSQL ↔ MySQL — невозможно!)                     │
│  • Replica ТОЛЬКО для чтения (read-only)                  │
│  • Replica должна быть на той же архитектуре (x86_64)     │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  ✅ Стандартная репликация PostgreSQL/MySQL                │
│  ✅ Disaster Recovery                                      │
│  ✅ Масштабирование чтения                                 │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  2. ЛОГИЧЕСКАЯ РЕПЛИКАЦИЯ (Logical)                       │
│     ────────────────────────────────────                   │
│     Копирование на уровне ДАННЫХ (SQL)                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ПРИНЦИП:                                                 │
│  • Master декодирует изменения в ЛОГИЧЕСКИЕ операции      │
│  • Отправляет Replica команды: INSERT, UPDATE, DELETE     │
│  • Replica выполняет эти команды                          │
│                                                            │
│  ФОРМАТ (концептуально):                                  │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ Logical Change:                                      │ │
│  │ • Table: users                                       │ │
│  │ • Operation: INSERT                                  │ │
│  │ • Data: {id: 123, name: "Ivan", email: "ivan@..."}   │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  Replica:                                                 │
│  Выполняет: INSERT INTO users VALUES (123, 'Ivan', '...');│
│                                                            │
│  НАСТРОЙКА (PostgreSQL 10+ Logical Replication):          │
│  ──────────────────────────────────────────────            │
│  # На Master (postgresql.conf)                            │
│  wal_level = logical                                      │
│                                                            │
│  # На Master: создать публикацию                          │
│  CREATE PUBLICATION my_pub FOR TABLE users, orders;       │
│  # или для всех таблиц:                                   │
│  CREATE PUBLICATION my_pub FOR ALL TABLES;                │
│                                                            │
│  # На Replica: создать подписку                           │
│  CREATE SUBSCRIPTION my_sub                               │
│    CONNECTION 'host=master.example.com dbname=mydb \      │
│                user=repl_user password=...'               │
│    PUBLICATION my_pub;                                    │
│                                                            │
│  # Репликация началась автоматически!                     │
│                                                            │
│  ГИБКОСТЬ:                                                │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ • Реплицировать только определённые таблицы          │ │
│  │   CREATE PUBLICATION pub FOR TABLE orders;           │ │
│  │                                                      │ │
│  │ • Реплицировать только INSERT/UPDATE (без DELETE)    │ │
│  │   CREATE PUBLICATION pub FOR TABLE users             │ │
│  │   WITH (publish = 'insert, update');                 │ │
│  │                                                      │ │
│  │ • Фильтр по строкам (PostgreSQL 15+)                 │ │
│  │   CREATE PUBLICATION pub FOR TABLE users             │ │
│  │   WHERE (status = 'active');                         │ │
│  │                                                      │ │
│  │ • Трансформация данных на Replica                    │ │
│  │   (через триггеры)                                   │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  MySQL (Logical Replication через сторонние инструменты): │
│  ──────────────────────────────────────────────────────────│
│  • Debezium (Kafka-based CDC)                             │
│  • Maxwell's Daemon                                       │
│  • Canal (Alibaba)                                        │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Гибкость (выборочная репликация)                       │
│  • Между разными версиями PostgreSQL                      │
│  • Replica может иметь дополнительные индексы             │
│  • Replica может быть read-write (!)                      │
│  • Репликация между разными СУБД (через CDC инструменты)  │
│  • Трансформация данных                                   │
│  • Миграция на новую версию без даунтайма                 │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Медленнее физической (декодирование WAL)               │
│  • Больше нагрузка на Master (CPU)                        │
│  • Не реплицирует DDL (CREATE TABLE, ALTER...)           │
│  • Конфликты при записи на Replica                        │
│  • Сложнее диагностика                                    │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  ✅ Выборочная репликация (только важные таблицы)          │
│  ✅ Миграция между версиями                                │

### 1. HASH-BASED SHARDING

text

```
┌────────────────────────────────────────────────────────────┐
│  1. HASH-BASED SHARDING ⭐ (САМЫЙ ПОПУЛЯРНЫЙ)              │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ПРИНЦИП:                                                 │
│  Применяется хеш-функция к ключу (обычно user_id),       │
│  результат определяет номер шарда                         │
│                                                            │
│  ФОРМУЛА:                                                 │
│  shard_number = hash(key) % total_shards                  │
│                                                            │
│  ПРИМЕР (4 шарда):                                        │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ user_id │ hash(user_id) │ % 4 │ Shard              │ │
│  ├─────────┼───────────────┼─────┼────────────────────┤ │
│  │ 1001    │ 87234892      │ 0   │ Shard 0            │ │
│  │ 1002    │ 23847293      │ 1   │ Shard 1            │ │
│  │ 1003    │ 98237492      │ 2   │ Shard 2            │ │
│  │ 1004    │ 34892734      │ 3   │ Shard 3            │ │
│  │ 1005    │ 92837492      │ 0   │ Shard 0            │ │
│  │ 1006    │ 48237492      │ 1   │ Shard 1            │ │
│  └─────────┴───────────────┴─────┴────────────────────┘ │
│                                                            │
│  РЕАЛИЗАЦИЯ (на уровне приложения):                       │
│  ──────────────────────────────────────                    │
│  import hashlib                                           │
│                                                            │
│  def get_shard(user_id, total_shards=4):                  │
│      hash_value = int(hashlib.md5(                        │
│          str(user_id).encode()                            │
│      ).hexdigest(), 16)                                   │
│      return hash_value % total_shards                     │
│                                                            │
│  # Использование                                          │
│  user_id = 1001                                           │
│  shard = get_shard(user_id)  # → 0                        │
│  db = get_db_connection(f"shard_{shard}")                 │
│  db.execute("INSERT INTO users ...")                      │
│                                                            │
│  КОНФИГУРАЦИЯ ШАРДОВ:                                     │
│  ───────────────────────                                   │
│  SHARD_CONFIG = {                                         │
│      0: {"host": "shard0.db.com", "port": 5432},          │
│      1: {"host": "shard1.db.com", "port": 5432},          │
│      2: {"host": "shard2.db.com", "port": 5432},          │
│      3: {"host": "shard3.db.com", "port": 5432},          │
│  }                                                        │
│                                                            │
│  def get_db_connection(user_id):                          │
│      shard_num = get_shard(user_id)                       │
│      config = SHARD_CONFIG[shard_num]                     │
│      return psycopg2.connect(**config)                    │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Равномерное распределение данных                       │
│  • Простая реализация                                     │
│  • Предсказуемая нагрузка на шарды                        │
│  • Нет "горячих" шардов (hot spots)                       │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • СЛОЖНОЕ РЕШАРДИРОВАНИЕ (resharding)                    │
│    При добавлении шардов нужно перемещать данные!         │
│                                                            │
│    Было: hash(user_id) % 4                                │
│    Стало: hash(user_id) % 5                               │
│    → Почти ВСЕ данные переедут на другие шарды! 💥        │
│                                                            │
│  • Нельзя выполнить запрос по диапазону                   │
│    (все пользователи с user_id от 1000 до 2000)           │
│    → Нужно опросить ВСЕ шарды                             │
│                                                            │
│  • Сложность JOIN между шардами                           │
│    (пользователи и заказы могут быть на разных шардах)    │
│                                                            │
│  РЕШЕНИЕ ПРОБЛЕМЫ РЕШАРДИРОВАНИЯ:                         │
│  ──────────────────────────────────────                    │
│  CONSISTENT HASHING (Консистентное хеширование)           │
│                                                            │
│  Принцип:                                                 │
│  • Хеш-кольцо (0 до 2^32)                                 │
│  • Шарды и ключи располагаются на кольце                  │
│  • При добавлении шарда перемещается только ~1/N данных   │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐ │
│  │        Hash Ring (2^32 значений)                     │ │
│  │                                                      │ │
│  │         0 ─────────────────────────▶ 2^32            │ │
│  │         │        │        │        │                 │ │
│  │      Shard0   Shard1   Shard2   Shard3               │ │
│  │                                                      │ │
│  │ user_id=1001 → hash → попадает между Shard1 и Shard2 │ │
│  │                       → идёт на Shard2               │ │
│  │                                                      │ │
│  │ Добавили Shard4 между Shard2 и Shard3:               │ │
│  │ → Только часть данных Shard3 переедет на Shard4      │ │
│  │ → Остальные шарды НЕ ЗАТРОНУТЫ! ✅                   │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  Библиотеки для Consistent Hashing:                       │
│  • hash-ring (Python)                                     │
│  • consistent-hash (Node.js)                              │
│  • Ketama (C)                                             │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  ✅ Равномерное распределение нагрузки критично            │
│  ✅ Большое количество данных                              │
│  ✅ Нет сложных запросов по диапазонам                     │
│  ✅ Стандартный выбор для sharding                         │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 2. RANGE-BASED SHARDING

text

```
┌────────────────────────────────────────────────────────────┐
│  2. RANGE-BASED SHARDING (По диапазонам)                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ПРИНЦИП:                                                 │
│  Данные разбиваются на шарды по ДИАПАЗОНАМ значений ключа │
│                                                            │
│  ПРИМЕР (по user_id):                                     │
│  ┌─────────────────────────────────────────────────────┐  │
│  │ Shard 0: user_id от 0        до 250,000,000         │  │
│  │ Shard 1: user_id от 250,000,001 до 500,000,000      │  │
│  │ Shard 2: user_id от 500,000,001 до 750,000,000      │  │
│  │ Shard 3: user_id от 750,000,001 до 1,000,000,000    │  │
│  └─────────────────────────────────────────────────────┘  │
│                                                            │
│  ROUTING (маршрутизация):                                 │
│  ────────────────────────                                  │
│  SHARD_RANGES = [                                         │
│      (0,         250_000_000, "shard0.db.com"),           │
│      (250_000_001, 500_000_000, "shard1.db.com"),         │
│      (500_000_001, 750_000_000, "shard2.db.com"),         │
│      (750_000_001, 1_000_000_000, "shard3.db.com"),       │
│  ]                                                        │
│                                                            │
│  def get_shard_by_range(user_id):                         │
│      for start, end, host in SHARD_RANGES:                │
│          if start <= user_id <= end:                      │
│              return host                                  │
│      raise ValueError("user_id out of range")             │
│                                                            │
│  # Использование                                          │
│  user_id = 123_456_789                                    │
│  shard = get_shard_by_range(user_id)  # → shard0.db.com   │
│                                                            │
│  ПРИМЕР (по дате):                                        │
│  ┌─────────────────────────────────────────────────────┐  │
│  │ Shard 2021: заказы за 2021 год                      │  │
│  │ Shard 2022: заказы за 2022 год                      │  │
│  │ Shard 2023: заказы за 2023 год                      │  │
│  │ Shard 2024: заказы за 2024 год (активный)           │  │
│  └─────────────────────────────────────────────────────┘  │
│                                                            │
│  def get_shard_by_year(order_date):                       │
│      year = order_date.year                               │
│      return f"shard_{year}.db.com"                        │
│                                                            │
│  # Запрос за 2023 год                                     │
│  SELECT * FROM orders                                     │
│  WHERE created_at BETWEEN '2023-01-01' AND '2023-12-31';  │
│  → Обращаемся ТОЛЬКО к shard_2023 ✅                      │
│                                                            │
│  ПРИМЕР (по алфавиту):                                    │
│  ┌─────────────────────────────────────────────────────┐  │
│  │ Shard A-G: фамилии от A до G                        │  │
│  │ Shard H-N: фамилии от H до N                        │  │
│  │ Shard O-T: фамилии от O до T                        │  │
│  │ Shard U-Z: фамилии от U до Z                        │  │
│  └─────────────────────────────────────────────────────┘  │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Простота понимания (интуитивно)                        │
│  • ЭФФЕКТИВНЫЕ ЗАПРОСЫ ПО ДИАПАЗОНУ                       │
│    SELECT * FROM users WHERE user_id BETWEEN 1000 AND 2000│
│    → Обращаемся ТОЛЬКО к нужному шарду!                   │
│                                                            │
│  • Легко добавить новый шард                              │
│    Новый год → новый шард для заказов                     │
│                                                            │
│  • Легко архивировать старые данные                       │
│    Shard 2020 → перенести в холодное хранилище            │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • НЕРАВНОМЕРНОЕ РАСПРЕДЕЛЕНИЕ (Hot Spots!)               │
│                                                            │
│    Пример: социальная сеть                                │
│    • Новые пользователи: user_id > 900,000,000            │
│    • Shard 3 получает ВСЮ нагрузку (новые регистрации)    │
│    • Shard 0-2 простаивают (старые пользователи неактивны)│
│                                                            │
│  • Сложность балансировки                                 │
│    Нужно постоянно перенастраивать диапазоны              │
│                                                            │
│  • Проблема быстрого роста                                │
│    Один диапазон вырос больше других → ребалансировка     │
│                                                            │
│  РЕШЕНИЕ HOT SPOTS:                                       │
│  ────────────────────                                      │
│  1. Динамическое разбиение диапазонов                     │
│     • Шард 3 переполнен? Разделить на Shard 3A и 3B       │
│                                                            │
│  2. Комбинация с hash                                     │
│     • Сначала диапазон (год), потом hash внутри диапазона │
│                                                            │
│  3. Мониторинг и ребалансировка                           │
│     • Автоматическое перераспределение при дисбалансе     │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  ✅ Данные имеют естественное упорядочивание               │
│     (время, ID, алфавит)                                  │
│  ✅ Частые запросы по диапазонам                           │
│  ✅ Архивирование старых данных                            │
│  ✅ Аналитика (OLAP)                                       │
│                                                            │
│  ❌ НЕ использовать:                                       │
│  • Неравномерный рост данных                              │
│  • Нужна равномерная нагрузка (используйте hash)          │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

│      "RU": "shard-eu.db.com",                             │
│      "CN": "shard-apac.db.com",                           │
│      "IN": "shard-apac.db.com",                           │
│      "JP": "shard-apac.db.com",                           │
│  }                                                        │
│                                                            │
│  def get_shard_by_country(user_country):                  │
│      return SHARD_BY_COUNTRY.get(user_country,            │
│                                  "shard-us.db.com")       │
│                                                            │
│  # Пользователь из Германии                               │
│  user_country = "DE"                                      │
│  shard = get_shard_by_country(user_country)               │
│  # → shard-eu.db.com                                      │
│                                                            │
│  ОПРЕДЕЛЕНИЕ РЕГИОНА:                                     │
│  ──────────────────────                                    │
│  1. По IP-адресу (GeoIP):                                 │
│     import geoip2.database                                │
│     reader = geoip2.database.Reader('GeoLite2-Country')   │
│     response = reader.country('92.168.1.1')               │
│     country = response.country.iso_code  # → "DE"         │
│                                                            │
│  2. По профилю пользователя:                              │
│     SELECT country FROM users WHERE id = 1001;            │
│     → Сохранено при регистрации                           │
│                                                            │
│  3. По DNS (GeoDNS):                                      │
│     • Пользователь запрашивает api.example.com            │
│     • DNS возвращает ближайший IP                         │
│     • api-us.example.com для USA                          │
│     • api-eu.example.com для EU                           │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • НИЗКАЯ ЗАДЕРЖКА (данные близко к пользователю)         │
│    USA пользователь → USA шард (10ms вместо 150ms)        │
│                                                            │
│  • СООТВЕТСТВИЕ ЗАКОНОДАТЕЛЬСТВУ (GDPR, data residency)   │
│    • GDPR: данные EU граждан должны храниться в EU        │
│    • Китай: данные должны храниться в Китае               │
│    → Geographic sharding решает эту проблему!             │
│                                                            │
│  • Изоляция при региональных сбоях                        │
│    Проблемы в USA → EU продолжает работать                │
│                                                            │
│  • Оптимизация по часовым поясам                          │
│    Бэкап в 2 AM локального времени каждого региона        │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • Сложность глобальных запросов                          │
│    "Все пользователи в мире" → опросить ВСЕ шарды         │
│                                                            │
│  • Пользователи путешествуют                              │
│    • Американец в Европе → обращается к USA шарду         │
│    • Высокая задержка                                     │
│    Решение: CDN, кэширование, temporary sharding          │
│                                                            │
│  • Неравномерное распределение                            │
│    • 60% пользователей в Азии                             │
│    • 30% в Европе                                         │
│    • 10% в USA                                            │
│    → APAC шард перегружен                                 │
│    Решение: дополнительный sharding внутри региона (hash) │
│                                                            │
│  • Сложность кросс-региональных фич                       │
│    Социальная сеть: USA пользователь дружит с EU          │
│    → Данные на разных шардах → сложные JOIN               │
│                                                            │
│  HYBRID APPROACH (гибридный подход):                      │
│  ─────────────────────────────────────                     │
│  Geographic + Hash:                                       │
│                                                            │
│  1. Сначала выбираем регион (geographic)                  │
│  2. Внутри региона — hash sharding                        │
│                                                            │
│  Пример:                                                  │
│  • EU Region:                                             │
│    - Shard EU-0: hash(user_id) % 4 == 0                   │
│    - Shard EU-1: hash(user_id) % 4 == 1                   │
│    - Shard EU-2: hash(user_id) % 4 == 2                   │
│    - Shard EU-3: hash(user_id) % 4 == 3                   │
│                                                            │
│  • US Region:                                             │
│    - Shard US-0, US-1, US-2, US-3                         │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  ✅ Глобальное приложение (многие страны)                  │
│  ✅ Низкая задержка критична                               │
│  ✅ Требования законодательства (GDPR, data residency)     │
│  ✅ Изоляция регионов (risk management)                    │
│                                                            │
│  ПРИМЕРЫ:                                                 │
│  • Facebook: шарды по регионам                            │
│  • Netflix: контент по регионам                           │
│  • Uber: шарды по городам                                 │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

```
┌────────────────────────────────────────────────────────────┐
│  4. DIRECTORY/LOOKUP SHARDING (Справочник)                 │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ПРИНЦИП:                                                 │
│  Отдельная БД (lookup service) хранит mapping:            │
│  "какой ключ на каком шарде находится"                    │
│                                                            │
│  АРХИТЕКТУРА:                                             │
│  ┌──────────────────────────────────────────────────────┐ │
│  │                  LOOKUP SERVICE                      │ │
│  │         (Маленькая быстрая БД/кэш)                   │ │
│  │                                                      │ │
│  │  ┌─────────────┬──────────────────┐                 │ │
│  │  │ Key         │ Shard            │                 │ │
│  │  ├─────────────┼──────────────────┤                 │ │
│  │  │ user:1001   │ shard-0.db.com   │                 │ │
│  │  │ user:1002   │ shard-2.db.com   │                 │ │
│  │  │ user:1003   │ shard-1.db.com   │                 │ │
│  │  │ user:1004   │ shard-0.db.com   │                 │ │
│  │  │ ...         │ ...              │                 │ │
│  │  └─────────────┴──────────────────┘                 │ │
│  └──────────────────────────────────────────────────────┘ │
│                    ↓                                       │
│  ┌─────────────┬─────────────┬─────────────┐             │
│  │  Shard 0    │  Shard 1    │  Shard 2    │             │
│  │  (users)    │  (users)    │  (users)    │             │
│  └─────────────┴─────────────┴─────────────┘             │
│                                                            │
│  ПРОЦЕСС ЗАПРОСА:                                         │
│  ───────────────────                                       │
│  1. Клиент: "Получить user_id=1001"                       │
│  2. Обращение к Lookup Service                            │
│     → Узнать: user:1001 находится на shard-0.db.com       │
│  3. Обращение к Shard 0                                   │
│     → SELECT * FROM users WHERE id = 1001;                │
│  4. Возврат данных клиенту                                │
│                                                            │
│  РЕАЛИЗАЦИЯ:                                              │
│  ─────────────                                             │
│  # Lookup Service (Redis для скорости)                    │
│  import redis                                             │
│                                                            │
│  lookup = redis.Redis(host='lookup.redis.com')            │
│                                                            │
│  def get_shard_for_user(user_id):                         │
│      shard = lookup.get(f"user:{user_id}")                │
│      if not shard:                                        │
│          raise ValueError("User not found in lookup")     │
│      return shard.decode()                                │
│                                                            │
│  def save_user_shard_mapping(user_id, shard):             │
│      lookup.set(f"user:{user_id}", shard)                 │
│                                                            │
│  # Использование                                          │
│  user_id = 1001                                           │
│  shard = get_shard_for_user(user_id)                      │
│  # → "shard-0.db.com"                                     │
│                                                            │
│  db = connect_to_shard(shard)                             │
│  db.execute("SELECT * FROM users WHERE id = %s", user_id) │
│                                                            │
│  LOOKUP TABLE (если не Redis, а БД):                      │
│  ──────────────────────────────────────                    │
│  CREATE TABLE shard_lookup (                              │
│      entity_type VARCHAR(50),  -- 'user', 'order', etc    │
│      entity_id BIGINT,                                    │
│      shard_host VARCHAR(255),                             │
│      PRIMARY KEY (entity_type, entity_id)                 │
│  );                                                       │
│                                                            │
│  CREATE INDEX idx_lookup ON shard_lookup(entity_id);      │
│                                                            │
│  INSERT INTO shard_lookup VALUES                          │
│  ('user', 1001, 'shard-0.db.com'),                        │
│  ('user', 1002, 'shard-2.db.com'),                        │
│  ('order', 5001, 'shard-1.db.com');                       │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • МАКСИМАЛЬНАЯ ГИБКОСТЬ                                  │
│    Любая логика распределения (произвольная)              │
│                                                            │
│  • Легкая миграция данных между шардами                   │
│    Переместить user:1001 с shard-0 на shard-1:            │
│    1. Скопировать данные                                  │
│    2. Обновить lookup: user:1001 → shard-1                │
│    3. Удалить со старого шарда                            │
│    → Клиенты автоматически используют новый шард!         │
│                                                            │
│  • Динамическая балансировка                              │
│    Shard перегружен? Переместить часть пользователей      │
│                                                            │
│  • Можно комбинировать стратегии                          │
│    • VIP пользователи → dedicated shard                   │
│    • Обычные пользователи → hash sharding                 │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  • ДОПОЛНИТЕЛЬНЫЙ ЗАПРОС (latency!)                       │
│    Каждый запрос = 2 запроса:                             │
│    1. Lookup (где данные?)                                │
│    2. Shard (получить данные)                             │
│    → +10-50ms задержка                                    │
│                                                            │
│  • Lookup Service = ЕДИНАЯ ТОЧКА ОТКАЗА                   │
│    Если lookup упал → ВСЯ система недоступна              │
│    Решение: репликация lookup, кэширование на клиенте     │
│                                                            │
│  • Сложность синхронизации lookup                         │
│    Lookup устарел → запросы к неправильному шарду         │
│                                                            │
│  • Дополнительная инфраструктура                          │
│    Нужно поддерживать ещё одну БД/сервис                  │
│                                                            │
│  ОПТИМИЗАЦИИ:                                             │
│  ──────────────                                            │
│  1. Кэширование lookup на клиенте                         │
│     from functools import lru_cache                       │
│                                                            │
│     @lru_cache(maxsize=10000)                             │
│     def get_shard_cached(user_id):                        │
│         return get_shard_for_user(user_id)                │
│                                                            │
│     # Первый запрос: обращение к lookup                   │
│     # Последующие: из кэша ⚡                              │
│                                                            │
│  2. Batch lookup (пакетные запросы)                       │
│     user_ids = [1001, 1002, 1003, 1004, 1005]             │
│     shards = lookup.mget([f"user:{uid}" for uid in user_ids])│
│     # → Один запрос вместо 5                              │
│                                                            │
│  3. Репликация lookup                                     │
│     • Master lookup (запись)                              │
│     • Replicas (чтение) ← масштабирование                 │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  ✅ Нужна максимальная гибкость                            │
│  ✅ Частая миграция данных между шардами                   │
│  ✅ Сложная логика распределения                           │
│  ✅ VIP пользователи (dedicated resources)                 │
│                                                            │
│  ❌ НЕ использовать:                                       │
│  • Низкая задержка критична (дополнительный hop)          │
│  • Простое распределение (hash достаточно)                │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## Детальные методы обеспечения HA

### 1. РЕПЛИКАЦИЯ (основа HA)

text

```
┌────────────────────────────────────────────────────────────┐
│  1. РЕПЛИКАЦИЯ КАК ОСНОВА ОТКАЗОУСТОЙЧИВОСТИ               │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  БАЗОВАЯ АРХИТЕКТУРА:                                      │
│  ┌──────────────────────────────────────────────────────┐ │
│  │                                                      │ │
│  │       ┌─────────────┐                                │ │
│  │       │   Primary   │ ← Write (активный)             │ │
│  │       │  (Master)   │                                │ │
│  │       └──────┬──────┘                                │ │
│  │              │                                       │ │
│  │         Репликация                                   │ │
│  │         (streaming)                                  │ │
│  │              │                                       │ │
│  │     ┌────────┴────────┐                              │ │
│  │     ▼                 ▼                              │ │
│  │ ┌─────────┐      ┌─────────┐                         │ │
│  │ │Standby 1│      │Standby 2│ ← Read (пассивные)      │ │
│  │ │(Replica)│      │(Replica)│   Готовы стать Primary  │ │
│  │ └─────────┘      └─────────┘                         │ │
│  │                                                      │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                           │
│  ТИПЫ STANDBY (реплик):                                   │
│  ───────────────────────                                   │
│  1. HOT STANDBY (горячий резерв)                          │
│     • Постоянно реплицирует данные                        │
