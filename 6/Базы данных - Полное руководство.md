## 1. Реляционные СУБД (RDBMS)

### Что это такое?

**Реляционная СУБД** — система управления базами данных, основанная на **реляционной модели данных** (предложена Эдгаром Коддом в 1970).

text

```
┌────────────────────────────────────────────────────────────┐
│         РЕЛЯЦИОННАЯ МОДЕЛЬ                                 │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Данные = Таблицы (Relations)                             │
│  Строки = Кортежи (Tuples)                                │
│  Столбцы = Атрибуты (Attributes)                          │
│  Связи = Внешние ключи (Foreign Keys)                     │
│                                                            │
│  Пример:                                                  │
│  ┌─────────────────────────────────────────────────────┐  │
│  │ Таблица: users                                      │  │
│  ├────┬──────────┬─────────────────┬───────────────────┤  │
│  │ id │ name     │ email           │ created_at        │  │
│  ├────┼──────────┼─────────────────┼───────────────────┤  │
│  │ 1  │ Иван     │ ivan@mail.ru    │ 2024-01-15        │  │
│  │ 2  │ Мария    │ maria@mail.ru   │ 2024-01-16        │  │
│  └────┴──────────┴─────────────────┴───────────────────┘  │
│                                                            │
│  ┌─────────────────────────────────────────────────────┐  │
│  │ Таблица: orders                                     │  │
│  ├────┬─────────┬──────────┬───────────────────────────┤  │
│  │ id │ user_id │ total    │ status                    │  │
│  ├────┼─────────┼──────────┼───────────────────────────┤  │
│  │ 1  │ 1  ───┐ │ 5000     │ completed                 │  │
│  │ 2  │ 1  ───┤ │ 3200     │ pending                   │  │
│  │ 3  │ 2  ───┼─┼──────────┼───────────────────────────┤  │
│  └────┴────│───┘─┴──────────┴───────────────────────────┘  │

│           └─ Foreign Key связывает таблицы                │
│                                                            │
```

#### Ключевые компоненты

text

```
┌────────────────────────────────────────────────────────────┐
│         КЛЮЧЕВЫЕ МЕХАНИЗМЫ                                 │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. QUERY OPTIMIZER (планировщик)                         │
│  ─────────────────────────────────                         │
│  Выбирает оптимальный способ выполнения запроса           │
│                                                            │
│  Запрос: SELECT * FROM orders WHERE user_id = 1           │
│                                                            │
│  Варианты выполнения:                                     │
│  • Full Table Scan (просканировать всю таблицу)           │
│  • Index Scan (использовать индекс по user_id) ⭐         │
│                                                            │
│  Optimizer выбирает на основе:                            │
│  • Статистика (сколько строк в таблице)                   │
│  • Наличие индексов                                       │
│  • Селективность (сколько строк вернётся)                 │
│                                                            │
│  EXPLAIN SELECT ... ← посмотреть план выполнения          │
│                                                            │
├────────────────────────────────────────────────────────────┤
│  2. ИНДЕКСЫ (B-Tree)                                      │
│  ────────────────────                                      │
│  Ускоряют поиск с O(N) до O(log N)                        │
│                                                            │
│  Структура B-Tree:                                        │
│                    [50]                                   │
│                  /      \                                 │
│              [25]        [75]                             │
│             /   \        /   \                            │
│         [10]  [30]   [60]  [90]                           │
│                                                            │
│  CREATE INDEX idx_email ON users(email);                  │
│                                                            │
│  Типы индексов:                                           │
│  • B-Tree (по умолчанию) — для =, <, >, BETWEEN          │
│  • Hash — только для =                                    │
│  • GIN/GiST (PostgreSQL) — для JSON, массивов             │
│  • Full-text — для текстового поиска                      │
│  • Spatial — для геоданных                                │
│                                                            │
├────────────────────────────────────────────────────────────┤
│  3. ТРАНЗАКЦИИ (ACID)                                     │
│  ─────────────────────                                     │
│  BEGIN;                                                   │
│    UPDATE accounts SET balance = balance - 100            │
│      WHERE id = 1;                                        │
│    UPDATE accounts SET balance = balance + 100            │
│      WHERE id = 2;                                        │
│  COMMIT;  -- Обе операции или ни одной                    │
│                                                            │
│  Уровни изоляции:                                         │
│  • READ UNCOMMITTED (грязное чтение)                      │
│  • READ COMMITTED (по умолчанию) ⭐                        │
│  • REPEATABLE READ                                        │
│  • SERIALIZABLE (самый строгий)                           │
│                                                            │
├────────────────────────────────────────────────────────────┤
│  4. WAL (Write-Ahead Log)                                 │
│  ─────────────────────────                                 │
│  Сначала пишем в лог, потом в основные файлы              │
│                                                            │
│  1. Запрос: UPDATE users SET name = 'New' WHERE id = 1    │
│  2. Запись в WAL: "изменить строку id=1 на name=New"      │
│  3. Flush WAL на диск (быстро, sequential write)          │
│  4. COMMIT успешен                                        │
│  5. Позже: применить изменения к основным файлам          │
│                                                            │
│  Гарантирует Durability даже при сбое!                    │
│                                                            │
├────────────────────────────────────────────────────────────┤
│  5. MVCC (Multi-Version Concurrency Control)              │
│  ────────────────────────────────────────────              │
│  Каждая транзакция видит свой "снимок" данных             │
│                                                            │
│  T1: BEGIN; SELECT balance FROM accounts WHERE id=1;      │
│      → видит balance = 1000                               │
│                                                            │
│  T2: BEGIN; UPDATE accounts SET balance=2000 WHERE id=1;  │
│      COMMIT;                                              │
│                                                            │
│  T1: SELECT balance FROM accounts WHERE id=1;             │
│      → всё ещё видит 1000! (REPEATABLE READ)              │
│                                                            │
│  Без блокировок для чтения!                               │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 2. Нереляционные СУБД (NoSQL)

### Что это такое?

**NoSQL** = "Not Only SQL" — базы данных, которые **отказываются от реляционной модели** в пользу других подходов для решения специфических задач.

text

```
┌────────────────────────────────────────────────────────────┐
│         ЗАЧЕМ ПОЯВИЛИСЬ NoSQL?                             │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ПРОБЛЕМЫ RDBMS В 2000-х:                                 │
│                                                            │
│  1. МАСШТАБИРОВАНИЕ                                       │
│     • Google, Facebook, Amazon — петабайты данных         │
│     • RDBMS плохо масштабируются горизонтально            │
│     • Вертикальное масштабирование = дорого               │
│                                                            │
│  2. СКОРОСТЬ                                              │
│     • Миллионы запросов в секунду                         │
│     • JOIN'ы медленные на больших объёмах                 │
│     • ACID накладные расходы                              │
│                                                            │
│  3. ГИБКОСТЬ СХЕМЫ                                        │
│     • Схема меняется часто (agile-разработка)             │
│     • Миграции БД = даунтайм                              │
│     • Разные объекты имеют разные поля                    │
│                                                            │
│  4. ГЕОГРАФИЧЕСКОЕ РАСПРЕДЕЛЕНИЕ                          │
│     • Данные в разных дата-центрах по всему миру          │
│     • ACID сложно поддерживать                            │
│                                                            │
│  РЕШЕНИЕ → NoSQL с другими компромиссами                  │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

```
│ ╱─────────────────────╲                                    │
│A                       P                                   │
│Availability            Partition Tolerance                 │
│(Доступность)           (Устойчивость к разделению)         │
│         ╲             ╱                                    │
│          ╲   AP     ╱                                      │
│           ╲  базы  ╱                                       │
│            ╲      ╱                                        │
│             ╲    ╱                                         │
│              ╲  ╱                                          │
│               ╲╱                                           │
│                                                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  C (Consistency):                                         │
│  Все узлы видят одинаковые данные одновременно            │
│                                                            │
│  A (Availability):                                        │
│  Система всегда отвечает на запросы (даже при сбоях)      │
│                                                            │
│  P (Partition Tolerance):                                 │
│  Система работает при разрыве связи между узлами          │
│                                                            │
├────────────────────────────────────────────────────────────┤
│  ПРИМЕРЫ:                                                 │
│                                                            │
│  CP (жертвуем доступностью):                              │
│  • MongoDB, HBase, Redis Cluster                          │
│  • При разделении сети — часть узлов недоступна           │
│  • Гарантия: данные согласованы                           │
│                                                            │
│  AP (жертвуем консистентностью):                          │
│  • Cassandra, DynamoDB, Riak, CouchDB                     │
│  • Всегда отвечают (может быть старая версия данных)      │
│  • Eventual Consistency (в конце концов станут одинаковы) │
│                                                            │
│  CA (без устойчивости к разделению):                      │
│  • Традиционные RDBMS (PostgreSQL, MySQL)                 │
│  • Один узел или синхронная репликация                    │
│  • На практике P неизбежно → выбор между CP и AP          │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### BASE vs ACID

text

```
┌────────────────────────────────────────────────────────────┐
│         ACID (RDBMS) vs BASE (NoSQL)                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ACID:                                                    │
│  ─────                                                     │
│  A — Atomicity        (атомарность)                       │
│  C — Consistency      (согласованность)                   │
│  I — Isolation        (изолированность)                   │
│  D — Durability       (долговечность)                     │
│                                                            │
│  Принцип: СТРОГИЕ ГАРАНТИИ                                │
│  • Данные всегда корректны                                │
│  • Транзакции блокируют ресурсы                           │
│  • Вертикальное масштабирование                           │
│                                                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  BASE:                                                    │
│  ─────                                                     │
│  BA — Basically Available    (в основном доступна)        │
│  S  — Soft state             (мягкое состояние)           │
│  E  — Eventual consistency   (согласованность в итоге)    │
│                                                            │
│  Принцип: ДОСТУПНОСТЬ ВАЖНЕЕ КОНСИСТЕНТНОСТИ              │
│  • Система всегда отвечает                                │
│  • Данные могут быть временно рассогласованы              │
│  • Горизонтальное масштабирование                         │
│                                                            │
├────────────────────────────────────────────────────────────┤
│  ПРИМЕР:                                                  │
│                                                            │
│  ACID (банк):                                             │
│  Перевод 100₽ со счёта A на счёт B                        │
│  → Блокировка обоих счетов                                │
│  → Списание и зачисление атомарно                         │
│  → Все видят изменения одновременно                       │
│  → COMMIT                                                 │
│                                                            │
│  BASE (лайки в Instagram):                                │
│  +1 лайк к посту                                          │
│  → Запись на узел в США                                   │
│  → Пользователь в Европе видит старое значение (100ms)    │
│  → Через секунду — данные синхронизированы                │
│  → Eventual consistency — это нормально!                  │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

```
│  Client → API → PostgreSQL (10-100ms)                     │
│                                                            │
│  С КЭШЕМ:                                                 │
│  Client → API → Redis (0.1-1ms) ← 100x быстрее!           │
│                    └─ miss → PostgreSQL → сохранить в Redis│
│                                                            │
│  Паттерны:                                                │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ Cache-Aside (Lazy Loading)                           │ │
│  │ ──────────────────────────                            │ │
│  │ 1. Запрос к Redis                                    │ │
│  │ 2. Если нет → запрос к БД → сохранить в Redis        │ │
│  │ 3. Вернуть данные                                    │ │
│  │                                                      │ │
│  │ Пример:                                              │ │
│  │ data = redis.get("product:123")                      │ │
│  │ if not data:                                         │ │
│  │     data = db.query("SELECT * FROM products...")     │ │
│  │     redis.set("product:123", data, ex=3600)          │ │
│  │ return data                                          │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ Write-Through                                        │ │
│  │ ─────────────                                         │ │
│  │ При записи → обновить БД И кэш одновременно          │ │
│  │                                                      │ │
│  │ db.update(...)                                       │ │
│  │ redis.set("product:123", new_data)                   │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  ЧТО КЭШИРОВАТЬ:                                          │
│  • Результаты SQL-запросов                                │
│  • Ответы API (третьих сторон)                            │
│  • HTML-страницы (page caching)                           │
│  • Computed values (тяжёлые вычисления)                   │
│  • Справочники (страны, категории)                        │
│                                                            │
│  TTL (Time To Live):                                      │
│  SET "product:123" "{...}" EX 3600  # Удалится через 1ч   │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  2. СЕССИИ ПОЛЬЗОВАТЕЛЕЙ                                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Зачем? Хранить состояние авторизованного пользователя    │
│                                                            │
│  Проблема с БД:                                           │
│  • Каждый запрос → SELECT session → медленно              │
│  • Блокировки при обновлении                              │
│                                                            │
│  Решение Redis:                                           │
│  • In-memory → <1ms                                       │
│  • Автоудаление (TTL) → не нужно чистить вручную          │
│                                                            │
│  Пример:                                                  │
│  # При логине                                             │
│  session_id = generate_uuid()                             │
│  redis.hset(f"session:{session_id}", mapping={            │
│      "user_id": 1001,                                     │
│      "role": "admin",                                     │
│      "ip": "192.168.1.1"                                  │
│  })                                                       │
│  redis.expire(f"session:{session_id}", 1800)  # 30 мин    │
│                                                            │
│  # При каждом запросе                                     │
│  session = redis.hgetall(f"session:{session_id}")         │
│  if session:                                              │
│      user_id = session['user_id']                         │
│      # Продлить TTL                                       │
│      redis.expire(f"session:{session_id}", 1800)          │
│                                                            │
│  АЛЬТЕРНАТИВА HASH:                                       │
│  redis.set(f"session:{sid}", json.dumps(data), ex=1800)   │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  3. ОЧЕРЕДИ ЗАДАЧ (Task Queues)                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Зачем? Отложенное выполнение фоновых задач               │
│                                                            │
│  Архитектура:                                             │
│  ┌─────────┐   LPUSH   ┌───────┐   BRPOP   ┌──────────┐  │
│  │ Web App │ ────────▶ │ Redis │ ────────▶ │  Worker  │  │
│  │         │  (task)   │ Queue │  (task)   │  Process │  │
│  └─────────┘           └───────┘           └──────────┘  │
│                                                            │
│  Пример:                                                  │
│  # Producer (добавить задачу)                             │
│  task = json.dumps({                                      │
│      "type": "send_email",                                │
│      "to": "user@example.com",                            │
│      "subject": "Welcome!",                               │
│      "body": "..."                                        │
│  })                                                       │
│  redis.lpush("queue:emails", task)                        │
│                                                            │
│  # Consumer (обработать задачу)                           │
│  while True:                                              │
│      # BRPOP блокирует до появления элемента             │
│      _, task_json = redis.brpop("queue:emails", timeout=0)│
│      task = json.loads(task_json)                         │
│      send_email(task)                                     │
│                                                            │
│  ФРЕЙМВОРКИ:                                              │
│  • Celery (Python) → использует Redis как брокер          │
│  • Sidekiq (Ruby) → Redis                                 │
│  • Bull (Node.js) → Redis                                 │
│  • RQ (Python) → Redis-based queue                        │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  4. RATE LIMITING (Ограничение запросов)                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Зачем? Защита API от перегрузки и DDoS                   │
│                                                            │
│  Алгоритм "Fixed Window":                                 │
│  • Максимум 100 запросов в минуту от одного пользователя  │
│                                                            │
│  Реализация:                                              │
│  def is_allowed(user_id):                                 │
│      key = f"rate:{user_id}:{current_minute()}"           │
│      count = redis.incr(key)                              │
│      if count == 1:                                       │
│          redis.expire(key, 60)  # TTL = 1 минута          │
│      return count <= 100                                  │
│                                                            │
│  Использование:                                           │
│  @app.route('/api/data')                                  │
│  def api_endpoint():                                      │
│      if not is_allowed(current_user.id):                  │
│          return {"error": "Too many requests"}, 429       │
│      return get_data()                                    │
│                                                            │
│  ПРОДВИНУТЫЕ АЛГОРИТМЫ:                                   │
│  • Sliding Window (точнее)                                │
│  • Token Bucket (Lua scripts)                             │
│  • Leaky Bucket                                           │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  5. LEADERBOARDS (Таблицы лидеров)                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Зачем? Рейтинги игроков, топ пользователей               │
│                                                            │
│  Структура: SORTED SET (хранит элементы + score)          │
│                                                            │
│  Пример (игра):                                           │
│  # Добавить/обновить счёт                                 │
│  redis.zadd("leaderboard", {                              │
│      "player:42": 9500,                                   │
│      "player:17": 8700,                                   │
│      "player:99": 12000                                   │
│  })                                                       │
│                                                            │
│  # Топ-10 игроков                                         │
│  top10 = redis.zrevrange("leaderboard", 0, 9,             │
│                          withscores=True)                 │
│  # [('player:99', 12000), ('player:42', 9500), ...]       │
│                                                            │
│  # Позиция игрока                                         │
│  rank = redis.zrevrank("leaderboard", "player:42")        │
│  # → 1 (второе место, 0-indexed)                          │
│                                                            │
│  # Увеличить счёт                                         │
│  redis.zincrby("leaderboard", 100, "player:42")           │
│                                                            │
│  # Игроки в диапазоне очков                               │
│  players = redis.zrangebyscore("leaderboard",             │
│                                8000, 10000)               │
│                                                            │
│  ПРИМЕНЕНИЕ:                                              │
│  • Игровые рейтинги                                       │
│  • Топ-постов (по лайкам)                                 │
│  • Trending topics                                        │
│  • Популярные товары                                      │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  6. PUB/SUB (Real-time messaging)                         │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Зачем? Рассылка событий множеству подписчиков            │
│                                                            │
│  Архитектура:                                             │
│  ┌──────────┐                ┌──────────┐                 │
│  │Publisher │ ─PUBLISH───▶   │ Channel  │                 │
│  └──────────┘    msg         │"chat:1"  │                 │
│                              └────┬─────┘                 │
│                    ┌──────────────┼──────────────┐        │
│                    ▼              ▼              ▼        │
│              ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│              │Subscriber│  │Subscriber│  │Subscriber│    │
│              │    1     │  │    2     │  │    3     │    │
│              └──────────┘  └──────────┘  └──────────┘    │
│                                                            │
│  Пример (чат):                                            │
│  # Publisher                                              │
│  redis.publish("chat:room1", json.dumps({                 │
│      "user": "Иван",                                      │
│      "message": "Привет!"                                 │
│  }))                                                      │
│                                                            │
│  # Subscriber                                             │
│  pubsub = redis.pubsub()                                  │
│  pubsub.subscribe("chat:room1")                           │
│  for message in pubsub.listen():                          │
│      if message['type'] == 'message':                     │
│          data = json.loads(message['data'])               │
│          print(f"{data['user']}: {data['message']}")      │
│                                                            │
│  ПРИМЕНЕНИЕ:                                              │
│  • Чаты                                                   │
│  • Live-обновления (цены, ставки)                         │
│  • Уведомления                                            │
│  • Event-driven архитектура (простая)                     │
│                                                            │
│  ⚠️ ОГРАНИЧЕНИЯ:                                           │
│  • Fire-and-forget (нет гарантии доставки)                │
│  • Если подписчик офлайн → сообщения теряются             │
│  • Для надёжности → Redis Streams                         │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  7. СЧЁТЧИКИ И АНАЛИТИКА                                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  INCR/DECR — атомарные операции                           │
│                                                            │
│  Примеры:                                                 │
│  # Просмотры статьи                                       │
│  redis.incr("views:article:123")                          │
│                                                            │
│  # Статистика по дням                                     │
│  redis.hincrby("stats:2024-01-15", "pageviews", 1)        │
│  redis.hincrby("stats:2024-01-15", "unique_users", 1)     │
│                                                            │
│  # Уникальные посетители (HyperLogLog)                    │
│  redis.pfadd("visitors:2024-01-15", "user:1001")          │
│  redis.pfadd("visitors:2024-01-15", "user:1002")          │
│  count = redis.pfcount("visitors:2024-01-15")             │
│  # Точность ±0.81%, память 12KB для миллиардов!           │
│                                                            │
│  # Онлайн-пользователи (BITMAP)                           │
│  redis.setbit("online:2024-01-15:10:00", user_id, 1)      │
│  online_count = redis.bitcount("online:2024-01-15:10:00") │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  8. РАСПРЕДЕЛЁННЫЕ БЛОКИРОВКИ (Distributed Locks)         │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Зачем? Критические секции в микросервисах                │
│                                                            │
│  Простая блокировка:                                      │
│  acquired = redis.set("lock:resource", "owner123",        │
│                       nx=True, ex=30)                     │
│  if acquired:                                             │
│      # Критическая секция                                 │
│      process_payment()                                    │
│      redis.delete("lock:resource")                        │
│  else:                                                    │
│      # Ресурс заблокирован                                │
│                                                            │
│  REDLOCK (для кластера):                                  │
│  • Захват блокировки на большинстве узлов                 │
│  • Устойчиво к сбоям                                      │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  9. ГЕОДАННЫЕ (Geospatial)                                │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Зачем? Поиск ближайших объектов                          │
│                                                            │
│  Пример (такси):                                          │
│  # Добавить машину                                        │
│  redis.geoadd("drivers", 37.6173, 55.7558, "driver:42")   │
│                                                            │
│  # Найти машины в радиусе 5км                             │
│  nearby = redis.georadius("drivers",                      │
│                           37.62, 55.75,                   │
│                           5, unit='km',                   │
│                           withdist=True)                  │
│  # [('driver:42', 0.5км), ...]                            │
│                                                            │
│  ПРИМЕНЕНИЕ:                                              │
│  • Uber/Taxi (поиск водителей)                            │
│  • Рестораны рядом                                        │
│  • Доставка (курьеры)                                     │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  10. REDIS STREAMS (Events/Logs) NEW!                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Зачем? Лог событий + Consumer Groups (альтернатива Kafka)│
│                                                            │
│  Пример:                                                  │
│  # Producer                                               │
│  redis.xadd("events", {                                   │
│      "user_id": "1001",                                   │
│      "action": "login",                                   │
│      "ip": "192.168.1.1"                                  │
│  })                                                       │
│                                                            │
│  # Consumer                                               │
│  events = redis.xread({"events": "0"}, count=10)          │
│                                                            │
│  # Consumer Groups (как Kafka)                            │
│  redis.xgroup_create("events", "analytics", id="0")       │
│  events = redis.xreadgroup("analytics", "consumer1",      │
│                            {"events": ">"}, count=10)     │
│                                                            │
│  ПРИМЕНЕНИЕ:                                              │
│  • Event Sourcing                                         │
│  • Activity Streams                                       │
│  • Change Data Capture (CDC)                              │
│  • IoT события                                            │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

```
│      createdAt: {                                         │
│        $gte: ISODate("2024-01-01"),                       │
│        $lt: ISODate("2024-02-01")                         │
│      },                                                   │
│      status: "completed"                                  │
│    }},                                                    │
│                                                            │
│    // 2. Развернуть массив items                         │
│    {$unwind: "$items"},                                   │
│                                                            │
│    // 3. JOIN с products (lookup)                        │
│    {$lookup: {                                            │
│      from: "products",                                    │
│      localField: "items.productId",                       │
│      foreignField: "_id",                                 │
│      as: "product"                                        │
│    }},                                                    │
│                                                            │
│    // 4. Группировка по категории                        │
│    {$group: {                                             │
│      _id: "$product.category",                            │
│      totalRevenue: {                                      │
│        $sum: {$multiply: [                                │
│          "$items.quantity",                               │
│          "$items.price"                                   │
│        ]}                                                 │
│      },                                                   │
│      orderCount: {$sum: 1}                                │
│    }},                                                    │
│                                                            │
│    // 5. Сортировка                                      │
│    {$sort: {totalRevenue: -1}},                           │
│                                                            │
│    // 6. Топ-5                                           │
│    {$limit: 5}                                            │
│  ])                                                       │
│                                                            │
│  Результат:                                               │
│  [                                                        │
│    {_id: "electronics", totalRevenue: 5000000, ...},      │
│    {_id: "clothing", totalRevenue: 3500000, ...},         │
│    ...                                                    │
│  ]                                                        │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  7. ЛОГИ И СОБЫТИЯ                                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ЗАЧЕМ MongoDB:                                           │
│  • Гибкая структура логов (разные типы событий)           │

│  • TTL индексы (автоудаление старых логов)                │
│                                                            │
│  Пример:                                                  │
│  {                                                        │
│    "timestamp": ISODate("2024-01-15T10:00:00Z"),          │
│    "level": "error",                                      │
│    "service": "auth-service",                             │
│    "message": "Failed login attempt",                     │
│    "context": {                                           │
│      "userId": "user:1001",                               │
│      "ip": "192.168.1.1",                                 │
│      "reason": "invalid_password"                         │
│    }                                                      │
│  }                                                        │
│                                                            │
│  TTL INDEX (автоудаление через 30 дней):                  │
│  db.logs.createIndex(                                     │
│    {"timestamp": 1},                                      │
│    {expireAfterSeconds: 2592000}                          │
│  )                                                        │
│                                                            │
│  CAPPED COLLECTIONS (фиксированный размер):               │
│  db.createCollection("logs", {                            │
│    capped: true,                                          │
│    size: 1073741824,  // 1GB                              │
│    max: 1000000       // Макс. документов                 │
│  })                                                       │
│  → Старые документы автоматически удаляются               │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### Практический пример использования

Python

```
# ОДИНАКОВЫЙ КОД ДЛЯ ОБЕИХ СУБД (ES 7.x / OpenSearch 2.x)

from elasticsearch import Elasticsearch  # или opensearchpy

# Подключение
es = Elasticsearch(['http://localhost:9200'])

# Создание индекса
es.indices.create(index='products', body={
    "mappings": {
        "properties": {
            "title": {"type": "text"},
            "price": {"type": "float"},
            "category": {"type": "keyword"}
        }
    }
})

# Индексация документа
es.index(index='products', id=1, body={
    "title": "MacBook Pro 16",
    "price": 249999,
    "category": "laptops"
})

# Поиск
result = es.search(index='products', body={
    "query": {
        "match": {
            "title": "macbook"
        }
    }
})

print(result['hits']['hits'])
```
