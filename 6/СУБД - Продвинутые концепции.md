### 1. Что такое схема в СУБД?

text

```
┌────────────────────────────────────────────────────────────┐
│         СХЕМА (SCHEMA) В СУБД                              │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Схема — это структурное описание базы данных             │
│                                                            │
│  В РЕЛЯЦИОННЫХ СУБД СХЕМА ОПРЕДЕЛЯЕТ:                     │
│  • Таблицы (Tables)                                       │
│  • Столбцы (Columns) + типы данных                        │
│  • Ограничения (Constraints)                              │
│  • Индексы (Indexes)                                      │
│  • Связи (Foreign Keys)                                   │
│  • Представления (Views)                                  │
│  • Триггеры, процедуры                                    │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ДВА ЗНАЧЕНИЯ ТЕРМИНА "СХЕМА":                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. ЛОГИЧЕСКАЯ СТРУКТУРА (общий смысл)                    │
│  ─────────────────────────────────────────                 │
│  Описание всей структуры БД                               │
│                                                            │
│  Пример схемы базы данных магазина:                       │
│                                                            │
│  CREATE TABLE users (                                     │
│      id          SERIAL PRIMARY KEY,                      │
│      email       VARCHAR(255) UNIQUE NOT NULL,            │
│      password    VARCHAR(255) NOT NULL,                   │
│      created_at  TIMESTAMP DEFAULT NOW()                  │
│  );                                                       │
│                                                            │
│  CREATE TABLE products (                                  │
│      id          SERIAL PRIMARY KEY,                      │
│      name        VARCHAR(255) NOT NULL,                   │
│      price       DECIMAL(10,2) NOT NULL,                  │
│      category_id INT REFERENCES categories(id)            │
│  );                                                       │
│                                                            │
│  CREATE TABLE orders (                                    │
│      id          SERIAL PRIMARY KEY,                      │
│      user_id     INT REFERENCES users(id),                │
│      total       DECIMAL(10,2),                           │
│      status      VARCHAR(50),                             │
│      created_at  TIMESTAMP DEFAULT NOW()                  │
│  );                                                       │
│                                                            │
│  ↑ Это "схема базы данных" — её структура                │
│                                                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  2. NAMESPACE (PostgreSQL, SQL Server, Oracle)            │
│  ──────────────────────────────────────────                │
│  Логический контейнер для объектов БД                     │
│                                                            │
│  База данных (Database)                                   │
│  ├── Schema "public"                                      │
│  │   ├── Table "users"                                    │
│  │   ├── Table "orders"                                   │
│  │   └── View "active_orders"                             │
│  ├── Schema "analytics"                                   │
│  │   ├── Table "stats"                                    │
│  │   └── View "monthly_report"                            │
│  └── Schema "archive"                                     │
│      └── Table "old_orders"                               │
│                                                            │
│  Зачем нужны схемы-namespace:                             │
│  • Организация объектов (как папки)                       │
│  • Разделение доступа (permissions)                       │
│  • Избежание конфликтов имён                              │
│                                                            │
│  PostgreSQL пример:                                       │
│  CREATE SCHEMA analytics;                                 │
│  CREATE TABLE analytics.stats (...);                      │
│                                                            │
│  SELECT * FROM analytics.stats;                           │
│  SELECT * FROM public.users;                              │
│                                                            │
│  ⚠️ MySQL НЕТ схем-namespace (Database = Schema)          │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  СХЕМА В NoSQL (ГИБКАЯ/ДИНАМИЧЕСКАЯ)                      │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  MongoDB, CouchDB — schema-less (гибкая схема)            │
│                                                            │
│  Документ 1:                                              │
│  {                                                        │
│    "name": "Иван",                                        │
│    "age": 30,                                             │
│    "email": "ivan@mail.ru"                                │
│  }                                                        │
│                                                            │
│  Документ 2 (другие поля!):                               │
│  {                                                        │
│    "name": "Мария",                                       │
│    "phone": "+79001234567",                               │
│    "address": {                                           │
│      "city": "Москва"                                     │
│    }                                                      │
│  }                                                        │
│                                                            │
│  ↑ Схема НЕ фиксирована, каждый документ уникален         │
│                                                            │
│  ⚖️ Преимущества:                                          │
│  • Быстрые изменения (нет миграций)                       │
│  • Гибкость                                               │
│                                                            │
│  ⚠️ Недостатки:                                            │
│  • Сложнее гарантировать целостность                      │
│  • Потенциальная несогласованность                        │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 2. Что такое индекс в РСУБД?

text

```
┌────────────────────────────────────────────────────────────┐
│         ИНДЕКС (INDEX) В РСУБД                             │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Индекс — это структура данных, ускоряющая поиск          │
│  в таблице (как предметный указатель в книге)             │
│                                                            │
│  БЕЗ ИНДЕКСА (Full Table Scan):                           │
│  ┌─────────────────────────────────────────────────────┐  │
│  │ id │ email                 │ name                   │  │
│  ├────┼───────────────────────┼────────────────────────┤  │
│  │ 1  │ alice@mail.ru         │ Alice                  │  │
│  │ 2  │ bob@mail.ru           │ Bob         ← сканируем│  │
│  │ 3  │ charlie@mail.ru       │ Charlie     ← всю       │  │
│  │ ...│ ...                   │ ...         ← таблицу!  │  │
│  │1000│ zack@mail.ru          │ Zack                   │  │
│  └────┴───────────────────────┴────────────────────────┘  │
│                                                            │
│  SELECT * FROM users WHERE email = 'bob@mail.ru';         │
│  → Сканирует 1000 строк → O(N) — медленно!                │
│                                                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  С ИНДЕКСОМ (B-Tree):                                     │
│  CREATE INDEX idx_email ON users(email);                  │
│                                                            │
│  ┌─────────────────────────────────────────────────────┐  │
│  │           ИНДЕКС (B-Tree структура)                 │  │
│  │                                                     │  │
│  │                  [m@...]                            │  │
│  │                 /        \                          │  │
│  │            [c@...]      [s@...]                     │  │
│  │           /      \      /      \                    │  │
│  │      [a@...]  [e@...] [n@...] [z@...]              │  │
│  │         │       │       │       │                   │  │
│  │         ▼       ▼       ▼       ▼                   │  │
│  │     Указатели на строки в таблице                   │  │
│  └─────────────────────────────────────────────────────┘  │
│                                                            │
│  SELECT * FROM users WHERE email = 'bob@mail.ru';         │
│  → Использует индекс → O(log N) ≈ 10 шагов для 1000 строк │
│  → ⚡ В 100 раз быстрее!                                   │
│                                                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  КАК РАБОТАЕТ ИНДЕКС:                                     │
│  ──────────────────────                                    │
│  1. Создаётся отдельная структура данных (обычно B-Tree)  │
│  2. Содержит копии значений индексируемого столбца        │
│  3. Отсортирована для быстрого поиска                     │
│  4. Хранит указатели на строки таблицы                    │
│                                                            │
│  СОЗДАНИЕ ИНДЕКСА:                                        │
│  CREATE INDEX idx_name ON table_name(column_name);        │
│                                                            │
│  ПРИМЕРЫ:                                                 │
│  CREATE INDEX idx_email ON users(email);                  │
│  CREATE INDEX idx_created ON orders(created_at);          │
│  CREATE INDEX idx_category_price ON products(category_id, price);│
│                                                            │
│  УНИКАЛЬНЫЙ ИНДЕКС:                                       │
│  CREATE UNIQUE INDEX idx_email ON users(email);           │
│  → Гарантирует уникальность + ускоряет поиск              │
│                                                            │
│  PRIMARY KEY автоматически создаёт уникальный индекс!     │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  КОГДА ИНДЕКСЫ ПОМОГАЮТ:                                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ✅ WHERE условия:                                         │
│  SELECT * FROM users WHERE email = '...';                 │
│  SELECT * FROM products WHERE price > 1000;               │
│                                                            │
│  ✅ JOIN:                                                  │
│  SELECT * FROM orders o                                   │
│  JOIN users u ON o.user_id = u.id;                        │
│  → Индексы на user_id и id ускоряют!                      │
│                                                            │
│  ✅ ORDER BY:                                              │
│  SELECT * FROM products ORDER BY price;                   │
│  → Индекс на price → данные уже отсортированы             │
│                                                            │
│  ✅ GROUP BY, DISTINCT:                                    │
│  SELECT category, COUNT(*) FROM products GROUP BY category;│
│                                                            │
│  ❌ КОГДА НЕ ПОМОГАЮТ:                                     │
│  • Малые таблицы (<1000 строк) — не нужны                │
│  • Низкая селективность (много одинаковых значений)       │
│  • SELECT * без WHERE                                     │
│  • Функции в WHERE: WHERE LOWER(email) = '...'            │
│    (нужен функциональный индекс!)                         │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ЦЕНА ИНДЕКСОВ (ПОЧЕМУ НЕ ИНДЕКСИРОВАТЬ ВСЁ):            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ⚠️ НЕДОСТАТКИ:                                            │
│                                                            │
│  1. МЕСТО НА ДИСКЕ                                        │
│     Каждый индекс = дополнительная копия данных           │
│     Индекс может быть размером с саму таблицу!            │
│                                                            │
│  2. ЗАМЕДЛЕНИЕ INSERT/UPDATE/DELETE                       │
│     При каждой записи нужно обновить ВСЕ индексы!         │
│                                                            │
│     БЕЗ ИНДЕКСОВ:                                         │
│     INSERT INTO users (...) VALUES (...);  ← 1 операция   │
│                                                            │
│     С 5 ИНДЕКСАМИ:                                        │
│     INSERT INTO users (...) VALUES (...);                 │
│     + обновить индекс по id                               │
│     + обновить индекс по email                            │
│     + обновить индекс по created_at                       │
│     + обновить индекс по name                             │
│     + обновить индекс по city                             │
│     ─────────────────────────                              │
│     = 6 операций вместо одной! ← Медленнее в 3-6 раз      │
│                                                            │
│  3. ЗАМЕДЛЕНИЕ ОБНОВЛЕНИЯ СТАТИСТИКИ                      │
│     СУБД собирает статистику для оптимизатора             │
│     Больше индексов → дольше сбор статистики              │
│                                                            │
│  ПРАВИЛО:                                                 │
│  Индексируйте ТОЛЬКО часто используемые в WHERE/JOIN      │
│  столбцы!                                                 │
│                                                            │
│  ✅ Хорошо: 2-5 индексов на таблицу                       │
│  ⚠️ Плохо: 10+ индексов на таблицу                        │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 3. Что такое VIEW в РСУБД?

text

```
┌────────────────────────────────────────────────────────────┐
│         ПРЕДСТАВЛЕНИЕ (VIEW) В РСУБД                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  View — виртуальная таблица, основанная на SQL-запросе    │
│  Данные НЕ хранятся физически, вычисляются при обращении  │
│                                                            │
│  СОЗДАНИЕ VIEW:                                           │
│  CREATE VIEW view_name AS                                 │
│      SELECT ...                                           │
│      FROM ...                                             │
│      WHERE ...;                                           │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРИМЕР:                                                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Таблицы:                                                 │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ orders                                               │ │
│  ├────┬─────────┬──────────┬───────────┬───────────────┤ │
│  │ id │ user_id │ total    │ status    │ created_at    │ │
│  ├────┼─────────┼──────────┼───────────┼───────────────┤ │
│  │ 1  │ 1001    │ 5000     │ completed │ 2024-01-10    │ │
│  │ 2  │ 1002    │ 3000     │ pending   │ 2024-01-15    │ │
│  │ 3  │ 1001    │ 7000     │ completed │ 2024-01-20    │ │
│  └────┴─────────┴──────────┴───────────┴───────────────┘ │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ users                                                │ │
│  ├────┬────────────┬──────────────────────────────────┐ │
│  │ id │ name       │ email                            │ │
│  ├────┼────────────┼──────────────────────────────────┤ │
│  │1001│ Иван       │ ivan@mail.ru                     │ │
│  │1002│ Мария      │ maria@mail.ru                    │ │
│  └────┴────────────┴──────────────────────────────────┘ │
│                                                            │
│  СОЗДАЁМ VIEW:                                            │
│  CREATE VIEW active_orders AS                             │
│  SELECT                                                   │
│      o.id,                                                │
│      u.name AS customer_name,                             │
│      o.total,                                             │
│      o.created_at                                         │
│  FROM orders o                                            │
│  JOIN users u ON o.user_id = u.id                         │
│  WHERE o.status = 'completed';                            │
│                                                            │
│  ИСПОЛЬЗОВАНИЕ VIEW (как обычная таблица):                │
│  SELECT * FROM active_orders;                             │
│                                                            │
│  Результат:                                               │
│  ┌────┬───────────────┬───────┬─────────────┐            │
│  │ id │ customer_name │ total │ created_at  │            │
│  ├────┼───────────────┼───────┼─────────────┤            │
│  │ 1  │ Иван          │ 5000  │ 2024-01-10  │            │
│  │ 3  │ Иван          │ 7000  │ 2024-01-20  │            │
│  └────┴───────────────┴───────┴─────────────┘            │
│                                                            │
│  SELECT * FROM active_orders WHERE total > 6000;          │
│  → Фильтрация работает!                                   │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ЗАЧЕМ НУЖНЫ VIEW:                                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. УПРОЩЕНИЕ СЛОЖНЫХ ЗАПРОСОВ                            │
│     Вместо:                                               │
│     SELECT o.id, u.name, ...                              │
│     FROM orders o JOIN users u ON ...                     │
│     WHERE ...                                             │
│                                                            │
│     Просто:                                               │
│     SELECT * FROM active_orders;                          │
│                                                            │
│  2. БЕЗОПАСНОСТЬ (СКРЫТИЕ СТОЛБЦОВ)                       │
│     CREATE VIEW public_users AS                           │
│     SELECT id, name, email FROM users;                    │
│     -- НЕТ password, phone                                │
│                                                            │
│     GRANT SELECT ON public_users TO analyst;              │
│     → Аналитик видит только нужные поля                   │
│                                                            │
│  3. ЛОГИКА НА УРОВНЕ БД                                   │
│     CREATE VIEW premium_users AS                          │
│     SELECT * FROM users WHERE subscription = 'premium';   │
│                                                            │
│  4. ОБРАТНАЯ СОВМЕСТИМОСТЬ                                │
│     При изменении структуры таблиц можно сохранить        │
│     старый API через VIEW                                 │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ТИПЫ VIEW:                                               │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. ОБЫЧНЫЙ VIEW (Virtual)                                │
│     Данные НЕ хранятся, вычисляются при каждом SELECT     │
│                                                            │
│  2. MATERIALIZED VIEW (Материализованное представление)   │
│     Данные ФИЗИЧЕСКИ хранятся на диске                    │
│     Обновляются вручную или по расписанию                 │
│                                                            │
│     PostgreSQL:                                           │
│     CREATE MATERIALIZED VIEW sales_summary AS             │
│     SELECT                                                │
│         product_id,                                       │
│         SUM(quantity) AS total_sold,                      │
│         SUM(amount) AS total_revenue                      │
│     FROM orders                                           │
│     GROUP BY product_id;                                  │
│                                                            │
│     REFRESH MATERIALIZED VIEW sales_summary;              │
│     → Пересчитать данные                                  │
│                                                            │
│     Зачем:                                                │
│     • Сложные агрегации (медленный запрос)                │
│     • Читаем из MATERIALIZED VIEW → быстро!               │
│     • Как кэш на уровне БД                                │
│                                                            │
│  3. UPDATABLE VIEW (Обновляемое представление)            │
│     Можно делать INSERT/UPDATE/DELETE через VIEW          │
│                                                            │
│     Условия:                                              │
│     • Один источник (одна таблица)                        │
│     • Нет агрегаций, DISTINCT, GROUP BY                   │
│                                                            │
│     CREATE VIEW active_users AS                           │
│     SELECT * FROM users WHERE active = true;              │
│                                                            │
│     UPDATE active_users SET name = 'New Name'             │
│     WHERE id = 1;                                         │
│     → Обновит базовую таблицу users!                      │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  НЕДОСТАТКИ VIEW:                                         │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ⚠️ ПРОИЗВОДИТЕЛЬНОСТЬ:                                    │
│  Обычный VIEW = запрос выполняется каждый раз             │
│  Может быть медленно для сложных JOIN/агрегаций           │
│                                                            │
│  Решение: MATERIALIZED VIEW или индексы                   │
│                                                            │
│  ⚠️ НЕ ВСЕ VIEW ОБНОВЛЯЕМЫ                                 │
│  JOIN, GROUP BY, агрегации → только чтение                │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 4. Что такое первичный ключ в РСУБД?

text

```
┌────────────────────────────────────────────────────────────┐
│         ПЕРВИЧНЫЙ КЛЮЧ (PRIMARY KEY)                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Столбец (или комбинация столбцов), УНИКАЛЬНО             │
│  идентифицирующий каждую строку в таблице                 │
│                                                            │
│  СВОЙСТВА PRIMARY KEY:                                    │
│  1. УНИКАЛЬНОСТЬ (UNIQUE) — нет дублей                    │
│  2. НЕ NULL — всегда имеет значение                       │
│  3. НЕИЗМЕНЯЕМОСТЬ — не меняется (best practice)          │
│  4. ОДИН на таблицу (только один PK)                      │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРИМЕРЫ:                                                 │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. АВТОИНКРЕМЕНТНЫЙ ID (самый частый):                   │
│  ────────────────────────────────────────                  │
│  PostgreSQL:                                              │
│  CREATE TABLE users (                                     │
│      id SERIAL PRIMARY KEY,  ← Автоинкремент              │
│      name VARCHAR(100),                                   │
│      email VARCHAR(255)                                   │
│  );                                                       │
│                                                            │
│  MySQL:                                                   │
│  CREATE TABLE users (                                     │
│      id INT AUTO_INCREMENT PRIMARY KEY,                   │
│      name VARCHAR(100),                                   │
│      email VARCHAR(255)                                   │
│  );                                                       │
│                                                            │
│  ┌────┬────────┬──────────────────┐                       │
│  │ id │ name   │ email            │                       │
│  ├────┼────────┼──────────────────┤                       │
│  │ 1  │ Иван   │ ivan@mail.ru     │ ← id = PK             │
│  │ 2  │ Мария  │ maria@mail.ru    │                       │
│  │ 3  │ Пётр   │ petr@mail.ru     │                       │
│  └────┴────────┴──────────────────┘                       │
│                                                            │
│  ⚠️ Нельзя:                                                │
│  INSERT INTO users (id, name) VALUES (1, 'Дубль');        │
│  → ERROR: duplicate key value violates unique constraint  │
│                                                            │
│  INSERT INTO users (id, name) VALUES (NULL, 'Test');      │
│  → ERROR: null value in column "id"                       │
│                                                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  2. UUID (для распределённых систем):                     │
│  ─────────────────────────────────────                     │
│  CREATE TABLE users (                                     │
│      id UUID PRIMARY KEY DEFAULT gen_random_uuid(),       │
│      name VARCHAR(100)                                    │
│  );                                                       │
│                                                            │
│  Зачем:                                                   │
│  • Нет конфликтов при слиянии баз                         │
│  • Можно генерировать на клиенте                          │
│  • Нет последовательности (безопаснее)                    │
│                                                            │
│  Минусы:                                                  │
│  • Больше места (16 байт vs 4-8 байт)                     │
│  • Медленнее индексы (не последовательны)                 │
│                                                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  3. ЕСТЕСТВЕННЫЙ КЛЮЧ (Natural Key):                      │
│  ────────────────────────────────────                      │
│  Существующее уникальное поле из реального мира           │
│                                                            │
│  CREATE TABLE countries (                                 │
│      code CHAR(2) PRIMARY KEY,  ← ISO код (RU, US, UK)    │
│      name VARCHAR(100)                                    │
│  );                                                       │
│                                                            │
│  ┌──────┬────────────┐                                    │
│  │ code │ name       │                                    │
│  ├──────┼────────────┤                                    │
│  │ RU   │ Россия     │                                    │
│  │ US   │ США        │                                    │
│  └──────┴────────────┘                                    │
│                                                            │
│  Когда использовать:                                      │
│  • Естественная уникальность (email, номер паспорта)      │
│  • Не меняется никогда                                    │
│                                                            │
│  ⚠️ Обычно НЕ рекомендуется:                               │
│  • Email может измениться → нарушится связь               │
│  • Лучше использовать суррогатный ключ (id)               │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  АВТОМАТИЧЕСКИЕ ЭФФЕКТЫ PRIMARY KEY:                      │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. СОЗДАЁТСЯ УНИКАЛЬНЫЙ ИНДЕКС                           │
│     PRIMARY KEY автоматически создаёт B-Tree индекс       │
│     → Быстрый поиск по id                                 │
│                                                            │
│  2. ИСПОЛЬЗУЕТСЯ В FOREIGN KEY                            │
│     Другие таблицы ссылаются на PK                        │
│                                                            │
│     CREATE TABLE orders (                                 │
│         id SERIAL PRIMARY KEY,                            │
│         user_id INT REFERENCES users(id)  ← FK на PK      │
│     );                                                    │
│                                                            │
│  3. КЛАСТЕРИЗАЦИЯ (в некоторых СУБД)                      │
│     MySQL InnoDB: данные физически сортируются по PK      │
│     → Быстрый доступ по диапазону PK                      │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  СОСТАВНОЙ ПЕРВИЧНЫЙ КЛЮЧ (Composite Primary Key)        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Несколько столбцов образуют уникальную комбинацию        │
│                                                            │
│  ПРИМЕР: Таблица связи "многие-ко-многим"                │
│                                                            │
│  CREATE TABLE user_roles (                                │
│      user_id INT,                                         │
│      role_id INT,                                         │
│      assigned_at TIMESTAMP DEFAULT NOW(),                 │
│      PRIMARY KEY (user_id, role_id)  ← Составной PK       │
│  );                                                       │
│                                                            │
│  ┌─────────┬─────────┬──────────────┐                     │
│  │ user_id │ role_id │ assigned_at  │                     │
│  ├─────────┼─────────┼──────────────┤                     │
│  │ 1       │ 1       │ 2024-01-01   │ ← OK                │
│  │ 1       │ 2       │ 2024-01-02   │ ← OK (разный role)  │
│  │ 2       │ 1       │ 2024-01-03   │ ← OK (разный user)  │
│  └─────────┴─────────┴──────────────┘                     │
│                                                            │
│  ❌ Нельзя:                                                │
│  INSERT INTO user_roles (user_id, role_id)                │
│  VALUES (1, 1);  ← уже есть!                              │
│  → ERROR: duplicate key                                   │
│                                                            │
│  ✅ Можно:                                                 │
│  INSERT INTO user_roles (user_id, role_id)                │
│  VALUES (1, 3);  ← новая комбинация                       │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  • Связующие таблицы (many-to-many)                       │
│  • Естественная уникальная комбинация                     │
│  • Временные ряды (device_id + timestamp)                 │
│                                                            │
│  ⚠️ НЕДОСТАТОК:                                            │
│  • Сложнее ссылаться из других таблиц                     │
│  • Часто лучше добавить суррогатный id                    │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 5. Что такое транзакция в РСУБД?

text

```
┌────────────────────────────────────────────────────────────┐
│         ТРАНЗАКЦИЯ (TRANSACTION)                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Последовательность операций БД, выполняемых как          │
│  единое целое (атомарно)                                  │
│                                                            │
│  ПРИНЦИП: "ВСЁ ИЛИ НИЧЕГО"                                │
│                                                            │
│  СИНТАКСИС:                                               │
│  BEGIN;  -- Начало транзакции                             │
│      SQL операции...                                      │
│  COMMIT; -- Зафиксировать изменения                       │
│                                                            │
│  или                                                      │
│                                                            │
│  BEGIN;                                                   │
│      SQL операции...                                      │
│  ROLLBACK; -- Отменить все изменения                      │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  КЛАССИЧЕСКИЙ ПРИМЕР: ПЕРЕВОД ДЕНЕГ                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Таблица accounts:                                        │
│  ┌────┬─────────┬─────────┐                               │
│  │ id │ owner   │ balance │                               │
│  ├────┼─────────┼─────────┤                               │
│  │ 1  │ Иван    │ 10000   │                               │
│  │ 2  │ Мария   │ 5000    │                               │
│  └────┴─────────┴─────────┘                               │
│                                                            │
│  Задача: перевести 1000₽ от Ивана к Марии                 │
│                                                            │
│  БЕЗ ТРАНЗАКЦИИ (❌ ОПАСНО!):                              │
│  ──────────────────────────────                            │
│  UPDATE accounts SET balance = balance - 1000             │
│  WHERE id = 1;  -- Списали у Ивана                        │
│                                                            │
│  ⚡ Внезапно отключили свет / сервер упал!                 │
│                                                            │
│  UPDATE accounts SET balance = balance + 1000             │
│  WHERE id = 2;  -- НЕ ВЫПОЛНИЛОСЬ!                        │
│                                                            │
│  Результат:                                               │
│  ┌────┬─────────┬─────────┐                               │
│  │ id │ owner   │ balance │                               │
│  ├────┼─────────┼─────────┤                               │
│  │ 1  │ Иван    │ 9000    │ ← Списалось                   │
│  │ 2  │ Мария   │ 5000    │ ← НЕ зачислилось!             │
│  └────┴─────────┴─────────┘                               │
│  → 1000₽ пропали! 💸                                       │
│                                                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  С ТРАНЗАКЦИЕЙ (✅ БЕЗОПАСНО):                             │
│  ──────────────────────────────                            │
│  BEGIN;                                                   │
│      UPDATE accounts SET balance = balance - 1000         │
│      WHERE id = 1;                                        │
│                                                            │
│      UPDATE accounts SET balance = balance + 1000         │
│      WHERE id = 2;                                        │
│  COMMIT;                                                  │
│                                                            │
│  Если произошла ошибка между командами:                   │
│  → Автоматический ROLLBACK                                │
│  → Обе операции отменяются                                │
│  → Данные остаются консистентными!                        │
│                                                            │
│  С явной проверкой:                                       │
│  BEGIN;                                                   │
│      UPDATE accounts SET balance = balance - 1000         │
│      WHERE id = 1 AND balance >= 1000;                    │
│                                                            │
│      -- Проверка: операция выполнена?                     │
│      GET DIAGNOSTICS rows = ROW_COUNT;                    │
│      IF rows = 0 THEN                                     │
│          ROLLBACK;  -- Недостаточно средств               │
│      ELSE                                                 │
│          UPDATE accounts SET balance = balance + 1000     │
│          WHERE id = 2;                                    │
│          COMMIT;                                          │
│      END IF;                                              │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ACID СВОЙСТВА ТРАНЗАКЦИЙ:                                │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  A — ATOMICITY (Атомарность)                              │
│  ─────────────────────────────                             │
│  Все операции выполняются или ни одна                     │
│                                                            │
│  BEGIN;                                                   │
│      INSERT INTO orders (...);                            │
│      UPDATE inventory SET stock = stock - 1;              │
│  COMMIT;                                                  │
│                                                            │
│  Если одна операция failed → откат обеих                  │
│                                                            │
│  C — CONSISTENCY (Согласованность)                        │
│  ──────────────────────────────────                        │
│  БД переходит из одного валидного состояния в другое      │
│  Все constraints соблюдаются                              │
│                                                            │
│  Пример:                                                  │
│  CHECK (balance >= 0)  ← constraint                       │
│                                                            │
│  BEGIN;                                                   │
│      UPDATE accounts SET balance = balance - 10000        │
│      WHERE id = 1;  -- balance станет отрицательным       │
│  COMMIT;                                                  │
│  → ERROR: constraint violation → ROLLBACK                 │
│                                                            │
│  I — ISOLATION (Изолированность)                          │
│  ────────────────────────────────                          │
│  Параллельные транзакции не влияют друг на друга          │
│  (подробнее см. "Уровни изоляции")                        │
│                                                            │
│  D — DURABILITY (Долговечность)                           │
│  ───────────────────────────────                           │
│  После COMMIT данные НЕ ПОТЕРЯЮТСЯ даже при сбое          │
│  Гарантируется через WAL (Write-Ahead Log)                │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  SAVEPOINT (Точки сохранения)                             │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Частичный откат внутри транзакции                        │
│                                                            │
│  BEGIN;                                                   │
│      INSERT INTO users (name) VALUES ('Alice');           │
│                                                            │
│      SAVEPOINT sp1;  ← Точка сохранения                   │
│                                                            │
│      INSERT INTO users (name) VALUES ('Bob');             │
│      INSERT INTO users (name) VALUES ('Invalid Data');    │
│      -- Ошибка!                                           │
│                                                            │
│      ROLLBACK TO sp1;  ← Откат до savepoint               │
│      -- Alice осталась, Bob отменён                       │
│                                                            │
│      INSERT INTO users (name) VALUES ('Charlie');         │
│  COMMIT;                                                  │
│                                                            │
│  Результат: Alice + Charlie сохранены                     │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

```

│         СОРТИРОВКА ДАННЫХ (ORDER BY)                       │

│  → O(N) — просто читаем по порядку                        │
│  → Быстро! ⚡                                              │
│                                                            │
│  EXPLAIN ANALYZE (посмотреть план):                       │
│  EXPLAIN ANALYZE                                          │

│                                                            │

│                                                            │
└────────────────────────────────────────────────────────────┘
```

## ЧАСТЬ 2: Продвинутые концепции

### 1. Почему нельзя проиндексировать все столбцы?

text

```
┌────────────────────────────────────────────────────────────┐
│         ПОЧЕМУ НЕ ИНДЕКСИРОВАТЬ ВСЁ?                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ИДЕЯ: "Создадим индексы на все столбцы → всё будет быстро"│
│                                                            │
│  РЕАЛЬНОСТЬ: ❌ КАТАСТРОФА                                 │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРОБЛЕМА 1: МЕСТО НА ДИСКЕ                               │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Каждый индекс = копия данных столбца + указатели         │
│                                                            │
│  Таблица users (10 столбцов, 1 млн строк):                │
│  • Размер таблицы: 500 MB                                 │
│                                                            │
│  1 индекс ≈ 50-100 MB (в зависимости от типа данных)      │
│                                                            │
│  10 индексов = 500-1000 MB                                │
│  → Индексы занимают СТОЛЬКО ЖЕ места как данные!          │
│  → Итого: 500 MB (данные) + 1000 MB (индексы) = 1.5 GB    │
│                                                            │
│  ⚠️ В 3 раза больше места!                                 │
│                                                            │
│  100 таблиц × 10 индексов = ОГРОМНЫЕ расходы на хранение  │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРОБЛЕМА 2: ЗАМЕДЛЕНИЕ ЗАПИСИ                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  При INSERT/UPDATE/DELETE нужно обновить ВСЕ индексы!     │
│                                                            │
│  БЕЗ ИНДЕКСОВ:                                            │
│  INSERT INTO users (name, email, age, city, ...)          │
│  VALUES ('Ivan', 'ivan@mail.ru', 30, 'Moscow', ...);      │
│  → 1 операция записи на диск                              │
│  → Время: 1 ms                                            │
│                                                            │
│  С 10 ИНДЕКСАМИ:                                          │
│  INSERT INTO users (...) VALUES (...);                    │
│  → Записать данные в таблицу          (1 ms)              │
│  → Обновить индекс по id              (0.5 ms)            │
│  → Обновить индекс по email           (0.5 ms)            │
│  → Обновить индекс по age             (0.5 ms)            │
│  → Обновить индекс по city            (0.5 ms)            │
│  → ...еще 6 индексов...               (3 ms)              │
│  ────────────────────────────────────────────              │
│  = 6-7 ms вместо 1 ms                                     │
│  → В 6-7 раз МЕДЛЕННЕЕ! 🐌                                │
│                                                            │
│  Для таблицы с 1000 INSERT/сек:                           │
│  • Без индексов: справляется легко                        │
│  • С 10 индексами: может не успевать → очередь растёт     │
│                                                            │
│  UPDATE ещё хуже:                                         │
│  UPDATE users SET email = 'new@mail.ru' WHERE id = 1;     │
│  → Нужно удалить старое значение из индекса email         │
│  → Вставить новое значение                                │
│  → Для КАЖДОГО индекса, затрагивающего изменённые поля    │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРОБЛЕМА 3: ФРАГМЕНТАЦИЯ ИНДЕКСОВ                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  B-Tree индексы со временем фрагментируются               │
│  (пустые страницы, неоптимальная структура)               │
│                                                            │
│  10 индексов → 10 структур, которые нужно:                │
│  • Перестраивать (REINDEX)                                │
│  • Анализировать (ANALYZE)                                │
│  • Мониторить                                             │
│                                                            │
│  PostgreSQL:                                              │
│  REINDEX TABLE users;  ← долго для больших таблиц!        │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРОБЛЕМА 4: КЭШИ И ПАМЯТЬ                                │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  БД кэширует индексы в RAM (Buffer Pool)                  │
│                                                            │
│  10 индексов × 100 MB каждый = 1 GB в памяти              │
│  → Меньше места для кэширования данных                    │
│  → Больше disk I/O                                        │
│  → Медленнее                                              │
│                                                            │
│  Optimal: 2-5 индексов на таблицу                         │
│  → Помещаются в RAM                                       │
│  → Быстрые запросы                                        │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРОБЛЕМА 5: QUERY OPTIMIZER ПЕРЕГРУЖЕН                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Планировщик выбирает лучший индекс для запроса           │
│                                                            │
│  2-3 индекса:                                             │
│  → Быстро выбирает оптимальный план                       │
│                                                            │
│  10+ индексов:                                            │
│  → Перебирает все комбинации                              │
│  → Дольше планирование запроса                            │
│  → Может выбрать неоптимальный индекс!                    │
│                                                            │
│  EXPLAIN показывает:                                      │
│  Planning time: 0.5 ms   ← с 2 индексами                  │
│  Planning time: 5 ms     ← с 20 индексами                 │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРОБЛЕМА 6: НИЗКАЯ СЕЛЕКТИВНОСТЬ                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Не все столбцы стоит индексировать!                      │
│                                                            │
│  Плохой пример:                                           │
│  CREATE INDEX idx_gender ON users(gender);                │
│                                                            │
│  gender: ['male', 'female'] — только 2 значения           │
│  → Низкая селективность                                   │
│  → Индекс почти бесполезен                                │
│                                                            │
│  SELECT * FROM users WHERE gender = 'male';               │
│  → Вернёт 50% таблицы                                     │
│  → Full Table Scan может быть БЫСТРЕЕ чем индекс!         │
│                                                            │
│  ПРАВИЛО:                                                 │
│  Индекс полезен если возвращает <15% строк                │
│                                                            │
│  Хороший индекс:                                          │
│  CREATE INDEX idx_email ON users(email);                  │
│  → email уникальны → высокая селективность                │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  КОГДА ИНДЕКСИРОВАТЬ:                                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ✅ ИНДЕКСИРОВАТЬ:                                         │
│  • PRIMARY KEY (автоматически)                            │
│  • FOREIGN KEY (для JOIN)                                 │
│  • Столбцы в WHERE часто используемые                     │
│  • Столбцы в ORDER BY                                     │
│  • Столбцы с высокой селективностью                       │
│                                                            │
│  ❌ НЕ ИНДЕКСИРОВАТЬ:                                      │
│  • Малые таблицы (<1000 строк)                            │
│  • Столбцы с низкой селективностью (boolean, enum)        │
│  • Столбцы, которые редко используются в WHERE            │
│  • Столбцы, которые часто обновляются                     │
│  • Временные таблицы                                      │
│                                                            │
│  АНАЛИЗ ИСПОЛЬЗОВАНИЯ:                                    │
│  PostgreSQL:                                              │
│  SELECT * FROM pg_stat_user_indexes                       │
│  WHERE idx_scan = 0;  ← Неиспользуемые индексы!           │
│                                                            │
│  → Удалить их!                                            │
│  DROP INDEX idx_unused;                                   │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 2. Что такое составной первичный ключ в РСУБД?

text

```
┌────────────────────────────────────────────────────────────┐
│         СОСТАВНОЙ ПЕРВИЧНЫЙ КЛЮЧ (Composite Primary Key)  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Первичный ключ, состоящий из НЕСКОЛЬКИХ столбцов,        │
│  которые вместе образуют уникальную комбинацию            │
│                                                            │
│  СИНТАКСИС:                                               │
│  CREATE TABLE table_name (                                │
│      column1 TYPE,                                        │
│      column2 TYPE,                                        │
│      column3 TYPE,                                        │
│      PRIMARY KEY (column1, column2)  ← Составной PK       │
│  );                                                       │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРИМЕР 1: Many-to-Many связь                             │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Задача: Студенты записываются на курсы                   │
│  • Один студент → много курсов                            │
│  • Один курс → много студентов                            │
│                                                            │
│  CREATE TABLE student_courses (                           │
│      student_id INT,                                      │
│      course_id INT,                                       │
│      enrolled_at TIMESTAMP DEFAULT NOW(),                 │
│      grade DECIMAL(3,2),                                  │
│      PRIMARY KEY (student_id, course_id),  ← Составной PK │
│      FOREIGN KEY (student_id) REFERENCES students(id),    │
│      FOREIGN KEY (course_id) REFERENCES courses(id)       │
│  );                                                       │
│                                                            │
│  Данные:                                                  │
│  ┌────────────┬───────────┬──────────────┬───────┐        │
│  │ student_id │ course_id │ enrolled_at  │ grade │        │
│  ├────────────┼───────────┼──────────────┼───────┤        │
│  │ 1          │ 101       │ 2024-01-01   │ 4.5   │ ✅ OK  │
│  │ 1          │ 102       │ 2024-01-05   │ 3.8   │ ✅ OK  │
│  │ 2          │ 101       │ 2024-01-02   │ 5.0   │ ✅ OK  │
│  │ 2          │ 102       │ 2024-01-06   │ 4.2   │ ✅ OK  │
│  └────────────┴───────────┴──────────────┴───────┘        │
│                                                            │
│  ✅ МОЖНО:                                                 │
│  INSERT INTO student_courses (student_id, course_id)      │
│  VALUES (1, 103);  ← новая комбинация                     │
│                                                            │
│  INSERT INTO student_courses (student_id, course_id)      │
│  VALUES (3, 101);  ← новая комбинация                     │
│                                                            │
│  ❌ НЕЛЬЗЯ (нарушение PK):                                 │
│  INSERT INTO student_courses (student_id, course_id)      │
│  VALUES (1, 101);  ← уже существует!                      │
│  → ERROR: duplicate key value violates unique constraint  │
│          "student_courses_pkey"                           │
│                                                            │
│  ЛОГИКА:                                                  │
│  • Студент 1 может быть на многих курсах                  │
│  • Курс 101 может иметь многих студентов                  │
│  • НО комбинация (student_id=1, course_id=101) уникальна! │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРИМЕР 2: Временные ряды (Time-series)                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Задача: Метрики с датчиков IoT                           │
│                                                            │
│  CREATE TABLE sensor_data (                               │
│      device_id VARCHAR(50),                               │
│      timestamp TIMESTAMP,                                 │
│      temperature DECIMAL(5,2),                            │
│      humidity DECIMAL(5,2),                               │
│      PRIMARY KEY (device_id, timestamp)  ← Составной PK   │
│  );                                                       │
│                                                            │
│  Данные:                                                  │
│  ┌───────────┬─────────────────────┬─────────┬──────────┐ │
│  │ device_id │ timestamp           │ temp    │ humidity │ │
│  ├───────────┼─────────────────────┼─────────┼──────────┤ │
│  │ sensor001 │ 2024-01-15 10:00:00 │ 22.5    │ 45.0     │ │
│  │ sensor001 │ 2024-01-15 10:01:00 │ 22.7    │ 45.2     │ │
│  │ sensor002 │ 2024-01-15 10:00:00 │ 18.3    │ 52.1     │ │
│  │ sensor002 │ 2024-01-15 10:01:00 │ 18.5    │ 51.8     │ │
│  └───────────┴─────────────────────┴─────────┴──────────┘ │
│                                                            │
│  ЛОГИКА:                                                  │
│  • Один датчик может иметь много измерений (разное время) │
│  • Одна временная метка может быть у разных датчиков      │
│  • НО (device_id, timestamp) уникальна!                   │
│                                                            │
│  ЗАПРОСЫ:                                                 │
│  -- Данные с конкретного датчика                          │
│  SELECT * FROM sensor_data                                │
│  WHERE device_id = 'sensor001'                            │
│  ORDER BY timestamp DESC                                  │
│  LIMIT 10;                                                │
│                                                            │
│  -- Средняя температура по всем датчикам за час           │
│  SELECT                                                   │
│      DATE_TRUNC('hour', timestamp) AS hour,               │
│      AVG(temperature) AS avg_temp                         │
│  FROM sensor_data                                         │
│  WHERE timestamp >= NOW() - INTERVAL '24 hours'           │
│  GROUP BY hour                                            │
│  ORDER BY hour;                                           │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРИМЕР 3: Корзина покупок                                │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  CREATE TABLE cart_items (                                │
│      user_id INT,                                         │
│      product_id INT,                                      │
│      quantity INT DEFAULT 1,                              │
│      added_at TIMESTAMP DEFAULT NOW(),                    │
│      PRIMARY KEY (user_id, product_id),                   │
│      FOREIGN KEY (user_id) REFERENCES users(id),          │
│      FOREIGN KEY (product_id) REFERENCES products(id)     │
│  );                                                       │
│                                                            │
│  Данные:                                                  │
│  ┌─────────┬────────────┬──────────┬─────────────┐        │
│  │ user_id │ product_id │ quantity │ added_at    │        │
│  ├─────────┼────────────┼──────────┼─────────────┤        │
│  │ 1001    │ 42         │ 2        │ 2024-01-15  │        │
│  │ 1001    │ 57         │ 1        │ 2024-01-16  │        │
│  │ 1002    │ 42         │ 3        │ 2024-01-15  │        │
│  └─────────┴────────────┴──────────┴─────────────┘        │
│                                                            │
│  ЛОГИКА:                                                  │
│  • Пользователь может иметь много товаров в корзине       │
│  • Товар может быть в корзинах многих пользователей       │
│  • НО один пользователь НЕ может дважды добавить          │
│    один товар (вместо этого увеличиваем quantity)         │
│                                                            │
│  ОПЕРАЦИИ:                                                │
│  -- Добавить товар в корзину (если нет)                   │
│  INSERT INTO cart_items (user_id, product_id, quantity)   │
│  VALUES (1001, 99, 1)                                     │
│  ON CONFLICT (user_id, product_id) DO UPDATE              │
│  SET quantity = cart_items.quantity + 1;                  │
│                                                            │
│  ↑ Если комбинация существует → увеличить quantity        │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ИНДЕКСЫ НА СОСТАВНОМ PRIMARY KEY                         │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  PRIMARY KEY (col1, col2) автоматически создаёт индекс    │
│  на (col1, col2)                                          │
│                                                            │
│  ✅ БЫСТРЫЕ ЗАПРОСЫ:                                       │
│  WHERE col1 = ... AND col2 = ...    ← использует PK index │
│  WHERE col1 = ...                   ← использует PK index │
│                                                            │
│  ⚠️ МЕДЛЕННЫЙ ЗАПРОС:                                      │
│  WHERE col2 = ...                   ← НЕ использует индекс│
│                                       (col2 не первый!)   │
│                                                            │
│  РЕШЕНИЕ:                                                 │
│  CREATE INDEX idx_col2 ON table_name(col2);               │
│                                                            │
│  Пример:                                                  │
│  PRIMARY KEY (student_id, course_id)                      │
│                                                            │
│  -- Быстро (использует PK индекс)                         │
│  SELECT * FROM student_courses                            │
│  WHERE student_id = 1;                                    │
│                                                            │
│  -- Медленно (не использует PK индекс)                    │
│  SELECT * FROM student_courses                            │
│  WHERE course_id = 101;                                   │
│                                                            │
│  -- Добавляем индекс                                      │
│  CREATE INDEX idx_course ON student_courses(course_id);   │
│  -- Теперь быстро!                                        │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  АЛЬТЕРНАТИВА: СУРРОГАТНЫЙ КЛЮЧ                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Вместо составного PK можно добавить отдельный id         │
│                                                            │
│  CREATE TABLE student_courses (                           │
│      id SERIAL PRIMARY KEY,          ← Суррогатный ключ   │
│      student_id INT,                                      │
│      course_id INT,                                       │
│      enrolled_at TIMESTAMP,                               │
│      grade DECIMAL(3,2),                                  │
│      UNIQUE (student_id, course_id), ← Уникальность       │
│      FOREIGN KEY (student_id) REFERENCES students(id),    │
│      FOREIGN KEY (course_id) REFERENCES courses(id)       │
│  );                                                       │
│                                                            │
│  ПРЕИМУЩЕСТВА СУРРОГАТНОГО КЛЮЧА:                         │
│  ✅ Проще ссылаться из других таблиц                       │
│  ✅ Короче (INT вместо двух столбцов)                      │
│  ✅ Не зависит от бизнес-логики                            │
│                                                            │
│  НЕДОСТАТКИ:                                              │
│  ❌ Дополнительный столбец                                 │
│  ❌ Нужен дополнительный UNIQUE constraint                 │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  • Составной PK: для связующих таблиц (many-to-many)      │
│  • Суррогатный ключ: для основных таблиц                  │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ОГРАНИЧЕНИЯ СОСТАВНОГО PRIMARY KEY                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ⚠️ СЛОЖНОСТЬ FOREIGN KEY:                                 │
│                                                            │
│  Если другая таблица ссылается на составной PK:           │
│                                                            │
│  CREATE TABLE enrollments (                               │
│      enrollment_id SERIAL PRIMARY KEY,                    │
│      student_id INT,                                      │
│      course_id INT,                                       │
│      semester VARCHAR(20),                                │
│      FOREIGN KEY (student_id, course_id)                  │
│          REFERENCES student_courses(student_id, course_id)│
│  );                                                       │
│                                                            │
│  → Нужно указывать ОБА столбца в FK                       │
│  → Громоздко                                              │
│                                                            │
│  ⚠️ ORDER OF COLUMNS ВАЖЕН:                                │
│                                                            │
│  PRIMARY KEY (col1, col2) ≠ PRIMARY KEY (col2, col1)      │
│                                                            │
│  Порядок влияет на эффективность индекса!                 │
│  Первый столбец должен быть наиболее селективный          │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 3. Что такое хранимая процедура?

```
┌────────────────────────────────────────────────────────────┐
│         ХРАНИМАЯ ПРОЦЕДУРА (Stored Procedure)              │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Набор SQL-команд и логики (IF, LOOP, переменные),        │
│  сохранённый в БД и выполняемый как единое целое          │
│                                                            │
│  СИНТАКСИС (PostgreSQL):                                  │
│  CREATE OR REPLACE PROCEDURE procedure_name(params)       │
│  LANGUAGE plpgsql                                         │
│  AS $$                                                    │
│  BEGIN                                                    │
│      -- SQL команды                                       │
│      -- Условия, циклы                                    │
│  END;                                                     │
│  $$;                                                      │
│                                                            │
│  ВЫЗОВ:                                                   │
│  CALL procedure_name(arguments);                          │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРИМЕР 1: Перевод денег между счетами                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  CREATE OR REPLACE PROCEDURE transfer_money(              │
│      sender_id INT,                                       │
│      recipient_id INT,                                    │
│      amount DECIMAL(10,2)                                 │
│  )                                                        │
│  LANGUAGE plpgsql                                         │
│  AS $$                                                    │
│  DECLARE                                                  │
│      sender_balance DECIMAL(10,2);                        │
│  BEGIN                                                    │
│      -- Проверка баланса отправителя                      │
│      SELECT balance INTO sender_balance                   │
│      FROM accounts                                        │
│      WHERE id = sender_id;                                │
│                                                            │
│      IF sender_balance < amount THEN                      │
│          RAISE EXCEPTION 'Insufficient funds';            │
│      END IF;                                              │
│                                                            │
│      -- Списать со счёта отправителя                      │
│      UPDATE accounts                                      │
│      SET balance = balance - amount                       │
│      WHERE id = sender_id;                                │
│                                                            │
│      -- Зачислить на счёт получателя                      │
│      UPDATE accounts                                      │
│      SET balance = balance + amount                       │
│      WHERE id = recipient_id;                             │
│                                                            │
│      -- Записать в лог транзакций                         │
│      INSERT INTO transaction_log (from_id, to_id, amount) │
│      VALUES (sender_id, recipient_id, amount);            │
│                                                            │
│      COMMIT;                                              │
│  END;                                                     │
│  $$;                                                      │
│                                                            │
│  ИСПОЛЬЗОВАНИЕ:                                           │
│  CALL transfer_money(1, 2, 1000.00);                      │
│  → Переведёт 1000₽ со счёта 1 на счёт 2                   │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРИМЕР 2: Очистка старых данных                          │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  CREATE OR REPLACE PROCEDURE cleanup_old_logs(            │
│      days_to_keep INT DEFAULT 30                          │
│  )                                                        │
│  LANGUAGE plpgsql                                         │
│  AS $$                                                    │
│  DECLARE                                                  │
│      deleted_count INT;                                   │
│  BEGIN                                                    │
│      DELETE FROM logs                                     │
│      WHERE created_at < NOW() - INTERVAL '1 day' * days_to_keep;│
│                                                            │
│      GET DIAGNOSTICS deleted_count = ROW_COUNT;           │
│                                                            │
│      RAISE NOTICE 'Deleted % old log entries', deleted_count;│
│  END;                                                     │
│  $$;                                                      │
│                                                            │
│  ИСПОЛЬЗОВАНИЕ:                                           │
│  CALL cleanup_old_logs(90);  -- Удалить логи старше 90 дней│
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ФУНКЦИИ vs ПРОЦЕДУРЫ                                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ФУНКЦИЯ (FUNCTION):                                      │
│  • Возвращает значение (RETURN)                           │
│  • Используется в SELECT                                  │
│  • Не может делать COMMIT/ROLLBACK                        │
│                                                            │
│  CREATE FUNCTION get_user_balance(user_id INT)            │
│  RETURNS DECIMAL(10,2)                                    │
│  AS $$                                                    │
│  BEGIN                                                    │
│      RETURN (SELECT balance FROM accounts WHERE id = user_id);│
│  END;                                                     │
│  $$ LANGUAGE plpgsql;                                     │
│                                                            │
│  SELECT get_user_balance(1);  ← Вызов в SELECT            │
│                                                            │
│  ПРОЦЕДУРА (PROCEDURE):                                   │
│  • НЕ возвращает значение (или через OUT параметры)       │
│  • Вызов через CALL                                       │
│  • Может делать COMMIT/ROLLBACK                           │
│  • Для сложной бизнес-логики                              │
│                                                            │
│  CALL transfer_money(1, 2, 1000);  ← Вызов через CALL     │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  ПРЕИМУЩЕСТВА ХРАНИМЫХ ПРОЦЕДУР                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. ПРОИЗВОДИТЕЛЬНОСТЬ                                    │
│  ────────────────────                                      │
│  • Компилируется один раз, выполняется много раз          │
│  • Меньше сетевого трафика (один CALL вместо 10 запросов) │
│  • Данные обрабатываются на сервере БД (ближе к данным)   │
│                                                            │
│  БЕЗ ПРОЦЕДУРЫ (из приложения):                           │
│  ┌──────────┐                      ┌──────────┐           │
│  │   App    │ ─SELECT balance─────▶│    DB    │           │
│  │          │ ◀────1000────────────│          │           │
│  │ if bal>  │                      │          │           │
│  │  amount  │ ─UPDATE sender───────▶│          │           │
│  │          │ ─UPDATE recipient────▶│          │           │
│  │          │ ─INSERT log──────────▶│          │           │
│  └──────────┘                      └──────────┘           │
│  → 4 запроса туда-обратно                                 │
│                                                            │
│  С ПРОЦЕДУРОЙ:                                            │
│  ┌──────────┐                      ┌──────────┐           │
│  │   App    │ ─CALL transfer(...)─▶│    DB    │           │
│  │          │ ◀────OK/ERROR────────│  Вся     │           │
│  └──────────┘                      │  логика  │           │
│                                    │  внутри! │           │
│                                    └──────────┘           │
│  → 1 запрос                                               │
│                                                            │
│  2. ИНКАПСУЛЯЦИЯ БИЗНЕС-ЛОГИКИ                            │
│  ──────────────────────────────                            │
│  • Логика в одном месте (DRY)                             │
│  • Изменения в процедуре → не нужно менять приложение     │
│  • Меньше дублирования кода                               │
│                                                            │
│  3. БЕЗОПАСНОСТЬ                                          │
│  ───────────────                                           │
│  • Пользователь может вызвать процедуру, но НЕ имеет      │
│    прямого доступа к таблицам                             │
│                                                            │
│  GRANT EXECUTE ON PROCEDURE transfer_money TO app_user;   │
│  REVOKE ALL ON accounts FROM app_user;                    │
│  → Приложение может переводить деньги, но не может        │
│    напрямую изменить balance                              │
│                                                            │
│  4. АТОМАРНОСТЬ                                           │
│  ──────────────                                            │
│  • Вся логика в одной транзакции                          │
│  • COMMIT/ROLLBACK внутри процедуры                       │
│                                                            │
│  5. ПЕРЕИСПОЛЬЗОВАНИЕ                                     │
│  ────────────────────                                      │
│  • Один раз написать → использовать из разных приложений  │
│  • Python, Java, PHP — все вызывают одну процедуру        │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  НЕДОСТАТКИ ХРАНИМЫХ ПРОЦЕДУР                             │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. VENDOR LOCK-IN (привязка к СУБД)                      │
│  ─────────────────────────────────                         │
│  • Синтаксис разный в PostgreSQL, MySQL, Oracle, SQL Server│
│  • Миграция на другую СУБД = переписать процедуры         │
│                                                            │
│  PostgreSQL: LANGUAGE plpgsql                             │
│  MySQL: LANGUAGE SQL / BEGIN...END                        │
│  Oracle: PL/SQL (DECLARE, BEGIN, END)                     │
│  SQL Server: T-SQL                                        │
│                                                            │
│  2. СЛОЖНОСТЬ ТЕСТИРОВАНИЯ                                │
│  ─────────────────────────────                             │
│  • Юнит-тесты сложнее писать для процедур                 │
│  • Нужны специальные инструменты (pgTAP, utPLSQL)         │
│  • Логика размазана между приложением и БД                │
│                                                            │
│  3. ВЕРСИОНИРОВАНИЕ                                       │
│  ──────────────────                                        │
│  • Процедуры хранятся В БД, не в Git (по умолчанию)       │
│  • Нужны миграции для изменения процедур                  │
│  • Сложнее code review                                    │
│                                                            │
│  4. ОТЛАДКА                                               │
│  ──────────                                                │
│  • Сложнее дебажить чем код приложения                    │
│  • Ограниченные инструменты отладки                       │
│  • RAISE NOTICE вместо нормального логирования            │
│                                                            │
│  5. НАГРУЗКА НА СУБД                                      │
│  ───────────────                                           │
│  • Сложная логика в процедурах → нагрузка на БД           │
│  • БД не всегда лучшее место для бизнес-логики            │
│  • Горизонтальное масштабирование приложений проще,       │
│    чем БД                                                 │
│                                                            │
│  6. ОГРАНИЧЕННАЯ ФУНКЦИОНАЛЬНОСТЬ                         │
│  ────────────────────────────────                          │
│  • Нельзя вызвать внешний API                             │
│  • Нельзя отправить email                                 │
│  • Нельзя работать с файловой системой                    │
│  • Ограниченная работа со строками, JSON                  │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  КОГДА ИСПОЛЬЗОВАТЬ ХРАНИМЫЕ ПРОЦЕДУРЫ?                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ✅ ИСПОЛЬЗУЙТЕ:                                           │
│  • Сложные транзакции (перевод денег, резервирование)     │
│  • Массовые операции с данными (ETL, cleanup)             │
│  • Критичная производительность (много запросов)          │
│  • Общая логика для разных приложений                     │
│  • Дополнительная безопасность (ограничение доступа)      │
│  • Legacy системы (уже используют процедуры)              │
│                                                            │
│  ❌ НЕ ИСПОЛЬЗУЙТЕ:                                        │
│  • Простая CRUD логика                                    │
│  • Когда возможна миграция на другую СУБД                 │
│  • Микросервисы (логика должна быть в сервисах)           │
│  • Если команда не знает SQL на продвинутом уровне        │
│  • Когда нужна гибкость и быстрые изменения               │
│                                                            │
│  СОВРЕМЕННЫЙ ПОДХОД:                                      │
│  • Минимум логики в БД                                    │
│  • Бизнес-логика в приложении (легче тестировать)         │
│  • Процедуры только для критичных операций                │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### INNER JOIN (ВНУТРЕННЕЕ СОЕДИНЕНИЕ)

text

```
┌────────────────────────────────────────────────────────────┐
│         INNER JOIN                                         │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Возвращает только строки, для которых ЕСТЬ совпадение    │
│  в ОБЕИХ таблицах                                         │
│                                                            │
│  ДИАГРАММА ВЕННА:                                         │
│        ┌─────────┐                                        │
│        │ users   │                                        │
│        │    ┌────┼────┐                                   │
│        │    │ ██ │    │ orders                            │
│        │    │ ██ │    │                                   │
│        └────┼────┘    │                                   │
│             └─────────┘                                   │
│                                                            │
│  ЗАПРОС:                                                  │
│  SELECT u.name, o.total, o.status                         │
│  FROM users u                                             │
│  INNER JOIN orders o ON u.id = o.user_id;                 │
│                                                            │
│  или короткая форма (INNER по умолчанию):                 │
│  SELECT u.name, o.total, o.status                         │
│  FROM users u                                             │
│  JOIN orders o ON u.id = o.user_id;                       │
│                                                            │
│  РЕЗУЛЬТАТ:                                               │
│  ┌────────┬───────┬───────────┐                           │
│  │ name   │ total │ status    │                           │
│  ├────────┼───────┼───────────┤                           │
│  │ Иван   │ 5000  │ completed │ ← user_id=1, order_id=1   │
│  │ Иван   │ 3000  │ pending   │ ← user_id=1, order_id=2   │
│  │ Мария  │ 7000  │ completed │ ← user_id=2, order_id=3   │
│  └────────┴───────┴───────────┘                           │
│                                                            │
│  ❌ НЕ ВОШЛИ:                                              │
│  • Пётр (нет заказов)                                     │
│  • Анна (нет заказов)                                     │
│  • Заказ id=4 (user_id=NULL)                              │
│                                                            │
│  ЛОГИКА: WHERE u.id = o.user_id AND o.user_id IS NOT NULL │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### LEFT JOIN (ЛЕВОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ)

text

```
┌────────────────────────────────────────────────────────────┐
│         LEFT JOIN (LEFT OUTER JOIN)                        │
├────────────────────────────────────────────────────────────┤

│        ┌─────────┐                                        │

│        │ ██┌────┼────┐                                   │
│        │ ██│ ██ │    │ orders                            │
│        │ ██│ ██ │    │                                   │

│  РЕЗУЛЬТАТ:                                               │
│  ┌────────┬───────┬───────────┐                           │

│  ├────────┼───────┼───────────┤                           │
│  │ Иван   │ 5000  │ completed │                           │
│  │ Иван   │ 3000  │ pending   │                           │
│  │ Мария  │ 7000  │ completed │                           │
│  │ Пётр   │ NULL  │ NULL      │ ← НЕТ заказов             │
│  │ Анна   │ NULL  │ NULL      │ ← НЕТ заказов             │
│  └────────┴───────┴───────────┘                           │
│                                                            │
│  ✅ ВСЕ пользователи включены                              │
│  ❌ Заказ id=4 (user_id=NULL) НЕ включён                   │
│                                                            │
│  ПРИМЕНЕНИЕ: Найти пользователей БЕЗ заказов              │

│  FROM users u                                             │

│  │ Анна   │                                               │
```

#### RIGHT JOIN (ПРАВОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ)

text

```
┌────────────────────────────────────────────────────────────┐
│         RIGHT JOIN (RIGHT OUTER JOIN)                      │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Возвращает ВСЕ строки из ПРАВОЙ таблицы + совпадения     │
│  из левой (если нет совпадения → NULL)                    │
│                                                            │
│  ДИАГРАММА ВЕННА:                                         │
│        ┌─────────┐                                        │
│        │ users   │                                        │
│        │    ┌────┼────┐                                   │
│        │    │ ██ │ ██ │                                   │
│        │    │ ██ │ ██ │                                   │
│        └────┼────┘ ██ │                                   │
│             └─────────┘                                   │
│                                                            │
│  ЗАПРОС:                                                  │
│  SELECT u.name, o.total, o.status                         │
│  FROM users u                                             │
│  RIGHT JOIN orders o ON u.id = o.user_id;                 │
│                                                            │
│  РЕЗУЛЬТАТ:                                               │
│  ┌────────┬───────┬───────────┐                           │
│  │ name   │ total │ status    │                           │
│  ├────────┼───────┼───────────┤                           │
│  │ Иван   │ 5000  │ completed │                           │
│  │ Иван   │ 3000  │ pending   │                           │
│  │ Мария  │ 7000  │ completed │                           │
│  │ NULL   │ 1000  │ cancelled │ ← Заказ без пользователя  │
│  └────────┴───────┴───────────┘                           │
│                                                            │
│  ✅ ВСЕ заказы включены                                    │
│  ❌ Пётр и Анна (нет заказов) НЕ включены                  │
│                                                            │
│  ⚠️ RIGHT JOIN используется редко                          │
│  Обычно переписывают как LEFT JOIN:                       │
│                                                            │
│  SELECT u.name, o.total, o.status                         │
│  FROM orders o                                            │
│  LEFT JOIN users u ON o.user_id = u.id;                   │
│  ↑ Эквивалентно RIGHT JOIN выше                           │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### FULL OUTER JOIN (ПОЛНОЕ ВНЕШНЕЕ СОЕДИНЕНИЕ)

text

```
┌────────────────────────────────────────────────────────────┐
│         FULL OUTER JOIN                                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Возвращает ВСЕ строки из ОБЕИХ таблиц                    │
│  Если нет совпадения → NULL с соответствующей стороны     │
│                                                            │
│  ДИАГРАММА ВЕННА:                                         │
│        ┌─────────┐                                        │
│        │ ██████  │                                        │
│        │ ██┌────┼────┐                                   │
│        │ ██│ ██ │ ██ │                                   │
│        │ ██│ ██ │ ██ │                                   │
│        └───┼────┘ ██ │                                   │
│            └─────────┘                                   │
│                                                            │
│  ЗАПРОС:                                                  │
│  SELECT u.name, o.total, o.status                         │
│  FROM users u                                             │
│  FULL OUTER JOIN orders o ON u.id = o.user_id;            │
│                                                            │
│  РЕЗУЛЬТАТ:                                               │
│  ┌────────┬───────┬───────────┐                           │
│  │ name   │ total │ status    │                           │
│  ├────────┼───────┼───────────┤                           │
│  │ Иван   │ 5000  │ completed │                           │
│  │ Иван   │ 3000  │ pending   │                           │
│  │ Мария  │ 7000  │ completed │                           │
│  │ Пётр   │ NULL  │ NULL      │ ← Нет заказов             │
│  │ Анна   │ NULL  │ NULL      │ ← Нет заказов             │
│  │ NULL   │ 1000  │ cancelled │ ← Нет пользователя        │
│  └────────┴───────┴───────────┘                           │
│                                                            │
│  ✅ ВСЕ пользователи + ВСЕ заказы                          │
│                                                            │
│  ПРИМЕНЕНИЕ:                                              │
│  Найти несоответствия (orphaned records)                  │
│                                                            │
│  SELECT u.name, o.total                                   │
│  FROM users u                                             │
│  FULL OUTER JOIN orders o ON u.id = o.user_id             │
│  WHERE u.id IS NULL OR o.id IS NULL;                      │
│                                                            │
│  → Найдёт:                                                │
│  • Пользователей без заказов (Пётр, Анна)                 │
│  • Заказы без пользователей (order id=4)                  │
│                                                            │
│  ⚠️ MySQL НЕ ПОДДЕРЖИВАЕТ FULL OUTER JOIN!                 │
│  Эмуляция через UNION:                                    │
│  SELECT * FROM users u LEFT JOIN orders o ON ...          │
│  UNION                                                    │
│  SELECT * FROM users u RIGHT JOIN orders o ON ...         │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### 1.1 B-TREE INDEX

text

```
┌────────────────────────────────────────────────────────────┐
│         B-TREE INDEX (По умолчанию)                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  СТРУКТУРА: Сбалансированное дерево                       │
│                                                            │
│                      [50]                                 │
│                    /      \                               │
│                [25]        [75]                           │
│               /   \        /   \                          │
│           [10]  [35]   [60]  [90]                         │
│          /  \    / \    / \    / \                        │
│        [...][...][...][...][...][...]                     │
│          ↓    ↓   ↓   ↓   ↓   ↓                          │
│     Указатели на строки таблицы                           │
│                                                            │
│  СВОЙСТВА:                                                │
│  • Высота дерева O(log N)                                 │
│  • Данные отсортированы                                   │
│  • Эффективен для диапазонов                              │
│                                                            │
│  СОЗДАНИЕ:                                                │
│  CREATE INDEX idx_name ON users(age);                     │
│  -- По умолчанию B-Tree                                   │
│                                                            │
│  или явно:                                                │
│  CREATE INDEX idx_name ON users USING BTREE (age);        │
│                                                            │
│  ✅ ПОДХОДИТ ДЛЯ:                                          │
│  • Точное совпадение: WHERE age = 30                      │
│  • Диапазоны: WHERE age BETWEEN 20 AND 40                 │
│  • Сортировка: ORDER BY age                               │
│  • Сравнения: WHERE age > 25, age <= 50                   │
│  • Префиксный поиск: WHERE name LIKE 'Ива%'               │
│  • MIN/MAX: SELECT MIN(age), MAX(age)                     │
│                                                            │
│  ❌ НЕ ПОДХОДИТ ДЛЯ:                                       │
│  • WHERE name LIKE '%иван%'  (поиск в середине)           │
│  • Битовые операции                                       │
│  • Full-text search                                       │
│                                                            │
│  ПРИМЕР:                                                  │
│  CREATE TABLE products (                                  │
│      id SERIAL PRIMARY KEY,  ← B-Tree автоматически       │
│      name VARCHAR(255),                                   │
│      price DECIMAL(10,2),                                 │
│      created_at TIMESTAMP                                 │
│  );                                                       │
│                                                            │
│  CREATE INDEX idx_price ON products(price);               │
│  CREATE INDEX idx_created ON products(created_at);        │
│                                                            │
│  -- Использует индекс                                     │
│  SELECT * FROM products                                   │
│  WHERE price BETWEEN 1000 AND 5000                        │
│  ORDER BY price;                                          │
│                                                            │
│  СОСТАВНОЙ B-TREE ИНДЕКС:                                 │
│  CREATE INDEX idx_category_price                          │
│  ON products(category_id, price);                         │
│                                                            │
│  ✅ Эффективен для:                                        │
│  WHERE category_id = 1 AND price > 1000                   │
│  WHERE category_id = 1  (только первый столбец)           │
│                                                            │
│  ❌ НЕ эффективен для:                                     │
│  WHERE price > 1000  (второй столбец без первого)         │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### 1.2 HASH INDEX

```
├────────────────────────────────────────────────────────────┤
```

#### 1.3 GIN INDEX (Generalized Inverted Index)

text

```
┌────────────────────────────────────────────────────────────┐
│         GIN INDEX (PostgreSQL)                             │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ: Инвертированный индекс для составных типов  │
│                                                            │
│  СТРУКТУРА:                                               │
│  ┌──────────────┬─────────────────────────────────────┐   │
│  │ Элемент      │ Строки, содержащие элемент          │   │
│  ├──────────────┼─────────────────────────────────────┤   │
│  │ "golang"     │ [doc1, doc3, doc7]                  │   │
│  │ "python"     │ [doc2, doc3, doc5]                  │   │
│  │ "javascript" │ [doc1, doc4, doc6]                  │   │
│  └──────────────┴─────────────────────────────────────┘   │
│                                                            │
│  СОЗДАНИЕ:                                                │
│  CREATE INDEX idx_tags ON articles USING GIN (tags);      │
│                                                            │
│  ✅ ПОДХОДИТ ДЛЯ:                                          │
│                                                            │
│  1. МАССИВЫ:                                              │
│  ────────────                                              │
│  CREATE TABLE articles (                                  │
│      id SERIAL PRIMARY KEY,                               │
│      title TEXT,                                          │
│      tags TEXT[]  ← Массив                                │
│  );                                                       │
│                                                            │
│  CREATE INDEX idx_tags ON articles USING GIN (tags);      │
│                                                            │
│  -- Поиск статей с тегом "postgresql"                     │
│  SELECT * FROM articles                                   │
│  WHERE tags @> ARRAY['postgresql'];                       │
│                                                            │
│  -- Поиск статей с любым из тегов                         │
│  SELECT * FROM articles                                   │
│  WHERE tags && ARRAY['postgresql', 'python'];             │
│                                                            │
│  2. JSONB:                                                │
│  ─────────                                                 │
│  CREATE TABLE products (                                  │
│      id SERIAL PRIMARY KEY,                               │
│      data JSONB                                           │
│  );                                                       │
│                                                            │
│  CREATE INDEX idx_data ON products USING GIN (data);      │
│                                                            │
│  INSERT INTO products (data) VALUES                       │
│  ('{"name": "Laptop", "tags": ["electronics", "new"]}');  │
│                                                            │
│  -- Поиск по JSON-полю                                    │
│  SELECT * FROM products                                   │
│  WHERE data @> '{"tags": ["electronics"]}';               │
│                                                            │
│  SELECT * FROM products                                   │
│  WHERE data->>'name' = 'Laptop';                          │
│                                                            │
│  3. FULL-TEXT SEARCH:                                     │
│  ──────────────────────                                    │
│  CREATE INDEX idx_content ON articles                     │
│  USING GIN (to_tsvector('russian', content));            │
│                                                            │
│  SELECT * FROM articles                                   │
│  WHERE to_tsvector('russian', content) @@                 │
│        to_tsquery('russian', 'PostgreSQL & индекс');      │
│                                                            │
│  4. HSTORE (key-value внутри PostgreSQL):                 │
│  ─────────────────────────────────────────                 │
│  CREATE EXTENSION hstore;                                 │
│                                                            │
│  CREATE TABLE settings (                                  │
│      user_id INT,                                         │
│      prefs HSTORE                                         │
│  );                                                       │
│                                                            │
│  CREATE INDEX idx_prefs ON settings USING GIN (prefs);    │
│                                                            │
│  INSERT INTO settings VALUES                              │
│  (1, 'theme=>dark, lang=>ru, notifications=>true');       │
│                                                            │
│  SELECT * FROM settings                                   │
│  WHERE prefs @> 'theme=>dark';                            │
│                                                            │
│  ⚠️ НЕДОСТАТКИ:                                            │
│  • Больше места на диске (чем B-Tree)                     │
│  • Медленнее INSERT/UPDATE (нужно обновить много элементов)│
│  • Только для специфичных типов данных                    │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### 1.5 BRIN INDEX (Block Range Index)

text

```
┌────────────────────────────────────────────────────────────┐
│         BRIN INDEX (PostgreSQL 9.5+)                       │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ: Индекс диапазонов блоков                    │
│  Очень компактный (в 100-1000 раз меньше B-Tree!)         │
│                                                            │
│  ПРИНЦИП РАБОТЫ:                                          │
│  Таблица разбита на блоки (по 8KB обычно)                 │
│  Индекс хранит MIN/MAX значения для каждого блока         │
│                                                            │
│  ┌────────────────────────────────────────────┐            │
│  │ Block 1: [2024-01-01 ... 2024-01-05]      │            │
│  │ Block 2: [2024-01-06 ... 2024-01-10]      │            │
│  │ Block 3: [2024-01-11 ... 2024-01-15]      │            │
│  └────────────────────────────────────────────┘            │
│                                                            │
│  WHERE created_at > '2024-01-07'                          │
│  → Пропустить Block 1, сканировать Block 2, 3             │
│                                                            │
│  СОЗДАНИЕ:                                                │
│  CREATE INDEX idx_created ON logs                         │
│  USING BRIN (created_at);                                 │
│                                                            │
│  ✅ ПОДХОДИТ ДЛЯ:                                          │
│  • Очень большие таблицы (миллиарды строк)                │
│  • Данные физически отсортированы (или почти)             │
│  • Time-series данные (логи, метрики, IoT)                │
│  • Append-only таблицы (только INSERT)                    │
│                                                            │
│  ПРИМЕР:                                                  │
│  CREATE TABLE logs (                                      │
│      id BIGSERIAL,                                        │
│      created_at TIMESTAMP DEFAULT NOW(),                  │
│      level VARCHAR(10),                                   │
│      message TEXT                                         │
│  );                                                       │
│                                                            │
│  CREATE INDEX idx_created ON logs                         │
│  USING BRIN (created_at) WITH (pages_per_range = 128);    │
│                                                            │
│  SELECT * FROM logs                                       │
│  WHERE created_at BETWEEN '2024-01-01' AND '2024-01-31';  │
│                                                            │
│  ПРЕИМУЩЕСТВА:                                            │
│  • Крошечный размер (1 MB индекс на 1 TB данных!)         │
│  • Быстрый INSERT (почти нет overhead)                    │
│  • Низкое потребление памяти                              │
│                                                            │
│  НЕДОСТАТКИ:                                              │
│  • Медленнее поиск чем B-Tree                             │
│  • Требует физической сортировки данных                   │
│  • Не подходит для случайных данных                       │
│                                                            │
│  СРАВНЕНИЕ (таблица 1 млрд строк, логи за год):           │
│  ┌─────────┬─────────┬─────────────────────────┐          │
│  │  Индекс │ Размер  │ Скорость INSERT         │          │
│  ├─────────┼─────────┼─────────────────────────┤          │
│  │ B-Tree  │ 10 GB   │ Средняя                 │          │
│  │ BRIN    │ 10 MB   │ Очень быстрая ⚡         │          │
│  └─────────┴─────────┴─────────────────────────┘          │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

```
│         BITMAP INDEX (Oracle, PostgreSQL в запросах)       │

│                                                            │

│                                                            │

│  CREATE TABLE users (                                     │

│  );                                                       │
│                                                            │

│  CREATE BITMAP INDEX idx_country ON users(country);       │
│  CREATE BITMAP INDEX idx_status ON users(status);         │
│                                                            │
│  SELECT * FROM users                                      │

│                                                            │

│                                                            │
│  POSTGRESQL:                                              │
│  Нет прямой поддержки, но использует Bitmap Index Scan    │

│                                                            │

│  • Медленные UPDATE/INSERT/DELETE (нужно обновить биты)   │
```

```
│         COVERING INDEX (Index-Only Scan)                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ: Индекс содержит ВСЕ столбцы из SELECT       │
│  → Не нужно обращаться к таблице!                         │
│                                                            │

│  CREATE INDEX idx_covering ON users(email)                │

│                                                            │

│                                                            │

│  CREATE INDEX idx_email ON users(email);                  │
│                                                            │
│  SELECT name, email FROM users                            │
│  WHERE email = 'ivan@mail.ru';                            │
│                                                            │
│  План выполнения:                                         │
│  1. Index Scan по idx_email                               │
│  2. Heap Fetch (читаем строку из таблицы для name)        │

│                                                            │

│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  • Быстрее в 2-10 раз (нет heap fetch)                    │
│  • Меньше disk I/O                                        │
│                                                            │
│  ⚠️ НЕДОСТАТКИ:                                            │
│  • Больше размер индекса                                  │
```

#### Детальное объяснение CAP

text

```
┌────────────────────────────────────────────────────────────┐
│  C — CONSISTENCY (Согласованность)                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Все узлы видят ОДИНАКОВЫЕ данные ОДНОВРЕМЕННО            │
│                                                            │
│  Пример:                                                  │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐             │
│  │ Node 1  │     │ Node 2  │     │ Node 3  │             │
│  │ bal=100 │ ═══ │ bal=100 │ ═══ │ bal=100 │             │
│  └─────────┘     └─────────┘     └─────────┘             │
│                                                            │
│  Клиент пишет на Node 1: balance = 50                     │
│                                                            │
│  С CONSISTENCY:                                           │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐             │
│  │ Node 1  │     │ Node 2  │     │ Node 3  │             │
│  │ bal=50  │ ──▶ │ bal=50  │ ──▶ │ bal=50  │             │
│  └─────────┘     └─────────┘     └─────────┘             │
│                                                            │
│  → Запись блокируется пока ВСЕ узлы не синхронизируются   │
│  → Все клиенты видят bal=50 ОДНОВРЕМЕННО                  │
│                                                            │
│  БЕЗ CONSISTENCY (Eventual Consistency):                  │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐             │
│  │ Node 1  │     │ Node 2  │     │ Node 3  │             │
│  │ bal=50  │     │ bal=100 │     │ bal=100 │ ← Старое!   │
│  └─────────┘     └─────────┘     └─────────┘             │
│                                                            │
│  → Клиент может прочитать старое значение (100)           │
│  → Через время (миллисекунды-секунды) все станут bal=50   │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  A — AVAILABILITY (Доступность)                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Система ВСЕГДА отвечает на запросы (даже при сбоях)      │
│                                                            │
│  ┌─────────┐     ┌─────────┐     ┌─────────┐             │
│  │ Node 1  │ ✅  │ Node 2  │ ❌  │ Node 3  │ ✅          │
│  │ работает│     │  упал   │     │ работает│             │
│  └─────────┘     └─────────┘     └─────────┘             │
│                                                            │
│  С AVAILABILITY:                                          │
│  Клиент обращается к Node 2 → перенаправление на Node 1   │
│  → Клиент ВСЕГДА получает ответ                           │
│  → Может быть неактуальный, но ответ есть!                │
│                                                            │
│  БЕЗ AVAILABILITY:                                        │
│  Клиент обращается к Node 2                               │
│  → ERROR: Service Unavailable                             │
│  → Нужно ждать восстановления Node 2                      │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  P — PARTITION TOLERANCE (Устойчивость к разделению)      │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Система работает при разрыве сети между узлами           │
│                                                            │
│  ┌─────────┐     ╳╳╳╳╳╳╳     ┌─────────┐                 │
│  │ Node 1  │ ←──┼─────┼───▶  │ Node 2  │                 │
│  │ Москва  │    ╳ нет ╳      │ Лондон  │                 │
│  │         │    ╳связи╳      │         │                 │
│  └─────────┘     ╳╳╳╳╳╳╳     └─────────┘                 │
│                                                            │
│  Network partition (разделение сети):                     │
│  • Обрыв кабеля                                           │
│  • DDoS атака                                             │
│  • Проблемы маршрутизации                                 │
│  • Датацентры в разных странах                            │
│                                                            │
│  С PARTITION TOLERANCE:                                   │
│  • Node 1 продолжает работать                             │
│  • Node 2 продолжает работать                             │
│  • Данные рассинхронизированы!                            │
│  • При восстановлении связи → конфликт разрешается        │
│                                                            │
│  БЕЗ PARTITION TOLERANCE:                                 │
│  • Вся система падает при разрыве связи                   │
│  • На практике НЕВОЗМОЖНО в распределённых системах!       │
│                                                            │
│  ⚠️ В реальности P НЕИЗБЕЖНО (сеть ненадёжна)             │
│  → Выбор между CP и AP                                    │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

```
┌────────────────────────────────────────────────────────────┐

│  ────────────────────────────────────────                  │

├────────────────────────────────────────────────────────────┤

│                                                            │
│  → Node 1 и Node 2 НЕ МОГУТ синхронизироваться            │
│  → Чтобы гарантировать consistency:                       │

│                                                            │

│  • Zookeeper                                              │
│  • Etcd                                                   │
│  • Consul                                                 │
│                                                            │
│  ПРИМЕНЕНИЕ:                                              │
│  • Финансовые системы (консистентность критична)          │
│  • Конфигурация кластеров                                 │
│  • Системы координации (лидер-выборы)                     │
│                                                            │

┌────────────────────────────────────────────────────────────┐
│  AP (Availability + Partition Tolerance)                  │

│  Жертвуем CONSISTENCY (согласованностью)                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ПОВЕДЕНИЕ ПРИ NETWORK PARTITION:                         │
│  ┌─────────┐     ╳╳╳╳╳     ┌─────────┐                   │
│  │ Node 1  │ ←───────────▶  │ Node 2  │                   │

│  └─────────┘                └─────────┘                   │
│                                                            │
│  Клиент A пишет на Node 1: balance = 50                   │
│  Клиент B пишет на Node 2: balance = 200                  │
│                                                            │
│  ┌─────────┐     ╳╳╳╳╳     ┌─────────┐                   │
│  │ Node 1  │                │ Node 2  │                   │
│  │ bal=50  │                │ bal=200 │  ← КОНФЛИКТ!      │
│  └─────────┘                └─────────┘                   │
│                                                            │
│  → ОБА узла отвечают OK!                                  │
│  → Данные рассинхронизированы                             │
│  → При восстановлении связи:                              │
│    - Автоматическое разрешение (last-write-wins)          │
│    - Или вектор версий (version vectors)                  │
│    - Или конфликт отдаётся клиенту                        │

│                                                            │
│  ПРИМЕРЫ:                                                 │
│  • Cassandra                                              │
│  • DynamoDB                                               │
│  • Riak                                                   │
│  • CouchDB                                                │
│  • Voldemort                                              │
│                                                            │
│  ПРИМЕНЕНИЕ:                                              │
│  • Социальные сети (лайки, подписки)                      │
│  • Корзины покупок                                        │
│  • DNS                                                    │
│  • CDN                                                    │
│  • Системы с географическим распределением                │
│                                                            │
│  ПРИНЦИП:                                                 │
│  "Лучше дать немного неточный ответ, чем не ответить"     │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  CA (Consistency + Availability)                          │
│  ────────────────────────────────────                      │
│  БЕЗ Partition Tolerance                                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ТЕОРЕТИЧЕСКИ:                                            │

│                                                            │
│  НА ПРАКТИКЕ:                                             │
│  ⚠️ НЕВОЗМОЖНО в распределённых системах!                  │

│  • PostgreSQL с синхронной репликацией                    │
│    (но при разрыве связи → теряем A или C!)               │
│                                                            │
│  ВЫВОД:                                                   │
│  В распределённых системах выбор ТОЛЬКО между CP и AP!    │
```

```
│                                                            │
│  Oracle тоже поддерживает стандартный синтаксис:          │
│  SELECT u.name, o.total                                   │
│  FROM users u                                             │
│  LEFT JOIN orders o ON u.id = o.user_id;  ✅              │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  5. NULL и пустые строки                                  │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Oracle:                                                  │
│  ───────                                                   │
│  • Пустая строка '' = NULL ⚠️                             │
│  • VARCHAR2('') сохраняется как NULL                      │
│                                                            │
│  INSERT INTO users (name) VALUES ('');                    │
│  SELECT * FROM users WHERE name IS NULL;  ← Найдёт!       │
│                                                            │
│  PostgreSQL:                                              │
│  ────────────                                              │
│  • Пустая строка '' ≠ NULL                                │
│  • VARCHAR('') сохраняется как ''                         │
│                                                            │
│  INSERT INTO users (name) VALUES ('');                    │
│  SELECT * FROM users WHERE name IS NULL;  ← НЕ найдёт!    │
│  SELECT * FROM users WHERE name = '';     ← Найдёт!       │
│                                                            │
│  ⚠️ КРИТИЧЕСКАЯ РАЗНИЦА!                                   │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  6. ПАГИНАЦИЯ                                             │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Oracle (до 12c):                                         │
│  ─────────────────                                         │
│  SELECT * FROM (                                          │
│      SELECT a.*, ROWNUM rnum FROM (                       │
│          SELECT * FROM users ORDER BY id                  │
│      ) a WHERE ROWNUM <= 20  -- 2 страница × 10           │
│  ) WHERE rnum > 10;           -- пропустить 10            │
│                                                            │
│  Oracle 12c+:                                             │
│  ─────────────                                             │
│  SELECT * FROM users                                      │
│  ORDER BY id                                              │
│  OFFSET 10 ROWS FETCH NEXT 10 ROWS ONLY;                  │
│                                                            │
│  PostgreSQL / MySQL:                                      │
│  ────────────────────                                      │
│  SELECT * FROM users                                      │
│  ORDER BY id                                              │
│  LIMIT 10 OFFSET 10;  ← Намного проще!                    │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  7. ХРАНИМЫЕ ПРОЦЕДУРЫ                                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Oracle (PL/SQL):                                         │
│  ─────────────────                                         │
│  CREATE OR REPLACE PROCEDURE transfer_money(              │
│      p_from IN NUMBER,                                    │
│      p_to IN NUMBER,                                      │
│      p_amount IN NUMBER                                   │
│  ) IS                                                     │
│      v_balance NUMBER;                                    │
│  BEGIN                                                    │
│      SELECT balance INTO v_balance                        │
│      FROM accounts WHERE id = p_from;                     │
│                                                            │
│      IF v_balance < p_amount THEN                         │
│          RAISE_APPLICATION_ERROR(-20001,                  │
│              'Insufficient funds');                       │
│      END IF;                                              │
│                                                            │
│      UPDATE accounts SET balance = balance - p_amount     │
│      WHERE id = p_from;                                   │
│                                                            │
│      UPDATE accounts SET balance = balance + p_amount     │
│      WHERE id = p_to;                                     │
│                                                            │
│      COMMIT;                                              │
│  END;                                                     │
│  /                                                        │
│                                                            │
│  EXEC transfer_money(1, 2, 1000);                         │
│                                                            │
│  PostgreSQL (PL/pgSQL):                                   │
│  ───────────────────────                                   │
│  CREATE OR REPLACE PROCEDURE transfer_money(              │
│      p_from INT,                                          │
│      p_to INT,                                            │
│      p_amount DECIMAL                                     │
│  )                                                        │
│  LANGUAGE plpgsql                                         │
│  AS $$                                                    │
│  DECLARE                                                  │
│      v_balance DECIMAL;                                   │
│  BEGIN                                                    │
│      SELECT balance INTO v_balance                        │
│      FROM accounts WHERE id = p_from;                     │
│                                                            │
│      IF v_balance < p_amount THEN                         │
│          RAISE EXCEPTION 'Insufficient funds';            │
│      END IF;                                              │
│                                                            │
│      UPDATE accounts SET balance = balance - p_amount     │
│      WHERE id = p_from;                                   │
│                                                            │
│      UPDATE accounts SET balance = balance + p_amount     │
│      WHERE id = p_to;                                     │
│                                                            │
│      COMMIT;                                              │
│  END;                                                     │
│  $$;                                                      │
│                                                            │
│  CALL transfer_money(1, 2, 1000);                         │
│                                                            │
│  РАЗЛИЧИЯ:                                                │
│  • IS vs AS $$                                            │
│  • RAISE_APPLICATION_ERROR vs RAISE EXCEPTION             │
│  • IN параметры vs без IN                                 │
│  • / vs $$                                                │
│  • EXEC vs CALL                                           │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  8. ИЕРАРХИЧЕСКИЕ ЗАПРОСЫ                                 │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  Oracle (CONNECT BY):                                     │
│  ─────────────────────                                     │
│  SELECT id, name, manager_id, LEVEL                       │
│  FROM employees                                           │
│  START WITH manager_id IS NULL                            │
│  CONNECT BY PRIOR id = manager_id;                        │
│                                                            │
│  PostgreSQL (WITH RECURSIVE):                             │
│  ─────────────────────────────                             │
│  WITH RECURSIVE emp_hierarchy AS (                        │
│      -- Якорь: топ-уровень                                │
│      SELECT id, name, manager_id, 1 AS level              │
│      FROM employees                                       │
│      WHERE manager_id IS NULL                             │
│                                                            │
│      UNION ALL                                            │
│                                                            │
│      -- Рекурсия: подчинённые                             │
│      SELECT e.id, e.name, e.manager_id, h.level + 1       │
│      FROM employees e                                     │
│      JOIN emp_hierarchy h ON e.manager_id = h.id          │
│  )                                                        │
│  SELECT * FROM emp_hierarchy;                             │
│                                                            │
│  ⚠️ Совершенно разный синтаксис!                           │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### Проблемы параллельного доступа

text

```
┌────────────────────────────────────────────────────────────┐
│  ПРОБЛЕМЫ, КОТОРЫЕ РЕШАЮТ УРОВНИ ИЗОЛЯЦИИ                 │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  1. DIRTY READ (Грязное чтение)                           │
│  ───────────────────────────────                           │
│  Чтение НЕЗАФИКСИРОВАННЫХ данных другой транзакции        │
│                                                            │
│  Время  │ Транзакция A              │ Транзакция B        │
│  ───────┼───────────────────────────┼─────────────────────│
│  t1     │ BEGIN;                    │                     │
│  t2     │ UPDATE accounts           │                     │
│         │ SET balance = 500         │                     │
│         │ WHERE id = 1;             │                     │
│  t3     │                           │ BEGIN;              │
│  t4     │                           │ SELECT balance      │
│         │                           │ FROM accounts       │
│         │                           │ WHERE id = 1;       │
│         │                           │ → 500 (грязное!)    │
│  t5     │ ROLLBACK;                 │                     │
│         │ (отменили!)               │                     │

│         │                           │ -- но реально 1000! │
│         │                           │ COMMIT;             │
│                                                            │

│                                                            │
├────────────────────────────────────────────────────────────┤
│                                                            │

│  t2     │ SELECT balance            │                     │
│         │ FROM accounts             │                     │
│         │ WHERE id = 1;             │                     │
│         │ → 1000                    │                     │
│  t3     │                           │ BEGIN;              │
│  t4     │                           │ UPDATE accounts     │

│         │                           │ WHERE id = 1;       │

│  t7     │ COMMIT;                   │                     │

│                                                            │

│                                                            │
│  3. PHANTOM READ (Фантомное чтение)                       │
│  ───────────────────────────────                           │
│  Появились НОВЫЕ строки между чтениями                    │

│                                                            │
```
```
│  ПРОБЛЕМЫ:                                                │

│  │ Non-Repeatable   │ ❌ ВОЗМОЖНО                     │   │
│  │ Phantom Read     │ ❌ ВОЗМОЖНО                     │   │
│  └──────────────────┴─────────────────────────────────┘   │
│                                                            │
│  ИСПОЛЬЗОВАНИЕ:                                           │
│  SET TRANSACTION ISOLATION LEVEL READ COMMITTED;          │
│  -- Или не указывать (по умолчанию в PostgreSQL, Oracle)  │
│                                                            │

│  SELECT balance FROM accounts WHERE id = 1;               │

│                                                            │

│  T1: SELECT balance FROM accounts WHERE id = 1; → 500     │

│  T1: COMMIT;                                              │
│                                                            │
│  КОГДА ПРИМЕНЯТЬ:                                         │
│  ✅ Большинство web-приложений                             │
│  ✅ Стандартный выбор для OLTP                             │
│  ✅ Баланс между производительностью и консистентностью    │
│                                                            │
│  ПО УМОЛЧАНИЮ:                                            │
│  • PostgreSQL ✅                                           │
│  • Oracle ✅                                               │
│  • SQL Server ✅                                           │
│  • MySQL InnoDB — REPEATABLE READ (!)                     │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  УРОВЕНЬ 3: REPEATABLE READ                               │
│  (Повторяемое чтение)                                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПИСАНИЕ:                                                │
│  Транзакция видит "снимок" данных на момент её начала     │
│  Повторное чтение ОДНИХ И ТЕХ ЖЕ строк даёт тот же рез-т │
│                                                            │
│  ПРОБЛЕМЫ:                                                │
│  ┌──────────────────┬─────────────────────────────────┐   │
│  │ Dirty Read       │ ✅ НЕВОЗМОЖНО                   │   │
│  │ Non-Repeatable   │ ✅ НЕВОЗМОЖНО                   │   │
│  │ Phantom Read     │ ⚠️ ЗАВИСИТ ОТ СУБД              │   │
│  │                  │ PostgreSQL: НЕТ                 │   │
│  │                  │ MySQL: ДА                       │   │
│  └──────────────────┴─────────────────────────────────┘   │
│                                                            │
│  ИСПОЛЬЗОВАНИЕ:                                           │
│  SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;         │
│                                                            │
│  BEGIN;                                                   │
│  SELECT balance FROM accounts WHERE id = 1; → 1000        │
│  -- Другая транзакция меняет balance на 500 и COMMIT      │
│  SELECT balance FROM accounts WHERE id = 1; → 1000 (!)    │
│  -- Видим СТАРОЕ значение (snapshot)                      │
│  COMMIT;                                                  │
│                                                            │
│  МЕХАНИЗМ (PostgreSQL):                                   │
│  • MVCC (Multi-Version Concurrency Control)               │
│  • Каждая транзакция видит "свой" снимок данных           │
│  • Старые версии строк хранятся до завершения транзакций  │
│                                                            │
│  ПРИМЕР ФАНТОМНОГО ЧТЕНИЯ (MySQL):                        │
│  T1: BEGIN;                                               │
│  T1: SELECT COUNT(*) FROM orders WHERE user_id = 1; → 5   │
│  T2: INSERT INTO orders (user_id) VALUES (1);             │
│  T2: COMMIT;                                              │
│  T1: SELECT COUNT(*) FROM orders WHERE user_id = 1; → 5   │
│      ← НЕ видит новую строку! (MySQL REPEATABLE READ)     │
│                                                            │
│  PostgreSQL:                                              │
│  • Фантомов тоже НЕТ (сильнее MySQL!)                     │
│  • Snapshot Isolation (близко к SERIALIZABLE)             │
│                                                            │
│  КОГДА ПРИМЕНЯТЬ:                                         │
│  ✅ Финансовые отчёты (нужна консистентность)              │
│  ✅ Расчёты, где важна согласованность данных              │
│  ✅ Миграции данных                                        │
│                                                            │
│  ПО УМОЛЧАНИЮ:                                            │
│  • MySQL InnoDB ✅ (по умолчанию)                          │
│  • PostgreSQL ❌ (нужно указывать)                         │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  УРОВЕНЬ 4: SERIALIZABLE                                  │
│  (Сериализуемость) — САМЫЙ СТРОГИЙ                        │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПИСАНИЕ:                                                │
│  Транзакции выполняются так, КАК БУДТО они были           │
│  выполнены ПОСЛЕДОВАТЕЛЬНО (одна за другой)               │
│                                                            │
│  ПРОБЛЕМЫ:                                                │
│  ┌──────────────────┬─────────────────────────────────┐   │
│  │ Dirty Read       │ ✅ НЕВОЗМОЖНО                   │   │
│  │ Non-Repeatable   │ ✅ НЕВОЗМОЖНО                   │   │
│  │ Phantom Read     │ ✅ НЕВОЗМОЖНО                   │   │
│  └──────────────────┴─────────────────────────────────┘   │
│                                                            │
│  ИСПОЛЬЗОВАНИЕ:                                           │
│  SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;            │
│                                                            │
│  BEGIN;                                                   │
│  SELECT * FROM accounts WHERE balance > 1000;             │
│  UPDATE accounts SET balance = balance - 100;             │
│  COMMIT;                                                  │
│                                                            │
│  МЕХАНИЗМ:                                                │
│                                                            │
│  PostgreSQL (SSI - Serializable Snapshot Isolation):      │
│  • Snapshot Isolation + проверка конфликтов               │
│  • Если конфликт → одна транзакция откатывается           │
│                                                            │
│  Пример конфликта:                                        │
│  T1: BEGIN;                                               │
│  T2: BEGIN;                                               │
│  T1: SELECT SUM(balance) FROM accounts; → 10000           │
│  T2: SELECT SUM(balance) FROM accounts; → 10000           │
│  T1: INSERT INTO accounts (balance) VALUES (500);         │
│  T2: INSERT INTO accounts (balance) VALUES (300);         │
│  T1: COMMIT; ✅                                            │
│  T2: COMMIT; ❌ ERROR: could not serialize access         │
│                                                            │
│  → PostgreSQL обнаружил конфликт и откатил T2             │
│                                                            │
│  MySQL (Lock-based):                                      │
│  • Блокировки диапазонов (range locks)                    │
│  • Медленнее, больше блокировок                           │
│                                                            │
│  КОГДА ПРИМЕНЯТЬ:                                         │
│  ✅ Критичная консистентность (банки)                      │
│  ✅ Сложные транзакции с read-modify-write                 │
│  ✅ Когда нужна 100% гарантия изоляции                     │
│                                                            │
│  ⚠️ НЕДОСТАТКИ:                                            │
│  • Медленнее всех уровней                                 │
│  • Много конфликтов → много откатов                       │
│  • Низкий concurrency (параллелизм)                       │
│  • Требует retry логики в приложении                      │
│                                                            │
│  ЦЕНА:                                                    │
│  • Производительность: -30% до -70%                       │
│  • Throughput (пропускная способность): -50%              │
│                                                            │
└────────────────────────────────────────────────────────────┘
```
