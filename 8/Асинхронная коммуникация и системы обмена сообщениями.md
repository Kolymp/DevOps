## 1. Асинхронная коммуникация

### Определение

Асинхронная коммуникация — это модель взаимодействия между компонентами системы, при которой **отправитель отправляет сообщение и НЕ ждёт немедленного ответа**. Отправитель продолжает свою работу, а получатель обрабатывает сообщение **когда будет готов** — это может произойти через миллисекунды, секунды или даже часы.

Ключевая идея: **разделение во времени** — отправитель и получатель не обязаны быть активными одновременно.

text

```
══════════════════ СИНХРОННАЯ КОММУНИКАЦИЯ ══════════════════

Клиент                              Сервер
  │                                    │
  │──── HTTP запрос ──────────────────▶│
  │          ⏳                        │ обрабатывает...
  │     клиент ждёт                    │ обрабатывает...
  │     и заблокирован                 │ обрабатывает...
  │◀──── HTTP ответ ──────────────────│
  │                                    │
  ▼ только теперь продолжает работу    ▼


══════════════════ АСИНХРОННАЯ КОММУНИКАЦИЯ ══════════════════

Клиент                Посредник              Сервер
  │                      │                      │
  │── сообщение ────────▶│                      │
  │                      │  хранит в очереди    │
  │ продолжает работу ✅ │                      │
  │ не заблокирован      │                      │
  │                      │── доставляет ───────▶│
  │ делает другие дела   │                      │ обрабатывает
  │                      │                      │ когда готов
  ▼                      ▼                      ▼
```

### Где встречается асинхронная коммуникация

text

```
1. МЕЖДУ СЕРВИСАМИ (микросервисы)
   Order Service ──событие "заказ создан"──▶ [Брокер] ──▶ Payment Service
                                                     ──▶ Notification Service
                                                     ──▶ Inventory Service

2. ВНУТРИ ПРИЛОЖЕНИЯ (асинхронные задачи)
   Web Controller ──задача──▶ [Очередь] ──▶ Background Worker
   (отвечает клиенту сразу)                 (генерирует PDF-отчёт)

3. МЕЖДУ СИСТЕМАМИ (интеграция)
   CRM ──данные клиента──▶ [Брокер] ──▶ Billing System
                                    ──▶ Analytics System

4. IoT
   10,000 датчиков ──телеметрия──▶ [Брокер] ──▶ Processing Pipeline
```

## 2. Очередь сообщений (Message Queue)

### Определение

Очередь сообщений — это **структура данных типа FIFO** (First In, First Out), которая выступает **промежуточным буфером** между отправителем (Producer) и получателем (Consumer). Она временно хранит сообщения, обеспечивая **надёжную передачу данных** даже при различной скорости работы отправителя и получателя.

text

```
                              ОЧЕРЕДЬ СООБЩЕНИЙ

                    ┌─────────────────────────────────┐
Producer ──msg──▶   │ MSG5 │ MSG4 │ MSG3 │ MSG2 │ MSG1│ ──msg──▶ Consumer
                    └─────────────────────────────────┘
                     вход ▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶▶ выход
                              First In → First Out

Жизненный цикл сообщения:
  1. Producer создаёт сообщение
  2. Сообщение попадает в очередь (enqueue)
  3. Сообщение хранится до момента обработки
  4. Consumer забирает сообщение (dequeue)
  5. Consumer отправляет подтверждение (ACK)
  6. Очередь удаляет сообщение
```

### Аналогия

text

```
📮 Почтовый ящик в подъезде:

  • Почтальон (Producer) кладёт письма в ящик
  • Ящик (Queue) хранит письма
  • Жилец (Consumer) забирает письма, когда приходит домой
  • Почтальону НЕ нужно ждать жильца
  • Жильцу НЕ нужно ждать почтальона
  • Письма не теряются, даже если жильца нет дома
```

### Типы очередей и паттернов (подробно)

#### 1. Point-to-Point (P2P) — Один к одному

text

```
                        ┌──────────┐
    Producer ──msg──▶   │  QUEUE   │ ──msg──▶ Consumer A  ✅ получил
                        └──────────┘          Consumer B  ❌ не получил
                                              Consumer C  ❌ не получил

    ПРАВИЛО: Каждое сообщение обрабатывается РОВНО ОДНИМ потребителем
```

**Как это работает:**

- Продюсер отправляет сообщение в очередь
- Если к очереди подключены несколько consumers — сообщение получит **только один**
- Брокер **балансирует** нагрузку между consumers (round-robin или другой алгоритм)
- После обработки consumer отправляет **ACK** — сообщение удаляется из очереди

**Реальные примеры:**

text

```
• Распределение задач между воркерами:
  [Resize Image Queue] ──▶ Worker 1 (обрабатывает фото #1)
                       ──▶ Worker 2 (обрабатывает фото #2)
                       ──▶ Worker 3 (обрабатывает фото #3)

• Обработка заказов:
  [Order Queue] ──▶ Order Processor 1
                ──▶ Order Processor 2

• Отправка email:
  [Email Queue] ──▶ Email Sender 1
                ──▶ Email Sender 2
```

#### 2. Publish/Subscribe (Pub/Sub) — Один ко многим

text

```
                         ┌───────────┐
                    ┌───▶│Subscriber A│  ✅ получил копию
                    │    └───────────┘
   ┌───────┐       │    ┌───────────┐
   │ TOPIC │───────┼───▶│Subscriber B│  ✅ получил копию
   └───────┘       │    └───────────┘
       ▲           │    ┌───────────┐
       │           └───▶│Subscriber C│  ✅ получил копию
   Publisher             └───────────┘

   ПРАВИЛО: Каждое сообщение доставляется ВСЕМ подписчикам
```

**Как это работает:**

- Publisher отправляет сообщение в **топик** (topic)
- Все подписчики (subscribers) получают **свою копию** сообщения
- Новые подписчики могут добавляться **без изменения** publisher'а
- Подписчики могут отписываться в любой момент

**Реальные примеры:**

text

```
Событие: "Заказ #1234 создан"

  ┌── Notification Service   → отправляет email клиенту
  │
  ├── Inventory Service      → резервирует товар на складе
  │
  ├── Analytics Service      → обновляет дашборд продаж
  │
  ├── Fraud Detection        → проверяет заказ на мошенничество
  │
  └── Loyalty Service        → начисляет бонусные баллы

Каждый сервис получает ОДНО И ТО ЖЕ событие
и обрабатывает его ПО-СВОЕМУ
```

#### 3. Request-Reply — Асинхронный запрос-ответ

text

```
    Client                                          Server
      │                                               │
      │──── запрос ────▶ [Request Queue] ────────────▶│
      │   (correlation_id: "abc-123")                  │
      │   (reply_to: "reply-queue-42")                │ обрабатывает
      │                                               │
      │◀── ответ ────── [Reply Queue 42] ◀────────────│
      │   (correlation_id: "abc-123")                  │
      │                                               │
      ▼ сопоставляет по correlation_id                 ▼

    • Correlation ID — связывает запрос с ответом
    • Reply-to       — указывает КУДА отправить ответ
    • Клиент может отправить много запросов параллельно
      и получать ответы в ЛЮБОМ порядке
```

**Зачем нужен, если есть HTTP?**

text

```
• Длительная обработка (генерация отчёта за 30 минут)
• Клиент не хочет держать соединение открытым
• Нужна гарантия, что запрос не потеряется
• Нужен retry при сбоях (брокер переотправит)
```

#### 4. Dead Letter Queue (DLQ) — Очередь «мёртвых» писем

text

```
                    ┌─────────────────┐
  Producer ──msg──▶ │   Main Queue    │──────▶ Consumer
                    └─────────────────┘           │
                                                  │ Ошибка обработки!
                                                  │ Попытка 1 ❌
                                                  │ Попытка 2 ❌
                                                  │ Попытка 3 ❌
                                                  │
                                                  ▼
                    ┌─────────────────┐     Максимум попыток
                    │ Dead Letter     │◀─── исчерпан
                    │ Queue (DLQ)     │
                    └─────────────────┘
                           │
                           ▼
                    Ручной анализ / Алерт / Повторная обработка
```

**Что попадает в DLQ:**

text

```
• Сообщение с невалидными данными (битый JSON, неизвестный формат)
• Превышено количество попыток обработки (max retries)
• Истёк TTL (Time To Live) сообщения
• Consumer явно отклонил сообщение (NACK + reject)
• Сообщение, для которого нет подходящей очереди (unroutable)
```

**Зачем нужна DLQ:**

text

```
• Проблемные сообщения НЕ блокируют основную очередь
• Можно проанализировать причину ошибки
• Можно повторно обработать после исправления бага
• Мониторинг: алерт, если в DLQ копятся сообщения
```

#### 5. Priority Queue — Очередь с приоритетами

text

```
    Обычная очередь (FIFO):
    [msg1] [msg2] [msg3] [msg4] → обработка в порядке поступления

    Priority Queue:
    [LOW:msg1] [LOW:msg2] [HIGH:msg3] [CRITICAL:msg4]
                                              ▲
                                   обрабатывается ПЕРВЫМ

    Порядок обработки: msg4 → msg3 → msg1 → msg2
```

**Реальные примеры:**

text

```
• VIP-клиенты обрабатываются раньше обычных
• Критические алерты безопасности → максимальный приоритет
• Платные заказы приоритетнее бесплатных пробных
```

#### 6. Delay / Scheduled Queue — Отложенные сообщения

text

```
    Producer ──msg──▶ [Delay Queue]
                      "доставить через 30 минут"
                           │
                           │  ⏰ ждёт 30 минут...
                           │
                           ▼
                      [Main Queue] ──▶ Consumer
```

**Реальные примеры:**

text

```
• Отправить напоминание клиенту через 1 час после регистрации
• Повторить попытку через 5 минут при ошибке (retry with backoff)
• Автоматически отменить заказ, если не оплачен через 30 минут
• Отправить follow-up email через 3 дня после покупки
```

#### 7. Competing Consumers — Конкурирующие потребители

text

```
                     ┌──────────┐
                ┌───▶│Consumer 1│ обработал msg1, msg4, msg7
                │    └──────────┘
  [Queue] ──────┼───▶│Consumer 2│ обработал msg2, msg5, msg8
                │    └──────────┘
                └───▶│Consumer 3│ обработал msg3, msg6, msg9
                     └──────────┘

  • Каждое сообщение → ОДНОМУ consumer
  • Нагрузка распределяется равномерно
  • Больше consumers = выше throughput
```

## 3. Брокер сообщений (Message Broker)

### Определение

Брокер сообщений — это **специализированное промежуточное программное обеспечение** (middleware), которое выступает **посредником** между отправителями и получателями сообщений. Он принимает сообщения, определяет кому их доставить, временно хранит и гарантирует доставку.

Брокер — это **«почтовое отделение»** для распределённых систем.

text

```
┌───────────────────────────────────────────────────────────────────┐
│                       MESSAGE BROKER                              │
│                                                                   │
│  ┌────────────┐    ┌──────────────────────────────┐    ┌────────┐ │
│  │            │    │      ROUTING ENGINE           │    │        │ │
│  │  ПРИЁМНИК  │───▶│                              │───▶│ДОСТАВКА│ │
│  │ (Receiver) │    │ Правила маршрутизации:       │    │        │ │
│  │            │    │ • По routing key             │    │ Push   │ │
│  └────────────┘    │ • По паттерну                │    │ или    │ │
│       ▲            │ • По заголовкам              │    │ Pull   │ │
│       │            │ • Broadcast                  │    │        │ │
│  Producers         └──────────────────────────────┘    └────────┘ │
│                              │                             │      │
│                              ▼                             │      │
│                    ┌──────────────────┐                    ▼      │
│                    │    ХРАНИЛИЩЕ     │              Consumers     │
│                    │                  │                           │
│                    │ • Очереди        │                           │
│                    │ • Топики         │                           │
│                    │ • Партиции       │                           │
│                    │ • Persistence    │                           │
│                    └──────────────────┘                           │
│                                                                   │
│                    ┌──────────────────┐                           │
│                    │ ГАРАНТИИ         │                           │
│                    │ • ACK / NACK     │                           │
│                    │ • Retry          │                           │
│                    │ • DLQ            │                           │
│                    │ • Ordering       │                           │
│                    │ • Deduplication  │                           │
│                    └──────────────────┘                           │
└───────────────────────────────────────────────────────────────────┘
```

### Ключевые функции брокера

text

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  1. ПРИЁМ СООБЩЕНИЙ                                              │
│     • Принимает от одного или нескольких продюсеров              │
│     • Валидирует формат (опционально)                           │
│     • Присваивает метаданные (timestamp, ID, offset)            │
│                                                                  │
│  2. МАРШРУТИЗАЦИЯ                                                │
│     • Определяет КУДА доставить сообщение                       │
│     • На основе routing key, паттернов, заголовков              │
│     • Может отправить в одну очередь, несколько или во все      │
│                                                                  │
│  3. ХРАНЕНИЕ                                                     │
│     • В памяти (быстро, но ненадёжно)                           │
│     • На диске (надёжно, переживает перезагрузку)               │
│     • Репликация на несколько узлов (высокая доступность)        │
│                                                                  │
│  4. ДОСТАВКА                                                     │
│     • Push: брокер САМА отправляет consumer'у (RabbitMQ)        │
│     • Pull: consumer САМА запрашивает сообщения (Kafka)         │
│                                                                  │
│  5. ГАРАНТИИ ДОСТАВКИ                                            │
│     • Acknowledgment (подтверждение обработки)                  │
│     • Retry (повторная доставка при ошибке)                     │
│     • Dead Letter Queue (для «неизлечимых» сообщений)           │
│     • Ordering (гарантия порядка)                               │
│                                                                  │
│  6. БАЛАНСИРОВКА НАГРУЗКИ                                        │
│     • Распределяет сообщения между consumers равномерно         │
│     • Round-robin или другие стратегии                           │
│                                                                  │
│  7. МОНИТОРИНГ И УПРАВЛЕНИЕ                                      │
│     • Размер очередей, throughput, latency                      │
│     • Consumer lag (насколько consumer отстаёт)                 │
│     • Web UI, API, метрики для Prometheus/Grafana               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### Модели доставки

text

```
═══════════ PUSH модель (RabbitMQ) ═══════════

Брокер АКТИВНО отправляет сообщения consumer'ам
как только они появляются в очереди.

  Broker ──▶ "Эй, Consumer! У меня новое сообщение для тебя!"

  ✅ Низкая задержка (сообщение доставлено сразу)
  ❌ Риск перегрузить consumer (нужен prefetch limit)


═══════════ PULL модель (Kafka) ═══════════

Consumer САМА периодически запрашивает новые сообщения.

  Consumer ──▶ "Брокер, есть что-нибудь новое для меня с offset 42?"

  ✅ Consumer контролирует скорость потребления
  ✅ Consumer может перечитать старые сообщения
  ❌ Чуть выше задержка (зависит от частоты polling)
```

### Гарантии доставки — подробно

text

```
╔════════════════════════════════════════════════════════════════╗
║                                                                ║
║  AT-MOST-ONCE (максимум один раз)                              ║
║  ──────────────────────────────────                            ║
║  «Выстрелил и забыл» (Fire and Forget)                         ║
║                                                                ║
║  Producer ──msg──▶ Broker ──msg──▶ Consumer                    ║
║  Нет подтверждений. Если что-то пошло не так — сообщение       ║
║  потеряно навсегда, но НИКОГДА не дублируется.                 ║
║                                                                ║
║  📌 Применение: логи, метрики, телеметрия                      ║
║     (потеря одного значения некритична)                         ║
║                                                                ║
╠════════════════════════════════════════════════════════════════╣
║                                                                ║
║  AT-LEAST-ONCE (минимум один раз)                              ║
║  ──────────────────────────────────                            ║
║  «Буду повторять, пока не подтвердишь»                         ║
║                                                                ║
║  Producer ──msg──▶ Broker ──msg──▶ Consumer                    ║
║  Consumer ──ACK──▶ Broker                                      ║
║                                                                ║
║  Если ACK потерялся → Broker отправит ПОВТОРНО →               ║
║  Consumer обработает ДВАЖДЫ.                                   ║
║                                                                ║
║  Сообщение ТОЧНО доставлено, но может быть ДУБЛИКАТ.           ║
║  Нужна ИДЕМПОТЕНТНОСТЬ на стороне consumer.                    ║
║                                                                ║
║  📌 Применение: большинство бизнес-задач                       ║
║     (обработка заказов, платежи — с идемпотентностью)           ║
║                                                                ║
╠════════════════════════════════════════════════════════════════╣
║                                                                ║
║  EXACTLY-ONCE (ровно один раз)                                 ║
║  ──────────────────────────────                                ║
║  «Святой Грааль» распределённых систем                         ║
║                                                                ║
║  Сообщение обработано РОВНО ОДИН раз — не теряется             ║
║  и не дублируется. Самое сложное в реализации.                 ║
║                                                                ║
║  Kafka реализует через:                                        ║
║  • Idempotent Producer (producer.id + sequence number)         ║
║  • Transactional API (атомарные read-process-write)            ║
║                                                                ║
║  📌 Применение: финансовые транзакции, критичные данные        ║
║                                                                ║
╚════════════════════════════════════════════════════════════════╝
```

## 4. Kafka vs RabbitMQ vs ActiveMQ — детальное сравнение

### Философия

text

```
╔════════════════════════════════════════════════════════════════════╗
║                                                                    ║
║  🐰 RabbitMQ — «УМНЫЙ БРОКЕР, ТУПОЙ ПОТРЕБИТЕЛЬ»                  ║
║                                                                    ║
║  Брокер берёт на себя всю сложность:                               ║
║  • Решает КОМУ доставить (routing)                                 ║
║  • Решает КАК доставить (push)                                     ║
║  • Отслеживает, что доставлено (ACK)                               ║
║  • Удаляет обработанные сообщения                                  ║
║  Consumer просто получает и обрабатывает.                           ║
║                                                                    ║
╠════════════════════════════════════════════════════════════════════╣
║                                                                    ║
║  ⚡ Kafka — «ТУПОЙ БРОКЕР, УМНЫЙ ПОТРЕБИТЕЛЬ»                      ║
║                                                                    ║
║  Брокер — простой журнал (append-only log):                        ║
║  • Просто записывает ВСЁ подряд                                   ║
║  • Хранит ВСЁ (не удаляет после обработки)                        ║
║  Consumer сам решает:                                              ║
║  • ЧТО читать (какие топики/партиции)                              ║
║  • ОТКУДА читать (с какого offset)                                 ║
║  • КОГДА читать (pull-модель)                                      ║
║  • Может перечитать старые данные (replay)                         ║
║                                                                    ║
╠════════════════════════════════════════════════════════════════════╣
║                                                                    ║
║  🏢 ActiveMQ — «КОРПОРАТИВНЫЙ СТАНДАРТ»                            ║
║                                                                    ║
║  Реализация JMS-стандарта (Java Message Service):                  ║
║  • Полная совместимость с Java EE экосистемой                      ║
║  • Поддержка множества протоколов из коробки                       ║
║  • Простота настройки и использования                               ║
║  • Фокус на совместимость, а не на производительность              ║
║                                                                    ║
╚════════════════════════════════════════════════════════════════════╝
```

### Архитектура каждого брокера

#### RabbitMQ — Exchange + Bindings + Queues

text

```
                        АРХИТЕКТУРА RabbitMQ

Producer                    BROKER                         Consumer
────────     ┌────────────────────────────────────┐      ──────────
             │                                    │
  msg ──────▶│  ┌──────────┐                      │
  (routing   │  │          │  binding              │
   key:      │  │ EXCHANGE │──(key=order.eu)──▶ [Queue EU] ──▶ Consumer EU
   "order.eu"│  │          │                      │
  )          │  │ (Direct) │──(key=order.us)──▶ [Queue US] ──▶ Consumer US
             │  │          │                      │
             │  └──────────┘                      │
             │                                    │
             └────────────────────────────────────┘

4 ТИПА EXCHANGE:

┌──────────┬────────────────────────────────────────────────────────┐
│ Direct   │ Точное совпадение routing key                          │
│          │ routing_key="order.eu" → Queue с binding "order.eu"   │
├──────────┼────────────────────────────────────────────────────────┤
│ Topic    │ Паттерн с wildcards                                    │
│          │ "order.*"  → order.eu, order.us, order.jp             │
│          │ "order.#"  → order.eu, order.eu.premium               │
│          │ *  = ровно одно слово,  # = ноль или более слов       │
├──────────┼────────────────────────────────────────────────────────┤
│ Fanout   │ Broadcast — всем привязанным очередям                  │
│          │ Routing key игнорируется                               │
├──────────┼────────────────────────────────────────────────────────┤
│ Headers  │ Маршрутизация по заголовкам сообщения                  │
│          │ headers: {region: "eu", type: "premium"}              │
└──────────┴────────────────────────────────────────────────────────┘

ОСОБЕННОСТИ RabbitMQ:
  • Erlang → отличная конкурентность (легковесные процессы)
  • Сообщение УДАЛЯЕТСЯ из очереди после ACK
  • Push-модель: брокер сам отправляет consumer'у
  • Prefetch count: ограничивает кол-во неподтверждённых сообщений
  • Plugins: management UI, shovel, federation, STOMP, MQTT
```

#### Apache Kafka — Distributed Commit Log

text

```
                        АРХИТЕКТУРА KAFKA

Producer                    KAFKA CLUSTER                    Consumer
────────     ┌──────────────────────────────────────────┐  ──────────
             │                                          │
  msg ──────▶│  Topic: "orders"                         │
  (key:      │  ┌─────────────────────────────────────┐ │
   "user42") │  │ Partition 0:                        │ │
             │  │ [off:0][off:1][off:2][off:3]...[off:N]│──▶ Consumer A
             │  ├─────────────────────────────────────┤ │     (Group 1)
             │  │ Partition 1:                        │ │
             │  │ [off:0][off:1][off:2]...[off:M]     │──▶ Consumer B
             │  ├─────────────────────────────────────┤ │     (Group 1)
             │  │ Partition 2:                        │ │
             │  │ [off:0][off:1]...[off:K]            │──▶ Consumer C
             │  └─────────────────────────────────────┘ │     (Group 1)
             │                                          │
             │  Replication Factor: 3                    │
             │  Partition 0: Leader(Broker1),            │
             │               Follower(Broker2, Broker3) │
             │                                          │
             └──────────────────────────────────────────┘
                                │
                                ▼
                    Consumer Group 2 (независимо!)
                    Читает ТЕ ЖЕ данные со своими offset'ами


КЛЮЧЕВЫЕ КОНЦЕПЦИИ:

  Topic (Топик)
    └── Логическая категория сообщений ("orders", "payments", "logs")

  Partition (Партиция)
    └── Физическое разделение топика для параллелизма
    └── Порядок гарантирован ВНУТРИ одной партиции
    └── Сообщения распределяются по hash(key) % num_partitions

  Offset
    └── Уникальный порядковый номер сообщения внутри партиции
    └── Consumer запоминает свой offset и может перемотать назад

  Consumer Group
    └── Группа consumers, которые ДЕЛЯТ партиции между собой
    └── Каждая партиция → ровно один consumer в группе
    └── Разные группы читают НЕЗАВИСИМО

  Retention Policy
    └── Сообщения хранятся ОПРЕДЕЛЁННОЕ ВРЕМЯ (по умолчанию 7 дней)
    └── Или до определённого РАЗМЕРА
    └── НЕ удаляются после чтения!

ОСОБЕННОСТИ KAFKA:
  • Append-only log → последовательная запись на диск → ОЧЕНЬ быстро
  • Pull-модель: consumer сам запрашивает данные
  • Zero-copy: данные передаются напрямую из файла в сеть
  • Горизонтальное масштабирование: больше партиций → больше параллелизма
  • Идеально для Event Sourcing, CDC, Stream Processing
```
