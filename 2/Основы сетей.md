### 2. Модель TCP/IP (стек протоколов) - сравнение

**Соответствие уровней:**
- **Прикладной TCP/IP** = 5,6,7 OSI.
- **Транспортный TCP/IP** = 4 OSI.
- **Сетевой (Internet) TCP/IP** = 3 OSI.
- **Уровень сетевых интерфейсов TCP/IP** = 1 и 2 OSI.
**Главные отличия:**
1. **Уровень представления и сеансовый** в TCP/IP отсутствуют как отдельные; их функции реализуются прикладными протоколами.
2. **Четкость границ:** OSI строго разграничивает уровни. TCP/IP допускает "сквозную" функциональность (например, протокол ARP работает "между" канальным и сетевым уровнями).
3. **Протоколы:** OSI - модель до протоколов, TCP/IP - модель, построенная _на основе_ существующих протоколов.

### 3. Протоколы по уровням (с пояснениями)

- **Канальный уровень:**
    - **Ethernet (IEEE 802.3):** Доминирующий стандарт для проводных сетей. Использует MAC-адреса. 
    - **ARP (Address Resolution Protocol):** Технически "протокол между уровнями". Спрашивает: "У кого IP 192.168.1.5? Сообщи свой MAC". Нужен для того, чтобы отправить кадр конкретному устройству внутри локальной сети.
    - **VLAN (802.1Q):** Не протокол как таковой, а расширение Ethernet, позволяющее резать один физический коммутатор на несколько изолированных логических сетей.
- **Сетевой уровень:**
    - **IPv4:** 32-битная адресация (4.3 млрд адресов, де-факто исчерпаны).
    - **IPv6:** 128-битная адресация (пришел на смену IPv4 из-за нехватки адресов).
    - **ICMP (Internet Control Message Protocol):** Передача служебных сообщений об ошибках. Именно его использует утилита `ping` (Echo Request / Echo Reply).
- **Транспортный уровень:**
    - **TCP:** С установлением соединения (3-way handshake: SYN -> SYN-ACK -> ACK). Нумерация последовательности пакетов, подтверждение получения (ACK), повторная отправка потерянных сегментов.
    - **UDP:** "Заглушил и полетел". Идеален для DNS, потокового видео (потеря пары кадров не страшна, в отличие от задержки на пересылку), голосовых вызовов (VoIP).
- **Прикладной уровень:**
    - **HTTP/HTTPS:** Веб-серфинг.
    - **FTP/FTPS:** Передача файлов (активный/пассивный режимы).
    - **SMTP/POP3/IMAP:** Отправка и получение электронной почты.
    - **SSH:** Удаленное защищенное управление устройством (замена Telnet).
    - **DHCP:** Автоматически выдает устройствам IP-адрес, маску, шлюз и DNS при подключении к сети.
    - **DNS:** Резолвинг имен.

### 4. TCP vs UDP - детализация

**TCP (Transmission Control Protocol):**
- **Сегменты:** Упорядочиваются.
- **Контроль перегрузок:** Если на пути возникают потери, TCP снижает скорость передачи.
- **Заголовок:** Большой (20-60 байт).
- **Когда использовать:** Загрузка файлов, просмотр веб-страниц, email — там важна целостность.
**UDP (User Datagram Protocol):**
- **Дейтаграммы:** Могут прийти в любом порядке или не прийти вообще.
- **Без "тормозов":** Отправил данные и забыл. Нет подтверждений.
- **Заголовок:** Маленький (8 байт).
- **Broadcast/Multicast:** Поддерживает широковещательную рассылку (в отличие от TCP).
- **Когда использовать:** Стриминг (Twitch, Youtube), онлайн-игры (CS, Valorant), DNS-запросы.

### 5. Публичные и приватные IP - NAT

**Важное дополнение: NAT (Network Address Translation)**  
Раз у нас есть приватные IP, а интернет требует публичные, как они работают? Используется NAT на роутере.
- **Как это работает:** Компьютер с IP `192.168.0.5` (приватный) просит открыть сайт. Роутер (у которого есть публичный IP `95.25.16.8`) подменяет адрес отправителя в пакете на свой публичный и запоминает, что ответ нужно переслать обратно на `192.168.0.5`. Для интернета весь ваш дом выглядит как один компьютер с IP `95.25.16.8`.
**Специальные IP-адреса:**
- **127.0.0.1 (localhost):** "Сам себе". Указывает на текущее устройство. Используется для тестов.
- **0.0.0.0:** Обычно означает "любой адрес" или "неизвестный адрес".

### 6. Маска подсети - как считать?

Маска подсети — это 32-битное число, используемое в сетях [IPv4](https://ru.wikipedia.org/wiki/%D0%9C%D0%B0%D1%81%D0%BA%D0%B0_%D0%BF%D0%BE%D0%B4%D1%81%D0%B5%D1%82%D0%B8) для разделения IP-адреса на адрес сети и адрес хоста (устройства). Она определяет размер подсети, указывая, какие устройства находятся в одной локальной сети и могут обмениваться данными напрямую. Обычный формат — 255.255.255.0 или CIDR-префикс (например, /24).
**CIDR нотация:** Вместо `255.255.255.0` пишут `/24`. Это значит, что первые 24 бита (3 октета) отведены под сеть.
- `/24` = `255.255.255.0` (маска 3 байта сети, 1 байт хосты)
- `/16` = `255.255.0.0` (сеть класса B)
- `/8` = `255.0.0.0` (сеть класса A)
- `/32` = `255.255.255.255` (один конкретный хост)
**Зачем еще нужна маска:**  
Чтобы определить **широковещательный адрес (Broadcast)**. Это адрес, пакеты на который получат все устройства в данной подсети. Вычисляется как IP-адрес сети + побитовая инверсия маски (все биты хоста = 1).
- Пример: Сеть `192.168.1.0/24`. Broadcast = `192.168.1.255`.

### 7. Инкапсуляция (полный пример с заголовками)

Допустим, вы отправляете письмо в Telegram: "Привет".
1. **Данные:** Текст "Привет" (в кодировке UTF-8). **PDU (Protocol Data Unit): Данные**.
2. **Транспорт (TCP):** Добавляет заголовок. В нем: Порт источника (случайный, например 54321), порт назначения (443 для HTTPS). **Получается: [TCP-заголовок + Данные] = TCP-сегмент**.
3. **Сеть (IP):** Добавляет свой заголовок. В нем: IP источника (ваш приватный `192.168.1.5`), IP назначения (сервер Telegram `149.154.167.51`), протокол верхнего уровня (6 - для TCP). **Получается: [IP-заголовок + TCP-заголовок + Данные] = IP-пакет**.
4. **Канал (Ethernet):** Добавляет свой заголовок. В нем: MAC-адрес источника (вашей сетевухи), MAC-адрес назначения (роутера, т.к. сервер Telegram не в локальной сети), тип протокола (0x0800 - для IPv4). В _конец_ кадра добавляется **FCS (Frame Check Sequence)** - контрольная сумма для проверки целостности. **Получается: [Ethernet-заголовок + IP-пакет + FCS] = Кадр**.
5. **Физический:** Кадр превращается в импульсы тока и уходит в розетку.

### 8. Сетевые интерфейсы - типы

**Сетевой интерфейс** — это точка подключения устройства (хоста) к сети. Проще говоря, это «порт», через который компьютер отправляет и получает сетевые данные.  
Он может быть:
- **Физическим** — например, Ethernet-порт, Wi-Fi-адаптер, LTE-модем.
- **Виртуальным** — создаётся программно (VPN-интерфейс, loopback `lo`, интерфейсы контейнеров и т.п.).
	- **Loopback (lo):** Всегда имеет IP `127.0.0.1`. Любой трафик, отправленный на этот интерфейс, уходит сразу на вход стека протоколов того же компьютера. Тестирование без сети.
	- **Туннельные интерфейсы (VPN):** Создаются, когда вы подключаетесь к VPN. Ваш компьютер получает новый IP (из сети VPN) и весь трафик "заворачивается" в этот интерфейс, шифруется и отправляется в туннель.
Каждый сетевой интерфейс обычно имеет:
		- MAC-адрес (для канального уровня),
		- один или несколько IP-адресов,
		- собственные параметры (маска, шлюз, MTU и др.).
#### Может ли у одного хоста быть несколько интерфейсов?
Да, это обычная ситуация.
Примеры:
- Ноутбук одновременно подключён по **Wi-Fi** и по **Ethernet**.
- Сервер с несколькими сетевыми картами для разных подсетей.
- Виртуальные машины и контейнеры создают дополнительные виртуальные интерфейсы.
- Один интерфейс может иметь несколько IP-адресов.
Операционная система просто выбирает, через какой интерфейс отправлять трафик (по таблице маршрутизации).

### 9. Порт - важные детали
**Порт** — это логический идентификатор процесса или службы на хосте.  
В модели **OSI** он относится к **транспортному уровню (L4)** и используется протоколами вроде TCP и UDP.
#### Зачем нужен порт
IP-адрес указывает **какому устройству** доставить пакет, а порт — **какой программе внутри устройства** его передать.
Без портов компьютер не смог бы одновременно:
- принимать веб-страницы,
- скачивать файлы,
- получать почту,
- держать открытые соединения разных приложений.
Порт решает задачу **мультиплексирования**: много соединений через один сетевой интерфейс и один IP.
- **Socket (Сокет):** Комбинация IP-адреса и номера порта. Это полный идентификатор соединения. `192.168.1.5:54321` (клиент) -> `149.154.167.51:443` (сервер).
- **Слушающий порт (Listening port):** Сервер открывает порт и "слушает" его, ожидая входящие соединения.
- **Ephemeral ports (Кратковременные порты):** Диапазон портов (обычно 32768-61000), которые ОС выделяет клиентским приложениям для исходящих соединений. Когда вы открываете 10 вкладок браузера, каждая получает свой временный порт, чтобы ответы не перепутались.

### 10. DNS - глубже

**DNS (Domain Name System)** — это «телефонная книга интернета». Он переводит удобные для человека доменные имена (например, `example.com`) в IP-адреса, которые понимают компьютеры (например, `93.184.216.34`).
#### Зачем он нужен
Людям проще запоминать имена, а сети работают по IP. DNS делает сопоставление **имя → IP**, чтобы браузер знал, к какому серверу подключаться.
**Типы записей DNS:**
- **A (Address):** Сопоставляет домен с IPv4-адресом (`example.com` -> `93.184.216.34`).
- **AAAA (Quad A):** Сопоставляет домен с IPv6-адресом.
- **CNAME (Canonical Name):** Псевдоним (алиас). Указывает, что `www.example.com` — это то же самое, что `example.com`.
- **MX (Mail Exchange):** Указывает, на какой сервер отправлять почту для домена (`@example.com`).
- **TXT (Text):** Произвольный текст. Часто используется для подтверждения владения доменом и для SPF/DKIM (защита почты от спама).
- **NS (Name Server):** Указывает, какие DNS-серверы отвечают за этот домен.
**Как происходит запрос (упрощенно):**
1. Вы ввели `mail.google.com`.
2. **Резолвер (ваш DNS-сервер, например 8.8.8.8)** проверяет кэш.
3. Если нет, он идет к **корневому DNS-серверу** (их 13 в мире). Корневой говорит: "Иди к серверам зоны .com".
4. Резолвер идет к серверу .com. Тот говорит: "Иди к серверам Google".
5. Резолвер идет к авторитативному серверу Google. Тот возвращает IP-адрес `mail.google.com`.
6. Резолвер отдает IP вашему компьютеру.

### 11. Дополнительные концепции (Бонус)

- **MTU (Maximum Transmission Unit):** Максимальный размер полезной нагрузки, которую можно передать в одном кадре на канальном уровне (обычно 1500 байт для Ethernet). Если IP-пакет больше, его фрагментируют.
- **Маршрутизация (Routing):** Процесс выбора пути для пакета в сети. Статическая (админ прописывает путь руками) и динамическая (протоколы OSPF, BGP сами строят карту сети).
- **Прокси vs NAT:**
    - _NAT_ работает на сетевом уровне, подменяя адреса (невидимо для приложения).
    - _Прокси_ работает на прикладном уровне. Приложение (браузер) специально отправляет запрос прокси-серверу, а тот уже идет на сайт.
### 12. Уровни модели OSI: Представления и Сеансовый

В модели OSI эти уровни часто объединяют с прикладным в модели TCP/IP, но их функции важны:
- **Уровень представления (Presentation Layer):**
    - **Задачи:** Трансляция данных из формата приложения в сетевой формат и обратно.
    - **Функции:** Шифрование/дешифрование данных (SSL/TLS на этом уровне), сжатие данных, перекодировка (например, из ASCII в EBCDIC).
    - _Примеры протоколов/форматов:_ **SSL/TLS**, JPEG, GIF, ASCII, MPEG.
- **Сеансовый уровень (Session Layer):**
    - **Задачи:** Управление диалогом между двумя устройствами (установление, поддержание и завершение сеанса связи).
    - **Функции:** Синхронизация (вставка контрольных точек в поток данных, чтобы при обрыве не начинать передачу заново), управление очередью передачи (кто говорит).
    - _Примеры протоколов:_ **NetBIOS**, **PPTP**, RPC.

### 14. Расчет хостов в сети 192.168.1.0/24

- **Запись:** `/24` означает, что под **сеть** (Network ID) выделено 24 бита из 32.
- **Остаток под хосты:** 32 - 24 = **8 бит**.
- **Формула расчета доступных хостов:**  
    2n−22n−2  
    Где `n` — количество бит, отведенных под хосты.
- **Почему минус 2?**
    1. Один адрес — **Network Address** (самый первый, где все биты хостовой части равны 0). В данном случае это `192.168.1.0`.
    2. Один адрес — **Broadcast Address** (самый последний, где все биты хостовой части равны 1). В данном случае это `192.168.1.255`.  
- **Расчет:** $2^8 - 2 = 256 - 2 = 254$ хоста.
- **Ответ:** В сети `192.168.1.0/24` может быть **254 хоста** (адреса с `192.168.1.1` по `192.168.1.254`).

### 15. Типы записей в DNS (Resource Records)

DNS использует различные типы записей для сопоставления имен и сервисов с IP-адресами и другой информацией:

|Тип записи|Полное имя|Назначение|
|---|---|---|
|**A**|Address|Сопоставляет доменное имя с **IPv4-адресом**.|
|**AAAA**|IPv6 Address|Сопоставляет доменное имя с **IPv6-адресом**.|
|**CNAME**|Canonical Name|Создает псевдоним (алиас) для другого доменного имени (например, `www` указывает на `@`).|
|**MX**|Mail Exchange|Указывает на почтовые серверы, обрабатывающие почту для домена (имеет приоритет).|
|**NS**|Name Server|Указывает на DNS-серверы, обслуживающие данный домен.|
|**TXT**|Text|Позволяет администраторам вставлять текст. Используется для верификации домена, SPF, DKIM (защита почты).|
|**PTR**|Pointer|Обратная запись: сопоставляет IP-адрес с доменным именем (обратный DNS).|
|**SOA**|Start of Authority|Содержит административную информацию о зоне (главный DNS-сервер, email админа, серийный номер зоны).|

### 16. Маршрутизация в сетях

**Маршрутизация (Routing)** — это процесс определения оптимального пути для передачи пакета данных от отправителя к получателю через составную сеть (интерсеть).  
Устройства, выполняющие эту функцию, называются **маршрутизаторами**. Они принимают решение на основе **таблицы маршрутизации**, которая содержит информацию об известных сетях и интерфейсах, через которые к ним можно добраться.

---

### 17. Статическая vs Динамическая маршрутизация

|Характеристика|Статическая маршрутизация|Динамическая маршрутизация|
|---|---|---|
|**Настройка**|Администратор **вручную** прописывает каждый маршрут.|Маршруты изучаются автоматически с помощью протоколов.|
|**Протоколы**|Не используется.|RIP, OSPF, EIGRP, BGP.|
|**Нагрузка на CPU**|Отсутствует.|Требует ресурсов для расчетов и обновлений.|
|**Масштабируемость**|Плохая (подходит для небольших сетей).|Отличная (используется в больших и глобальных сетях).|
|**Адаптивность**|Нет. При обрыве канала трафик не пойдет, пока админ не исправит маршрут вручную.|Высокая. Протоколы автоматически перестраивают таблицы при изменениях топологии.|
|**Безопасность**|Выше (маршрут известен только администратору).|Ниже (маршруты анонсируются наружу, если не настроена аутентификация).|

### 18. NAT (Network Address Translation)

**NAT (Преобразование сетевых адресов)** — это механизм, который преобразует IP-адреса в заголовках пакетов по мере их прохождения через маршрутизирующее устройство (обычно роутер или файрвол).
**Основное назначение:**
1. **Экономия IPv4-адресов:** Позволяет множеству устройств в локальной сети (с "серыми" частными адресами, RFC 1918) выходить в интернет, используя один или несколько публичных ("белых") IP-адресов.
2. **Безопасность:** Скрывает внутреннюю структуру сети от внешнего мира. Инициация соединения извне внутрь без специальных настроек (проброса портов) невозможна.
**Как это работает (SNAT/Masquerade):**  
Устройство (ПК) с адресом `192.168.1.5` хочет открыть сайт. Роутер с публичным IP `93.180.10.25` подменяет адрес отправителя в пакете на свой публичный. Когда приходит ответ, роутер по специальной таблице соединений (NAT Table) "вспоминает", кому предназначался ответ, и подменяет адрес получателя обратно на `192.168.1.5`.
### 19. DNS в Linux

#### Как получить значение DNS записи?
Для запросов к DNS-серверам используются специализированные утилиты.
- **`dig` (Domain Information Groper):** Самый подробный инструмент, используемый для диагностики.
    - `dig example.com` — получить A запись.   
    - `dig example.com MX` — получить MX записи.    
    - `dig @8.8.8.8 example.com` — запросить у конкретного сервера (Google Public DNS).  
- **`nslookup`:** Более старый, но всё ещё популярный инструмент (интерактивный режим).
    - `nslookup example.com` 
    - `nslookup example.com 1.1.1.1` (указать сервер)   
- **`host`:** Упрощенная утилита для быстрых запросов.
    - `host -t A example.com`
#### Как узнать адрес DNS-сервера в Linux?
DNS-серверы, которые использует система, хранятся в файле `/etc/resolv.conf`. Однако в современных системах с `systemd` и NetworkManager информация может управляться динамически.
1. **Посмотреть содержимое `/etc/resolv.conf`:**
    bash
    cat /etc/resolv.conf
    Строки `nameserver` содержат IP-адреса DNS-серверов.
2. **Использовать `resolvectl` (systemd-resolved):**  
    Если используется `systemd-resolved`, можно посмотреть статус:
    bash
    resolvectl status
    Эта команда покажет DNS-серверы для каждого сетевого интерфейса.
3. **Через NetworkManager (если используется):**
    bash
    nmcli dev show <имя_интерфейса> | grep DNS

### 20. Маршрутизация в Linux

#### Как добавить статический маршрут?

Добавление записей в таблицу маршрутизации ядра Linux.

- **Временное добавление (до перезагрузки):**  
    Используется команда `ip route` (новый стандарт) или `route` (устаревшая).
    bash
	Формат: ip route add <сеть> via <шлюз> dev <интерфейс>
    sudo ip route add 192.168.10.0/24 via 10.0.0.1 dev eth0
    Маршрут по умолчанию (default gateway)
    sudo ip route add default via 192.168.1.1
- **Постоянное добавление:**  
    Зависит от дистрибутива и менеджера сети.
    - **NetworkManager:** Настройка через `nmcli` или создание файлов в `/etc/NetworkManager/system-connections/`.
    - **systemd-networkd:** В файле конфигурации интерфейса (`/etc/systemd/network/20-wired.network`) добавляем секцию:
        ini
        [Route]
        Destination=192.168.10.0/24
        Gateway=10.0.0.1
    - **Старые дистрибутивы (SysVinit):** Прописывание команд в файл `/etc/rc.local` или в конфиги интерфейсов (например, `/etc/sysconfig/network-scripts/route-eth0` в CentOS 6).
        

### 21. Динамическая маршрутизация

Протоколы, позволяющие роутерам обмениваться информацией друг с другом для автоматического построения таблиц маршрутизации.
- **IGP (Interior Gateway Protocol):** Для работы внутри одной автономной системы (AS).
    - **RIP (Routing Information Protocol):** Старый, простой. Использует количество хопов (hop count) как метрику (макс. 15). Медленная сходимость.
    - **OSPF (Open Shortest Path First):** Самый популярный. Использует алгоритм состояния каналов (Dijkstra). Быстрая сходимость, масштабируется.
    - **IS-IS (Intermediate System to Intermediate System):** Похож на OSPF, чаще используется в сетях провайдеров.
    - **EIGRP (Enhanced Interior Gateway Routing Protocol):** Проприетарный протокол Cisco (гибридный, частично открыт).
- **EGP (Exterior Gateway Protocol):** Для обмена маршрутами между разными автономными системами (в интернете).
    - **BGP (Border Gateway Protocol):** Основной протокол глобального интернета. Работает с огромными таблицами маршрутизации, использует сложные политики, а не просто метрики.

### 22. Типы NAT (Network Address Translation)

Трансляция сетевых адресов, используемая для экономии IPv4 адресов и обеспечения безопасности.
- **SNAT (Source NAT):** Меняется IP-адрес источника пакета (обычно при выходе из локальной сети в интернет). Маршрутизация обратно работает автоматически.
- **DNAT (Destination NAT):** Меняется IP-адрес назначения пакета (обычно для проброса портов извне внутрь сети).
- **Статический NAT (1-to-1):** Один внутренний адрес всегда соответствует одному внешнему. Используется для серверов.
- **Динамический NAT (Many-to-Many):** Внутренние адреса маппятся на пул внешних адресов.
- **PAT / NAT Overload (Port Address Translation):** Самый распространенный тип (Masquerade). Множество внутренних адресов маскируются под один внешний IP, но с разными портами.
    

### 23. Настройка NAT на Linux

Linux использует подсистему Netfilter и утилиту `iptables` (или `nftables`).
**Сценарий:** Сделать из Linux-сервера роутер для локальной сети (`eth0` — LAN, `eth1` — WAN/Интернет).
1. **Включить форвардинг пакетов (пересылку между интерфейсами):**
    bash
    Временное включение
    sudo sysctl -w net.ipv4.ip_forward=1
    Постоянное включение (раскомментировать строку в /etc/sysctl.conf)
    net.ipv4.ip_forward = 1
2. **Настроить iptables (правило MASQUERADE):**  
    Это включит NAT для трафика, уходящего через внешний интерфейс (`eth1`).
    bash
    sudo iptables -t nat -A POSTROUTING -o eth1 -j MASQUERADE
3. **(Опционально) Добавить базовый файрвол:**  
    Разрешить установленные соединения и трафик для LAN.
    bash
    sudo iptables -A FORWARD -i eth0 -o eth1 -j ACCEPT
    sudo iptables -A FORWARD -i eth1 -o eth0 -m state --state ESTABLISHED,RELATED -j ACCEPT
    

### 24. Алгоритмы балансировки нагрузки

Методы распределения входящего трафика между несколькими серверами (бэкендами).
- **Round Robin:**
    - _Принцип:_ Запросы распределяются по кругу (1-й серверу, 2-й, 3-й, ...).
    - _Плюсы:_ Простота, равномерное распределение при одинаковой мощности серверов.
    - _Минусы:_ Не учитывает загрузку сервера, длительность запроса.
- **Least Connections:**
    - _Принцип:_ Новый запрос отправляется серверу с наименьшим количеством активных соединений.
    - _Плюсы:_ Хорошо работает для длительных сессий.
    - _Минусы:_ Сложнее в реализации.
- **IP Hash (Source IP Hash):**
    - _Принцип:_ IP-адрес клиента хэшируется; клиент всегда попадает на один и тот же сервер (если сервер не упал).
    - _Плюсы:_ Решает проблему сессий (session stickiness).
    - _Минусы:_ Может создать неравномерную нагрузку, если много трафика из одной подсети.
- **Weighted Round Robin / Least Connections:**
    - _Принцип:_ Тот же Round Robin или Least Connections, но с учетом весов (мощности) серверов. Более мощным отдается больше трафика.
    - _Плюсы:_ Гибкость настройки под гетерогенную среду.
- **Random with Two Choices:**
    - _Принцип:_ Выбираются два случайных сервера из пула, запрос отправляется тому, у которого меньше нагрузка.
    - _Плюсы:_ Хорошо распределяет нагрузку в больших распределенных системах.
- **Сравнение:**
    - _Round Robin_ — для простых HTTP-сервисов с короткими запросами.
    - _Least Connections_ — для WebSockets, API с долгими операциями.
    - _IP Hash_ — когда важно, чтобы пользователь не потерял сессию (например, корзина интернет-магазина, если сессия хранится локально на сервере). В современных системах часто предпочитают внешнее хранилище сессий (Redis/Memcached) и используют Round Robin.
