### 1. Структура HTTP-запроса

Каждый HTTP-запрос, отправляемый клиентом (браузером, curl и т.д.) на сервер, состоит из трех основных частей:
1. **Стартовая строка (Request Line):**
    - Содержит метод, URL (путь к ресурсу) и версию протокола.
    - _Пример:_ `GET /index.html HTTP/1.1`
2. **Заголовки (Headers):**
    - Набор пар `ключ: значение`, которые передают служебную информацию (метаданные) о запросе, клиенте и ожидаемом ответе.    
    - Заканчиваются **пустой строкой**.

3. **Тело сообщения (Body):**
    - Необязательная часть. Содержит данные, которые передаются серверу (например, JSON при отправке формы или загружаемый файл). Присутствует обычно в методах `POST`, `PUT`, `PATCH`.
_Схема:_
text
Метод URI HTTP/Версия
Заголовок1: значение1
Заголовок2: значение2
(пустая строка)
Тело запроса (данные)

---

### 2. Методы HTTP-запросов

Метод (глагол) указывает серверу, какое действие нужно выполнить с ресурсом.
- **GET** — получение данных. Самый распространенный метод. Не должен изменять состояние сервера (идемпотентен) и не имеет тела.
- **POST** — отправка данных на сервер. Обычно используется для создания нового ресурса (например, добавление пользователя) или отправки формы. Может изменять состояние.
- **PUT** — полное обновление существующего ресурса или создание ресурса с конкретным URI. Идемпотентен (сколько раз ни отправь — результат один).
- **PATCH** — частичное обновление ресурса (изменение только переданных полей).
- **DELETE** — удаление ресурса на сервере.
- **HEAD** — аналогичен GET, но сервер возвращает **только заголовки** (без тела). Используется для проверки существования ресурса или времени его последней модификации.
- **OPTIONS** — получение информации о том, какие методы поддерживает сервер для конкретного ресурса.

### 3. Заголовки HTTP (Headers)
Заголовки позволяют клиенту и серверу обмениваться дополнительной информацией.
#### Обязательны ли заголовки?
Строго обязательных заголовков с точки зрения стандарта для работы HTTP **нет** (кроме `Host` в HTTP/1.1, см. ниже). Однако на практике серверы могут требовать определенные заголовки (например, `Content-Length` или `Content-Type` при отправке тела).
- **Обязательный в HTTP/1.1:** `Host`
    - Указывает доменное имя сервера, к которому обращаются. Это обязательно, так как на одном IP-адресе может находиться множество сайтов (виртуальный хостинг).

#### Основные группы заголовков:
- **Заголовки запроса (Request Headers):** От клиента к серверу.
    - `Host`: [example.com](https://example.com/)
    - `User-Agent`: Mozilla/5.0 (информация о браузере/клиенте)
    - `Accept`: application/json (какой формат ответа ожидает клиент)
    - `Authorization`: Basic YWxhZGRpbjpvcGVuc2VzYW1l (данные для авторизации)
    - `Cookie`: sessionid=abc123
- **Заголовки ответа (Response Headers):** От сервера к клиенту.
    - `Server`: Apache/2.4.1 (информация о сервере)
    - `Set-Cookie`: sessionid=abc123 (установка куки в браузере)
    - `Cache-Control`: max-age=3600 (управление кэшированием)

- **Заголовки сущности (Entity Headers):** Описывают тело сообщения.
    - `Content-Type`: application/json (формат данных в теле)
    - `Content-Length`: 128 (длина тела в байтах)
    - `Content-Encoding`: gzip (способ сжатия тела)

### 4. Коды ответа HTTP (Status Codes)

Сервер всегда отвечает трехзначным кодом, который показывает результат обработки запроса.
#### Группы кодов (по первой цифре):

1. **1xx — Информационные (Informational):**
    - Запрос получен, обработка продолжается. Редко используются на практике.
    - _Пример:_ `100 Continue`
2. **2xx — Успех (Success):**
    - Запрос был успешно получен, понят и обработан.
    - _Примеры:_ `200 OK`, `201 Created` (создано), `204 No Content` (нет содержимого).
3. **3xx — Перенаправление (Redirection):**
    - Клиенту нужно выполнить дополнительное действие для завершения запроса (обычно перейти по другому URI).
    - _Примеры:_ `301 Moved Permanently` (переехало навсегда), `302 Found` (временный переезд), `304 Not Modified` (используй кэш).
4. **4xx — Ошибка клиента (Client Error):**
    - Запрос содержит ошибку или не может быть выполнен по вине клиента.
    - _Примеры:_ `400 Bad Request` (плохой запрос), `401 Unauthorized` (не авторизован), `403 Forbidden` (доступ запрещен), `404 Not Found` (не найдено).
5. **5xx — Ошибка сервера (Server Error):**
    - Сервер не смог выполнить корректный запрос из-за внутренней ошибки.
    - _Примеры:_ `500 Internal Server Error`, `502 Bad Gateway`, `503 Service Unavailable` (сервер временно недоступен).

### 5. Отправка HTTP-запросов из консоли Linux

Самый популярный инструмент — `curl`. Другой вариант — `wget`.
#### Примеры использования `curl`:

1. **Простой GET-запрос:**
    bash
    curl https://api.example.com/users
2. **GET с выводом только заголовков ответа (как HEAD):**
    bash
    curl -I https://api.example.com/users
3. **GET с дополнительными заголовками:**
    bash
    curl -H "Accept: application/json" -H "Authorization: Bearer TOKEN123" https://api.example.com/users
4. **POST-запрос с отправкой JSON (данные в теле):**
    bash
    curl -X POST https://api.example.com/users \
         -H "Content-Type: application/json" \
         -d '{"name": "Иван", "email": "ivan@example.com"}'
5. **POST с отправкой данных формы:**
    bash
    curl -X POST -F "username=ivan" -F "password=secret" https://example.com/login
6. **Сохранение вывода в файл:**
    bash
    curl -o saved_page.html https://example.com
7. **Показать ВСЕ (и заголовки запроса, и ответа, и тело):**
    bash
    curl -v https://api.example.com/users

#### Ключи `curl`, которые нужно знать:
- `-X` или `--request`: Указать метод (GET, POST, PUT, DELETE).
- `-H` или `--header`: Добавить заголовок.
- `-d` или `--data`: Отправить данные в теле (автоматически подразумевает метод POST и заголовок `Content-Type: application/x-www-form-urlencoded`).
- `-I` или `--head`: Получить только заголовки.
- `-v` или `--verbose`: Подробный вывод (дебаг). 
- `-o` или `--output`: Сохранить в файл.

### 6. Отличия HTTP/1.1 от HTTP/2

HTTP/2 кардинально меняет способ передачи данных по сети, оставаясь обратно совместимым с API HTTP/1.1 (те же методы, заголовки, коды).

|Характеристика|HTTP/1.1|HTTP/2|
|---|---|---|
|**Формат данных**|Текстовый (читаемый, но "болтливый")|**Бинарный** (эффективнее для парсинга, меньше ошибок)|
|**Мультиплексирование**|Одно соединение -> один запрос/ответ в момент времени (ведет к блокам).|**Одно соединение -> множество потоков**. Несколько запросов и ответов передаются параллельно.|
|**Сжатие заголовков**|Заголовки передаются как есть (повторы)|**HPACK** — сжатие заголовков (убирает дублирование, экономит трафик)|
|**Server Push**|Нет|Сервер может сам отправить ресурсы (CSS, JS) клиенту до того, как клиент их запросит.|
|**Приоритизация**|Нет|Есть возможность указать приоритет запросов.|

_Итог:_ HTTP/2 быстрее за счет параллельной загрузки и сжатия служебных данных.

### 7. Как работает HTTPS?

**HTTPS = HTTP + TLS/SSL**
Это не отдельный протокол, а надстройка поверх HTTP, которая шифрует данные.
#### Принцип работы (упрощенно):
1. **Приветствие (Handshake):** Клиент подключается к серверу и они договариваются о версии шифрования и наборе шифров (ciphers).
2. **Проверка сертификата:** Сервер отправляет клиенту свой **SSL-сертификат** (публичный ключ и данные о владельце). Клиент проверяет, доверяет ли он этому сертификату (см. CA).
3. **Обмен ключами:** Клиент создает временный симметричный ключ (сеансовый), шифрует его публичным ключом сервера (из сертификата) и отправляет обратно.
4. **Шифрованная передача:** Сервер расшифровывает сеансовый ключ своим **приватным ключом**. С этого момента начинается симметричное шифрование данных с использованием этого общего ключа, что обеспечивает конфиденциальность и целостность данных.

### 8. Сертификаты SSL/TLS

Сертификат — это цифровой паспорт сайта/сервера.
#### Какая информация содержится в сертификате?
- **Владелец:** Доменное имя (Common Name - CN), организация, город, страна.
- **Издатель:** Кто выдал сертификат (CA).
- **Публичный ключ:** Ключ для шифрования данных клиентом.
- **Срок действия:** С какого по какое число сертификат валиден.
- **Серийный номер и цифровая подпись издателя.**

#### Типы сертификатов (по валидации):
1. **DV (Domain Validation) — Подтверждение домена:**
    - Самый простой и дешевый (часто бесплатный, как Let's Encrypt).
    - CA проверяет только право управления доменом (например, через email).
    - В строке адреса будет только зеленый замочек.
2. **OV (Organization Validation) — Подтверждение организации:**
    - CA проверяет, что организация реально существует и имеет право на домен.
    - В сертификате появляются данные компании. В браузере можно кликнуть на замочек и увидеть "Кому выдан".
3. **EV (Extended Validation) — Расширенная проверка:**
    - Самая строгая и дорогая проверка. CA проверяет юридический статус, физический адрес и т.д.
    - Раньше в браузерах закрашивалась зеленая строка или показывалось название компании прямо в адресной строке. Сейчас это упростили.

### 9. Что такое CA (Certificate Authority) и зачем он нужен?

**CA (Центр сертификации)** — это доверенная третья сторона (организация), которая выпускает (подписывает) цифровые сертификаты.
- **Зачем он нужен?** Чтобы решить проблему "доверия". Если сервер пришлет самодельный сертификат, как клиент узнает, что это действительно сервер `google.com`, а не хакер? CA — это "нотариус" интернета.
- **Цепочка доверия:** В операционную систему или браузер уже "зашиты" корневые сертификаты основных CA (DigiCert, Sectigo, GlobalSign и др.). Если сертификат сервера подписан одним из этих CA, браузер доверяет ему автоматически.

### 10. Самоподписанный (Self-signed) vs Доверенный (Trusted)

- **Доверенный сертификат:**
    - Подписан авторитетным CA, который есть в хранилище доверенных корневых центров вашего браузера/ОС.    
    - Браузер показывает **зеленый замочек** и не выдает предупреждений.
    - Используется на публичных сайтах.
- **Самоподписанный сертификат:**
    - Создан и подписан вами же для своего сервера.
    - Нет цепочки доверия к корневому CA.
    - Браузер показывает **страшное красное предупреждение** "Ваше подключение не защищено" или "Ваше соединение не является приватным".
    - Используется для тестирования (`localhost`) или внутри закрытых корпоративных сетей, где админ сам добавил такой сертификат в доверенные на всех компьютерах.

### 11. Что такое CORS?

**CORS (Cross-Origin Resource Sharing)** — это механизм безопасности, реализованный в браузерах, который контролирует доступ к ресурсам (шрифтам, скриптам, API) с веб-страниц, загруженных с другого домена (**origin**).
- **Проблема (Same-Origin Policy - SOP):** По умолчанию браузер блокирует запросы с сайта `frontend.com` к API `api.backend.com`. Это делается для безопасности (чтобы скрипты с вредоносного сайта не лазили в ваш банк).
- **Решение (CORS):** Сервер (`api.backend.com`) должен явно разрешить доступ для `frontend.com`, отправив специальный заголовок в ответе.
#### Как это работает (про Preflight):
1. **Простой запрос (Simple Request):** Если запрос "простой" (GET/POST с простыми заголовками), браузер сразу отправляет запрос, но смотрит на заголовок ответа.
2. **Preflight (Предварительный запрос):** Если запрос сложный (например, с нестандартным заголовком `Authorization` или методом `PUT`), браузер сначала отправляет служебный запрос **OPTIONS** (не кэшируется) к серверу, спрашивая: "А можно ли моему сайту слать сюда `PUT` с токеном?".

#### Ключевые заголовки ответа сервера для CORS:
- `Access-Control-Allow-Origin: https://frontend.com` (или `*` для всех, но это небезопасно с куками)
- `Access-Control-Allow-Methods: GET, POST, PUT`
- `Access-Control-Allow-Headers: Content-Type, Authorization`
- `Access-Control-Allow-Credentials: true` (разрешить отправку кук и авторизационных данных)

### 12. Что такое цепочка сертификатов (Certificate Chain)?
Цепочка сертификатов (или иерархия доверия) — это связка из нескольких сертификатов, соединяющая сертификат твоего сайта (листовой) с корневым сертификатом CA (Certificate Authority), которому доверяет браузер.
#### Зачем это нужно?
Корневые сертификаты самых главных CA хранятся прямо в операционной системе и браузере. Если бы эти ключи были скомпрометированы, пришлось бы обновлять всю ОС. Поэтому корневые ключи хранят в максимально защищенных оффлайн-хранилищах и используют их крайне редко.
#### Уровни цепочки:
1. **Корневой сертификат (Root Certificate):** Самый главный. Находится в хранилище доверенных ОС/браузера. Подписывает промежуточные сертификаты.
2. **Промежуточный сертификат (Intermediate Certificate):** Подписан корневым. Именно они используются для подписи сертификатов конечных сайтов. Их может быть несколько в цепочке.
3. **Сертификат сервера (Leaf/Server Certificate):** Твой сертификат, подписанный промежуточным.

_Как это выглядит:_ Сервер отправляет клиенту свой сертификат + все промежуточные. Браузер, имея корневой, проверяет подпись промежуточного, а затем подпись твоего сертификата.

### 13. Как создать самоподписанный TLS-сертификат?

Для создания используется утилита `openssl`. Это полезно для разработки на `localhost`.
#### Создание самоподписанного сертификата (на 365 дней):
bash
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes
#### Что здесь происходит?
- `req -x509`: Создаем самоподписанный сертификат (x509 — стандарт).
- `-newkey rsa:4096`: Генерируем новый приватный ключ длиной 4096 бит.
- `-keyout key.pem`: Сохраняем приватный ключ в файл `key.pem`.
- `-out cert.pem`: Сохраняем сертификат в файл `cert.pem`.
- `-days 365`: Срок действия — 1 год.
- `-nodes` (No DES): Не шифровать приватный ключ паролем (чтобы nginx не спрашивал пароль при запуске).

После выполнения команды система задаст вопросы (страна, город, имя организации). Самый важный пункт — **Common Name (CN)**. Впиши сюда домен, для которого создаешь сертификат (например, `localhost` или `mysite.local`).

### 14. Как получить доверенный TLS-сертификат от CA?

Есть два основных пути: платный и бесплатный. Бесплатный вариант от Let's Encrypt — самый популярный.
#### Вариант А: Автоматический (Let's Encrypt + Certbot)
Этот способ подходит, если у тебя есть сервер с публичным IP и доменом, который на него смотрит.
1. **Установи Certbot** (менеджер сертификатов):
    bash
    sudo apt update
    sudo apt install certbot python3-certbot-nginx  # Для Nginx
    Или для Apache: sudo apt install certbot python3-certbot-apache
2. **Запусти получение сертификата:**
    bash
    sudo certbot --nginx -d example.com -d www.example.com
    - Certbot свяжется с серверами Let's Encrypt.
    - Докажет, что ты владеешь доменом (разместив временный файл на твоем сервере или настроив DNS).
    - Получит сертификаты.
    - Автоматически пропишет их в конфигурацию Nginx.
    - Настроит автоматическое продление (сертификаты Let's Encrypt живут 90 дней).

#### Вариант Б: Ручной (Платные CA или особые случаи)
1. **Генерация CSR (Certificate Signing Request):** На своем сервере генерируешь приватный ключ и запрос на подпись.
    bash
    openssl req -new -newkey rsa:2048 -nodes -out example.csr -keyout example.key
2. **Отправка CSR в CA:** Содержимое файла `example.csr` отправляешь в центр сертификации (например, Comodo, DigiCert, GlobalSign) и проходишь их проверку (email, телефон, документы).
3. **Получение сертификата:** CA присылает тебе файл сертификата и цепочку промежуточных сертификатов.
4. **Установка вручную:** Кладешь файлы на сервер и прописываешь пути в конфигурации веб-сервера.

### 15. Как настроить HTTPS в Nginx?

HTTPS в nginx настраивается путём подключения SSL-сертификата и включения защищённого порта 443.
#### 1. Получение SSL-сертификата
Для шифрования трафика используется сертификат, выданный центром сертификации.  
Чаще всего применяют бесплатные сертификаты от **Let's Encrypt**.
Сертификат содержит:
- публичный ключ сервера
- данные о домене
- цифровую подпись центра сертификации
#### 2. Настройка сервера nginx для HTTPS

В конфигурации сайта создаётся серверный блок с поддержкой SSL:
- nginx слушает порт **443**
- указываются пути к сертификату и приватному ключу
- включаются безопасные версии TLS
Основные параметры:
- `listen 443 ssl;`
- `ssl_certificate` — путь к сертификату
- `ssl_certificate_key` — путь к приватному ключу
#### 3. Перенаправление HTTP на HTTPS
Для обеспечения безопасности весь трафик с порта 80 перенаправляется на защищённый протокол:
- создаётся серверный блок для порта 80
- настраивается редирект 301 на https
#### 4. Проверка конфигурации
После настройки необходимо:
- проверить конфигурацию nginx
- перезапустить или перезагрузить сервер
### Итог
Настройка HTTPS в nginx включает:
1. получение SSL-сертификата
2. подключение сертификата в конфигурации nginx
3. запуск сервера на порту 443
4. редирект HTTP на HTTPS
Это обеспечивает шифрование трафика и безопасную передачу данных между клиентом и сервером.
### 16. Как разрешить стороннему сайту показывать картинки с нашего сайта?
По умолчанию сервер может ограничивать доступ к файлам по источнику запроса (Referer) или политике CORS. Чтобы внешний сайт мог загружать изображения, нужно явно разрешить такие запросы.
#### 1. Разрешение через CORS-заголовок
Сервер должен отправлять HTTP-заголовок, позволяющий кросс-доменный доступ к ресурсам.
В конфигурации nginx для изображений добавляется заголовок:
- `Access-Control-Allow-Origin` — указывает, каким сайтам разрешён доступ  
- можно указать конкретный домен или `*` (всем)
Смысл: браузер получает разрешение загружать файл с другого домена.
#### 2. Если включена защита от хотлинка
Иногда доступ к картинкам ограничен по заголовку Referer.  
Чтобы разрешить конкретный сайт:
- добавить его домен в список разрешённых источников
- либо отключить проверку Referer для нужной директории
Это позволяет внешнему сайту напрямую запрашивать изображения.
#### 3. Итог
Чтобы сторонний сайт мог показывать изображения:
1. разрешить кросс-доменный доступ через CORS
2. при необходимости настроить или отключить проверку Referer
3. перезапустить nginx после изменений
### 17. Можно ли обойти CORS?
**Краткий ответ:** Для браузера — **нет** (это его неотключаемая политика безопасности). Но полностью заблокировать доступ к данным нельзя.
#### Почему CORS не панацея?
CORS защищает _пользователя_ в браузере от того, чтобы скрипты с сайта А ходили на сайт Б от его имени. Но данные как таковые не защищены.
#### Как можно получить данные, несмотря на CORS?
1. **Через бэкенд (Backend Proxy):** Самый легальный способ. Если тебе нужно с сайта `frontend.com` получать данные с `api.protected.com`, у которого стоит блокировка CORS, ты создаешь простой прокси-эндпоинт на своем сервере (`/api/proxy`). Браузер идет на твой сервер, а твой сервер уже ходит на `api.protected.com`. Для `api.protected.com` запрос приходит _от сервера_, а не от браузера, поэтому политика CORS не применяется.
2. **Через небраузерные инструменты:** CORS работает только в браузере. `curl`, Postman, мобильное приложение, десктопная программа — они игнорируют CORS и спокойно получат данные.
3. **Отключение безопасности браузера (для разработки):** Можно запустить браузер с флагами, отключающими защиту.
    - _Chrome:_ `chrome --disable-web-security --user-data-dir="~/chromeTemp"`
    - Это опасно и используется только для локального тестирования.
