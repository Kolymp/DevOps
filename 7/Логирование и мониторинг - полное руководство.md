## 1. Уровни логирования

text

```
┌────────────────────────────────────────────────────────────┐
│         УРОВНИ ЛОГИРОВАНИЯ (LOG LEVELS)                    │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  СТАНДАРТНЫЕ УРОВНИ (от менее к более критичным):         │
│                                                            │
│  ┌──────────┬──────────────────────────────────────────┐  │
│  │ Уровень  │ Описание                                 │  │
│  ├──────────┼──────────────────────────────────────────┤  │
│  │ TRACE    │ Детальная отладочная информация         │  │
│  │          │ • Вход/выход из функций                 │  │
│  │          │ • Значения всех переменных              │  │
│  │          │ • Только для глубокой отладки           │  │
│  │          │ ⚠️ ОГРОМНЫЙ объём логов                 │  │
│  │          │                                          │  │
│  │          │ Пример:                                  │  │
│  │          │ TRACE: Entering function calculate()    │  │
│  │          │ TRACE: x=10, y=20                        │  │
│  │          │ TRACE: Exiting function, result=30      │  │
│  ├──────────┼──────────────────────────────────────────┤  │
│  │ DEBUG    │ Отладочная информация                   │  │
│  │          │ • Внутреннее состояние приложения       │  │
│  │          │ • Важные переменные                     │  │
│  │          │ • Логика принятия решений               │  │
│  │          │ • Для разработчиков                     │  │
│  │          │                                          │  │
│  │          │ Пример:                                  │  │
│  │          │ DEBUG: User authentication successful   │  │
│  │          │ DEBUG: Fetched 150 records from DB      │  │
│  │          │ DEBUG: Cache hit for key 'user:1001'    │  │
│  ├──────────┼──────────────────────────────────────────┤  │
│  │ INFO ⭐  │ Информационные сообщения                │  │
│  │          │ • Нормальное поведение приложения       │  │
│  │          │ • Важные события                        │  │
│  │          │ • Старт/стоп сервисов                   │  │
│  │          │ • По умолчанию в production             │  │
│  │          │                                          │  │
│  │          │ Пример:                                  │  │
│  │          │ INFO: Application started on port 8080  │  │
│  │          │ INFO: User 'john' logged in              │  │
│  │          │ INFO: Order #12345 created               │  │
│  ├──────────┼──────────────────────────────────────────┤  │
│  │ WARN     │ Предупреждения                          │  │
│  │          │ • Потенциальные проблемы                │  │
│  │          │ • Устаревшие методы (deprecated)        │  │
│  │          │ • Медленные запросы                     │  │
│  │          │ • НЕ критично, но требует внимания      │  │
│  │          │                                          │  │
│  │          │ Пример:                                  │  │
│  │          │ WARN: Database query took 5.2 seconds   │  │
│  │          │ WARN: Cache miss, fallback to DB        │  │
│  │          │ WARN: API rate limit: 90% used          │  │
│  ├──────────┼──────────────────────────────────────────┤  │
│  │ ERROR    │ Ошибки                                  │  │
│  │          │ • Что-то пошло не так                   │  │
│  │          │ • Операция не выполнена                 │  │
│  │          │ • НО приложение продолжает работать     │  │
│  │          │ • Требует исправления                   │  │
│  │          │                                          │  │
│  │          │ Пример:                                  │  │
│  │          │ ERROR: Failed to send email to user@... │  │
│  │          │ ERROR: Database connection timeout      │  │
│  │          │ ERROR: Payment gateway returned 500     │  │
│  ├──────────┼──────────────────────────────────────────┤  │
│  │ FATAL/   │ Критические ошибки                      │  │
│  │ CRITICAL │ • Приложение НЕ МОЖЕТ продолжать работу │  │
│  │          │ • Требуется немедленное вмешательство   │  │
│  │          │ • Crash, out of memory                  │  │
│  │          │                                          │  │
│  │          │ Пример:                                  │  │
│  │          │ FATAL: Cannot connect to database       │  │
│  │          │ FATAL: Out of memory, shutting down     │  │
│  │          │ CRITICAL: Configuration file not found  │  │
│  └──────────┴──────────────────────────────────────────┘  │
│                                                            │
│  ИЕРАРХИЯ (включение логов):                              │
│  ┌──────────────────────────────────────────────────────┐ │
│  │                                                      │ │
│  │  TRACE → всё (включая DEBUG, INFO, WARN, ERROR)     │ │
│  │    ↓                                                 │ │
│  │  DEBUG → DEBUG + INFO + WARN + ERROR + FATAL        │ │
│  │    ↓                                                 │ │
│  │  INFO  → INFO + WARN + ERROR + FATAL  ⭐ Production  │ │
│  │    ↓                                                 │ │
│  │  WARN  → WARN + ERROR + FATAL                        │ │
│  │    ↓                                                 │ │
│  │  ERROR → ERROR + FATAL                               │ │
│  │    ↓                                                 │ │
│  │  FATAL → только FATAL                                │ │
│  │                                                      │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  НАСТРОЙКА УРОВНЯ (примеры):                              │
│  ───────────────────────────                               │
│  Python (logging):                                        │
│  import logging                                           │
│  logging.basicConfig(level=logging.INFO)                  │
│                                                            │
│  Java (Log4j):                                            │
│  <logger name="com.example" level="INFO"/>                │
│                                                            │
│  Node.js (Winston):                                       │
│  const logger = winston.createLogger({                    │
│    level: 'info'                                          │
│  });                                                      │
│                                                            │
│  Go (logrus):                                             │
│  log.SetLevel(log.InfoLevel)                              │
│                                                            │
│  РЕКОМЕНДАЦИИ ПО ОКРУЖЕНИЯМ:                              │
│  ──────────────────────────────                            │
│  Development:  DEBUG (видеть всё)                         │
│  Staging:      INFO (как production)                      │
│  Production:   INFO (баланс детальности/объёма)           │
│  Troubleshoot: DEBUG временно (для диагностики)           │
│                                                            │
│  СТРУКТУРИРОВАННОЕ ЛОГИРОВАНИЕ (JSON):                    │
│  ────────────────────────────────────────                  │
│  {                                                        │
│    "timestamp": "2024-01-20T10:15:30.123Z",               │
│    "level": "ERROR",                                      │
│    "message": "Failed to process order",                  │
│    "context": {                                           │
│      "order_id": "12345",                                 │
│      "user_id": "1001",                                   │
│      "error_code": "PAYMENT_FAILED"                       │
│    },                                                     │
│    "stack_trace": "..."                                   │
│  }                                                        │
│                                                            │
│  → Легко парсится, индексируется, фильтруется             │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 3. Logstash

text

```
┌────────────────────────────────────────────────────────────┐
│         LOGSTASH                                           │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ОПРЕДЕЛЕНИЕ:                                             │
│  Серверный data processing pipeline для сбора, парсинга   │
│  и отправки логов                                         │
│                                                            │
│  PIPELINE (конвейер):                                     │
│  ┌──────────────────────────────────────────────────────┐ │
│  │                                                      │ │
│  │  INPUT  →  FILTER  →  OUTPUT                         │ │
│  │  (сбор)   (обработка)  (отправка)                   │ │
│  │                                                      │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  КОНФИГУРАЦИЯ (logstash.conf):                            │
│  ──────────────────────────────────                        │
│                                                            │
│  # INPUT: откуда брать данные                             │
│  input {                                                  │
│    # Читать из файла                                      │
│    file {                                                 │
│      path => "/var/log/app/*.log"                         │
│      start_position => "beginning"                        │
│      sincedb_path => "/dev/null"                          │
│    }                                                      │
│                                                            │
│    # Получать по TCP                                      │
│    tcp {                                                  │
│      port => 5000                                         │
│      codec => json                                        │
│    }                                                      │
│                                                            │
│    # Из Beats                                             │
│    beats {                                                │
│      port => 5044                                         │
│    }                                                      │
│  }                                                        │
│                                                            │
│  # FILTER: обработка, парсинг                             │
│  filter {                                                 │
│    # Парсинг JSON                                         │
│    json {                                                 │
│      source => "message"                                  │
│    }                                                      │
│                                                            │
│    # Grok — парсинг неструктурированных логов             │
│    grok {                                                 │
│      match => {                                           │
│        "message" => "%{COMBINEDAPACHELOG}"                │
│      }                                                    │
│    }                                                      │
│                                                            │
│    # Парсинг даты                                         │
│    date {                                                 │
│      match => [ "timestamp", "ISO8601" ]                  │
│      target => "@timestamp"                               │
│    }                                                      │
│                                                            │
│    # Добавление полей                                     │
│    mutate {                                               │
│      add_field => {                                       │
│        "environment" => "production"                      │
│      }                                                    │
│      remove_field => [ "temp_field" ]                     │
│    }                                                      │
│                                                            │
│    # GeoIP (определение локации по IP)                    │
│    geoip {                                                │
│      source => "client_ip"                                │
│    }                                                      │
│  }                                                        │
│                                                            │
│  # OUTPUT: куда отправлять                                │
│  output {                                                 │
│    # В Elasticsearch                                      │
│    elasticsearch {                                        │
│      hosts => ["elasticsearch:9200"]                      │
│      index => "logstash-%{+YYYY.MM.dd}"                   │
│    }                                                      │
│                                                            │
│    # В stdout (для отладки)                               │
│    stdout {                                               │
│      codec => rubydebug                                   │
│    }                                                      │
│  }                                                        │
│                                                            │
│  ПРИМЕР: Парсинг Apache логов                             │
│  ─────────────────────────────────                         │
│  Input (сырой лог):                                       │
│  127.0.0.1 - - [20/Jan/2024:10:15:30 +0000] "GET /api/users HTTP/1.1" 200 1234│
│                                                            │
│  После grok filter:                                       │
│  {                                                        │
│    "client_ip": "127.0.0.1",                              │
│    "timestamp": "20/Jan/2024:10:15:30 +0000",             │
│    "method": "GET",                                       │
│    "request": "/api/users",                               │
│    "http_version": "1.1",                                 │
│    "status_code": 200,                                    │
│    "bytes": 1234                                          │
│  }                                                        │
│                                                            │
│  ✅ ПРЕИМУЩЕСТВА:                                          │
│  ───────────────                                           │
│  • Мощный парсинг (Grok, regex)                           │
│  • Множество плагинов (200+)                              │
│  • Трансформация данных                                   │
│  • Обогащение (GeoIP, DNS lookup)                         │
│  • Фильтрация (drop ненужных логов)                       │
│  • Множество источников (file, syslog, kafka, databases)  │
│  • Множество выходов (Elasticsearch, S3, Kafka, email)    │
│  • Буферизация, retry при ошибках                         │
│                                                            │
│  ❌ НЕДОСТАТКИ:                                            │
│  ─────────────                                             │
│  • ТЯЖЕЛОВЕСНЫЙ (JVM, >1GB RAM)                           │
│  • Медленнее альтернатив (Fluentd, Fluent Bit)            │
│  • Сложная конфигурация (Grok сложен)                     │
│  • Высокое потребление CPU                                │
│  • Не подходит для edge устройств                         │
│                                                            │
│  КОГДА ИСПОЛЬЗОВАТЬ:                                      │
│  ──────────────────                                        │
│  ✅ Сложный парсинг (неструктурированные логи)             │
│  ✅ Обогащение данных (GeoIP, lookups)                     │
│  ✅ Трансформация (сложная логика)                         │
│  ✅ Множество источников и выходов                         │
│  ✅ Центральный сервер (не на каждой ноде)                 │
│                                                            │
│  ❌ НЕ использовать:                                       │
│  • Малые ресурсы (IoT, edge)                              │
│  • Простые логи (уже JSON) → Filebeat напрямую            │
│  • Нужна высокая производительность → Fluent Bit          │
│                                                            │
│  DOCKER ПРИМЕР:                                           │
│  ───────────────                                           │
│  docker run -d \                                          │
│    -v $(pwd)/logstash.conf:/usr/share/logstash/pipeline/logstash.conf \│
│    -p 5000:5000 \                                         │
│    docker.elastic.co/logstash/logstash:8.11.0             │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 4. Fluentd vs Fluent Bit

text

```
┌────────────────────────────────────────────────────────────┐
│         FLUENTD vs FLUENT BIT                              │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ИСТОРИЯ:                                                 │
│  • Fluentd — 2011 год, CNCF проект                        │
│  • Fluent Bit — 2015 год, легковесная версия Fluentd      │
│                                                            │
│  ОБЩЕЕ:                                                   │
│  • Оба от Treasure Data (теперь Calyptia)                 │
│  • Unified logging layer                                  │
│  • Open-source                                            │
│  • Популярны в Kubernetes                                 │
│                                                            │
└────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────┐
│  FLUENTD                                                   │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ХАРАКТЕРИСТИКИ:                                          │
│  • Язык: Ruby (плагины могут быть на C)                   │
│  • Размер: ~40 MB                                         │
│  • Память: ~40-80 MB                                      │
│  • CPU: Средняя нагрузка                                  │
│  • Плагины: 1000+ (огромная экосистема)                   │
│                                                            │
│  АРХИТЕКТУРА:                                             │
│  ┌──────────────────────────────────────────────────────┐ │
│  │                                                      │ │
│  │  INPUT → PARSER → FILTER → BUFFER → OUTPUT           │ │
│  │                                                      │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  КОНФИГУРАЦИЯ (fluentd.conf):                             │
│  ─────────────────────────────                             │
│  # Source (откуда)                                        │
│  <source>                                                 │
│    @type tail                                             │
│    path /var/log/app/*.log                                │
│    pos_file /var/log/fluentd/app.pos                      │
│    tag app.logs                                           │
│    <parse>                                                │
│      @type json                                           │
│    </parse>                                               │

│                                                            │
```

## Полный пример: Production-ready Loki Stack

Bash

```
#!/bin/bash
# deploy-loki-stack.sh - Production установка Loki в K8s

# 1. Создать namespace
kubectl create namespace logging

# 2. Установить Loki Stack
helm repo add grafana https://grafana.github.io/helm-charts
helm repo update

# 3. Кастомные values
cat > loki-production-values.yaml <<'EOF'
loki:
  enabled: true
  persistence:
    enabled: true
    size: 100Gi
  config:
    table_manager:
      retention_deletes_enabled: true
      retention_period: 720h  # 30 дней

promtail:
  enabled: true
  config:
    clients:
      - url: http://loki:3100/loki/api/v1/push

grafana:
  enabled: true
  adminPassword: "CHANGE_ME_PLEASE"
  persistence:
    enabled: true
    size: 20Gi
  resources:
    limits:
      cpu: 500m
      memory: 512Mi
    requests:
      cpu: 250m
      memory: 256Mi
  
prometheus:
  enabled: true
  alertmanager:
    enabled: false
  server:
    persistentVolume:
      enabled: true
      size: 50Gi
EOF

# 4. Установить
helm install loki-stack grafana/loki-stack \
  -f loki-production-values.yaml \
  --namespace logging

# 5. Получить пароль Grafana
kubectl get secret --namespace logging loki-stack-grafana \
  -o jsonpath="{.data.admin-password}" | base64 --decode
echo

# 6. Port-forward для доступа
kubectl port-forward --namespace logging service/loki-stack-grafana 3000:80

echo "Grafana доступна на: http://localhost:3000"
echo "Логин: admin"
```
